一般的によく求められる機能はプラグインとして利用できます；

- [@next/mdx](https://github.com/vercel/next.js/tree/canary/packages/next-mdx)
- [@next/bundle-analyzer](https://github.com/vercel/next.js/tree/canary/packages/next-bundle-analyzer)

`webpack`の使用を拡張するには、`next.config.js`の中でその設定を拡張する関数を定義します；

```js title="next.config.js"
module.exports = {
  webpack: (
    config,
    { buildId, dev, isServer, defaultLoaders, nextRuntime, webpack }
  ) => {
    // 重要: 修正されたconfigを返す
    return config
  },
}
```

> `webpack`関数は3回実行されます、2回はサーバー（nodejs / edge runtime）のためで、1回はクライアントのためです；これにより、`isServer`プロパティを使用してクライアントとサーバーの設定を区別できます；

`webpack`関数の第2引数は次のプロパティを持つオブジェクトです；

- `buildId`: `String` - ビルドの一意の識別子として使用されるビルドid
- `dev`: `Boolean` - コンパイルが開発モードで行われるかどうか
- `isServer`: `Boolean` - サーバーサイドのコンパイルでは`true`で、クライアントサイドのコンパイルでは`false`
- `nextRuntime`: `String | undefined` - サーバーサイドコンパイルのターゲットランタイム。`"edge"`または`"nodejs"`, クライアントサイドのコンパイルには`undefined`
- `defaultLoaders`: `Object` - Next.jsが内部で使用するデフォルトのローダー：
  - `babel`: `Object` - デフォルトの`babel-loader`構成

`defaultLoaders.babel`の使用例です；

```js
// babel-loaderに依存するローダーを追加するためのサンプル設定
// このソースは@next/mdxプラグインソースから取得されました：
// https://github.com/vercel/next.js/tree/canary/packages/next-mdx
module.exports = {
  webpack: (config, options) => {
    config.module.rules.push({
      test: /\.mdx/,
      use: [
        options.defaultLoaders.babel,
        {
          loader: '@mdx-js/loader',
          options: pluginOptions.options,
        },
      ],
    })

    return config
  },
}
```

#### `nextRuntime`

`nextRuntime`が`"edge"`または`"nodejs"`の場合、`isServer`は`true`となることに注意してください；nextRuntimeの`"edge"`は現在、edge runtimeでのミドルウェア及びServer Components専用です；
