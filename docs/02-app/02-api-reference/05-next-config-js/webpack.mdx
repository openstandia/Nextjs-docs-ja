一般的に求められる機能のいくつかはプラグインとして利用可能です：

- [@next/mdx](https://github.com/vercel/next.js/tree/canary/packages/next-mdx)
- [@next/bundle-analyzer](https://github.com/vercel/next.js/tree/canary/packages/next-bundle-analyzer)

`webpack` の使用を拡張するには、`next.config.js` 内でその設定を拡張する関数を定義できます：

```js title="next.config.js"
module.exports = {
  webpack: (
    config,
    { buildId, dev, isServer, defaultLoaders, nextRuntime, webpack }
  ) => {
    // 重要: 変更された設定を返す
    return config
  },
}
```

> `webpack` 関数は 3 回実行され、サーバー用に 2 回（nodejs / edge runtime）、クライアント用に 1 回です。これにより、`isServer` プロパティを使用して、クライアントとサーバの構成を区別できます。

`webpack` 関数への第2引数は、以下のプロパティを持つオブジェクトです：

- `buildId`: `String` - ビルド間の一意識別子として使用するビルドID
- `dev`: `Boolean` - コンパイルが開発モードで行われるかどうかを示す
- `isServer`: `Boolean` - サーバーサイドコンパイルでは `true` で、クライアントサイドコンパイルでは `false`
- `nextRuntime`: `String | undefined` - サーバーサイドコンパイルのターゲットランタイム；`"edge"` または `"nodejs"` のいずれかで、クライアントサイドの場合は `undefined`
- `defaultLoaders`: `Object` - Next.jsによって内部的に使用されるデフォルトのローダー：
  - `babel`: `Object` - デフォルトの `babel-loader` 設定

`defaultLoaders.babel` の使用例：

```js
// babel-loaderに依存するローダーを追加するための設定例
// このソースは@next/mdxプラグインのソースから取得されました：
// https://github.com/vercel/next.js/tree/canary/packages/next-mdx
module.exports = {
  webpack: (config, options) => {
    config.module.rules.push({
      test: /\.mdx/,
      use: [
        options.defaultLoaders.babel,
        {
          loader: '@mdx-js/loader',
          options: pluginOptions.options,
        },
      ],
    })

    return config
  },
}
```

#### `nextRuntime` {#nextruntime}

`nextRuntime` が `"edge"` または `"nodejs"` のとき、`isServer` は `true` であることに注意してください。`nextRuntime` `"edge"` は現在、middleware と edge runtime の Server Component のみ対象です。
