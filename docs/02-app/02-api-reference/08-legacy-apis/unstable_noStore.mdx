> このAPIは将来的に廃止される予定です。バージョン15では、`unstable_noStore`の代わりに[`connection`](/docs/app/api-reference/functions/connection)を使用することをお勧めします。

`unstable_noStore`は、静的レンダリングを宣言的に回避し、特定のコンポーネントをキャッシュしないように示すために使用できます。

```jsx
import { unstable_noStore as noStore } from 'next/cache';

export default async function ServerComponent() {
  noStore(); // キャッシュしないことを示します。
  const result = await db.query(...); // データベースにクエリを送信します。
  ...
}
```

> **Good to know**:
>
> - `unstable_noStore`は`fetch`の`cache: 'no-store'`と同等です
> - `unstable_noStore`は、より細かな粒度でコンポーネントごとに使用できるため、`export const dynamic = 'force-dynamic'`よりも推奨されます

- `unstable_cache`内で`unstable_noStore`を使用しても、静的生成は回避されません。代わりに、キャッシュ構成に従って結果をキャッシュするかどうかを決定します。

## 使い方 {#usage}

`fetch`を使用できない場合や、`fetch`に`cache: 'no-store'`や`next: { revalidate: 0 }`といった追加オプションを渡したくない場合、これらすべてのユースケースの代わりに`noStore()`を使用できます。

```jsx
import { unstable_noStore as noStore } from 'next/cache';

export default async function ServerComponent() {
  noStore(); // キャッシュしないことを示します。
  const result = await db.query(...); // データベースにクエリを送信します。
  ...
}
```
