## 14 から 15 へのアップグレード {#upgrading-from-14-to-15}

Next.js バージョン 15 に更新するには、`upgrade` codemod を使用します：

```bash title="Terminal"
npx @next/codemod@canary upgrade latest
```

手動で行う場合は、Next と React の最新の RC をインストールしてください。例：

```bash title="Terminal"
npm i next@latest react@rc react-dom@rc eslint-config-next@latest
```

> **Good to know:**
>
> - ピア依存関係の警告が表示された場合は、推奨されるバージョンに `react` と `react-dom` を更新する必要があるか、警告を無視するために `--force` または `--legacy-peer-deps` フラグを使用します。Next.js 15 と React 19 の両方が安定版になった時には不要になります。
> - TypeScript を使用している場合、React タイプを一時的にオーバーライドする必要があります。詳細は [React 19 RC アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide#installing)を参照してください。

## React 19 {#react-19}

- 'react' と 'react-dom' の最低バージョンは19になりました。
- `useFormState` は `useActionState` に置き換えられました。`useFormState` フックは React 19 でも利用可能ですが、非推奨であり、将来のリリースで削除される予定です。`useActionState` は推奨され、`pending` 状態の直接的な読み取りなど、追加のプロパティを含みます。[詳しく知る](https://react.dev/reference/react/useActionState)。
- `useFormStatus` には、追加のキー `data`、`method`、および `action` が含まれています。React 19 を使用していない場合、`pending` キーのみが利用可能です。[詳しく知る](https://react.dev/reference/react-dom/hooks/useFormStatus)。
- 詳細は [React 19 アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)を参照してください。

## 非同期リクエスト API（破壊的変更） {#async-request-apis-breaking-change}

以前は同期であったランタイム情報に依存する動的APIが**非同期**になりました：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout)、[`page.js`](/docs/app/api-reference/file-conventions/page)、[`route.js`](/docs/app/api-reference/file-conventions/route)、[`default.js`](/docs/app/api-reference/file-conventions/default)、[`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons)、および [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons) の `params`
- [`page.js`](/docs/app/api-reference/file-conventions/page) の `searchParams`

移行の負担を軽減するために、プロセスを自動化する [codemod が利用可能](/docs/app/building-your-application/upgrading/codemods#150) であり、API に一時的に同期アクセスができます。

### `cookies` {#cookies}

#### 推奨される非同期の使用法 {#recommended-async-usage}

```tsx
import { cookies } from 'next/headers'

// 変更前
const cookieStore = cookies()
const token = cookieStore.get('token')

// 変更後
const cookieStore = await cookies()
const token = cookieStore.get('token')
```

## `runtime` 設定（破壊的変更） {#runtime-configuration-breaking-change}

以前の `runtime` [セグメント設定](/docs/app/api-reference/file-conventions/route-segment-config#runtime)には、`edge` に加えて `experimental-edge` という値がサポートされていました。これらの設定は同じものを指すため、オプションを簡素化するために `experimental-edge` が使用されるとエラーが発生します。これを修正するには、`runtime` 設定を `edge` に更新します。[codemod](/docs/app/building-your-application/upgrading/codemods#app-dir-runtime-config-experimental-edge) が自動でこれを行うために利用可能です。

</AppOnly>

## `fetch` リクエスト {#fetch-requests}

[`fetch` リクエスト](/docs/app/api-reference/functions/fetch) はデフォルトでキャッシュされなくなりました。

特定の `fetch` リクエストをキャッシュに対応させるには、`cache: 'force-cache'` オプションを渡すことができます。

```js title="app/layout.js"
export default async function RootLayout() {
  const a = await fetch('https://...') // 非キャッシュ
  const b = await fetch('https://...', { cache: 'force-cache' }) // キャッシュ

  // ...
}
```

レイアウトまたはページ内のすべての `fetch` リクエストをキャッシュに対応させるには、`export const fetchCache = 'default-cache'` [セグメント設定オプション](/docs/app/api-reference/file-conventions/route-segment-config) を使用できます。個々の `fetch` リクエストが `cache` オプションを指定している場合、それが使用されます。

```js title="app/layout.js"
// これは root レイアウトであるため、個々の fetch リクエストが自分自身の
// キャッシュオプションを設定しない限り、アプリ内のすべての fetch リクエストがキャッシュされます。
export const fetchCache = 'default-cache'

export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュ
  const b = await fetch('https://...', { cache: 'no-store' }) // 非キャッシュ

  // ...
}
```

## ルートハンドラー {#route-handlers}

[ルートハンドラー](/docs/app/api-reference/file-conventions/route)内の `GET` 関数は、デフォルトでキャッシュされなくなりました。`GET` メソッドをキャッシュに対応させるには、ルートハンドラーファイル内で `export const dynamic = 'force-static'` のような [ルート設定オプション](/docs/app/api-reference/file-conventions/route-segment-config) を使用できます。

```js title="app/api/route.js"
export const dynamic = 'force-static'

export async function GET() {}
```

## クライアントサイドルーターキャッシュ {#client-side-router-cache}

`<Link>` または `useRouter` を介してページ間を移動する際に、[ページ](/docs/app/api-reference/file-conventions/page) セグメントはクライアントサイドのルーターキャッシュから再利用されなくなりました。ただし、ブラウザーの前後のナビゲーション中や共有レイアウトのために引き続き再利用されます。

ページセグメントをキャッシュに対応させるには、[`staleTimes`](/docs/app/api-reference/next-config-js/staleTimes) 設定オプションを使用できます：

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
}

module.exports = nextConfig
```

[レイアウト](/docs/app/api-reference/file-conventions/layout) および [読み込み状態](/docs/app/api-reference/file-conventions/loading) は依然としてキャッシュされ、ナビゲーションの際に再利用されます。

## `NextRequest` ジオロケーション {#nextrequest-geolocation}

`NextRequest` 上の `geo` および `ip` プロパティは、これらの値がホスティングプロバイダーによって提供されるため削除されました。[codemod](/docs/app/building-your-application/upgrading/codemods#150) がこの移行を自動化するために利用可能です。

Vercel を使用している場合は、代わりに [`@vercel/functions](https://vercel.com/docs/functions/vercel-functions-package) から `geolocation` および `ipAddress` 関数を使用することもできます：

```ts title="middleware.ts"
import { geolocation } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { city } = geolocation(request)

  // ...
}
```

```ts title="middleware.ts"
import { ipAddress } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const ip = ipAddress(request)

  // ...
}
```
