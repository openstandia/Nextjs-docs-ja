## バージョン 14から15へのアップグレード {#upgrading-from-14-to-15}

Next.js バージョン 15 に更新するには、`upgrade` codemod を使用できます：

```bash title="ターミナル"
npx @next/codemod@canary upgrade
```

手動で実行することを希望する場合は、最新の Next & React RC をインストールしていることを確認してください。例えば：

```bash title="ターミナル"
npm i next@canary react@rc react-dom@rc eslint-config-next@rc
```

> **Good to know:**
>
> - ピアデペンデンシーの警告が表示された場合は、提案されたバージョンに `react` と `react-dom` を更新するか、警告を無視するために `--force` または `--legacy-peer-deps` フラグを使用する必要があります。これは、Next.js 15 および React 19 が安定すると必要なくなります。
> - TypeScriptを使用している場合、Reactタイプを一時的に上書きする必要があります。詳細については、[React 19 RC アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide#installing)をご覧ください。

## React 19 {#react-19}

- `react` と `react-dom` の最小バージョンは19になりました
- `useFormState` は `useActionState` に置き換えられました。React 19 でも `useFormState` フックを使用できますが、非推奨となり将来のリリースで削除される予定です。`useActionState` が推奨され、`pending` 状態を直接読み取るなどの追加プロパティを含んでいます。[詳しくはこちら](https://react.dev/reference/react/useActionState)
- `useFormStatus` は `data`、`method`、`action` などの追加キーを含むようになりました。React 19 を使用していない場合は、`pending` キーのみが利用可能です。[詳しくはこちら](https://react.dev/reference/react-dom/hooks/useFormStatus)
- [React 19 アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)で詳細を確認できます。

## 非同期リクエストAPI（破壊的変更） {#async-request-apis-breaking-change}

以前は同期していた、ランタイム情報に依存する動的APIが**非同期**になりました：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout)、[`page.js`](/docs/app/api-reference/file-conventions/page)、[`route.js`](/docs/app/api-reference/file-conventions/route)、[`default.js`](/docs/app/api-reference/file-conventions/default)、[`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons)、[`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons)の `params`
- [`page.js`](/docs/app/api-reference/file-conventions/page)の `searchParams`

移行の負担を軽減するため、プロセスを自動化する[コーデモッドが利用可能](/docs/app/building-your-application/upgrading/codemods#150)であり、APIは一時的に同期的にアクセスすることができます。

### `cookies` {#cookies}

#### 推奨非同期使用法 {#recommended-async-usage}

```tsx
import { cookies } from 'next/headers'

// 以前
const cookieStore = cookies()
const token = cookieStore.get('token')

// 以降
const cookieStore = await cookies()
const token = cookieStore.get('token')
```

## `runtime` 設定（破壊的変更） {#runtime-configuration-breaking-change}

`runtime` [segment config](/docs/app/api-reference/file-conventions/route-segment-config#runtime)は以前、`edge` に加えて `experimental-edge` の値をサポートしていました。両方の設定は同じものを指しており、オプションを簡素化するために、`experimental-edge` が使用されるとエラーになります。これを修正するには、`runtime` 設定を `edge` に更新してください。[コーデモッド](/docs/app/building-your-application/upgrading/codemods#app-dir-runtime-config-experimental-edge)がこれを自動的に行います。

</AppOnly>

## `fetch` リクエスト {#fetch-requests}

[`fetch` リクエスト](/docs/app/api-reference/functions/fetch)は、デフォルトでキャッシュされなくなりました。

特定の `fetch` リクエストをキャッシュにオプトインするには、`cache: 'force-cache'` オプションを渡すことができます。

```js title="app/layout.js"
export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされない
  const b = await fetch('https://...', { cache: 'force-cache' }) // キャッシュされる

  // ...
}
```

レイアウトやページ内のすべての `fetch` リクエストをキャッシュにオプトインするには、`export const fetchCache = 'default-cache'` [segment config option](/docs/app/api-reference/file-conventions/route-segment-config)を使用できます。個々の `fetch` リクエストが `cache` オプションを指定する場合、それが代わりに使用されます。

```js title="app/layout.js"
// これは root レイアウトであるため、アプリ内のキャッシュオプションを設定していないすべての fetch リクエストがキャッシュされます。
export const fetchCache = 'default-cache'

export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされる
  const b = await fetch('https://...', { cache: 'no-store' }) // キャッシュされない

  // ...
}
```

## Route Handlers {#route-handlers}

[Route Handlers](/docs/app/api-reference/file-conventions/route) 内の `GET` 関数は、デフォルトでキャッシュされなくなりました。`GET` メソッドをキャッシュにオプトインするには、Route Handler ファイル内で `export const dynamic = 'force-static'` などの[ルートconfigオプション](/docs/app/api-reference/file-conventions/route-segment-config)を使用できます。

```js title="app/api/route.js"
export const dynamic = 'force-static'

export async function GET() {}
```

## クライアントサイドのルーターキャッシュ {#client-side-router-cache}

`<Link>` または `useRouter` 経由でページ間を移動する際、[ページ](/docs/app/api-reference/file-conventions/page) セグメントはクライアントサイドのルーターキャッシュから再利用されなくなりました。ただし、ブラウザの戻る・進むナビゲーションおよび共有レイアウトのために再利用されます。

ページセグメントをキャッシュにオプトインするには、[`staleTimes`](/docs/app/api-reference/next-config-js/staleTimes) configオプションを使用できます：

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
}

module.exports = nextConfig
```

[Layouts](/docs/app/api-reference/file-conventions/layout) と [loading states](/docs/app/api-reference/file-conventions/loading) は依然としてナビゲーション時にキャッシュされ、再利用されます。

## `next/font` {#next-font}

`@next/font` パッケージは、組み込みの[`next/font`](/docs/app/api-reference/components/font) に置き換えられました。[コーデモッドが利用可能](/docs/app/building-your-application/upgrading/codemods#built-in-next-font)で、安全かつ自動的にインポートをリネームできます。

```js title="app/layout.js"
// 以前
import { Inter } from '@next/font/google'

// 以降
import { Inter } from 'next/font/google'
```

## Speed Insights {#speed-insights}

Speed Insights に対する自動計測は Next.js 15 で削除されました。

Speed Insights の使用を継続するために、[Vercel Speed Insights クイックスタート](https://vercel.com/docs/speed-insights/quickstart)ガイドを参照してください。

## `NextRequest` Geolocation {#nextrequest-geolocation}

`NextRequest` 上の `geo` プロパティおよび `ip` プロパティは削除されました。これらの値はホスティングプロバイダーによって提供されます。[コーデモッド](/docs/app/building-your-application/upgrading/codemods#150)がこの移行を自動化します。

Vercel を使用している場合は、代わりに[`@vercel/functions`](https://vercel.com/docs/functions/vercel-functions-package) の `geolocation` および `ipAddress` 関数を使用することもできます：

```ts title="middleware.ts"
import { geolocation } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { city } = geolocation(request)

  // ...
}
```

```ts title="middleware.ts"
import { ipAddress } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const ip = ipAddress(request)

  // ...
}
```
