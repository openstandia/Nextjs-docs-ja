Next.jsは、[ESLint](https://eslint.org/)を標準で組み込んでいます。`package.json`に `next lint` をスクリプトとして追加してください：

```json title="package.json"
{
  "scripts": {
    "lint": "next lint"
  }
}
```

次に、`npm run lint` または `yarn lint` を実行してください：

```bash title="Terminal"
yarn lint
```

アプリケーションでまだESLintを設定していない場合、インストールと設定のプロセスが案内されます。

```bash title="Terminal"
yarn lint
```

> 次のようなプロンプトが表示されます：
>
> ? ESLintをどのように設定しますか？
>
> ❯ Strict（推奨）  
>  Base  
>  Cancel

以下の3つのオプションが選択できます：

- **Strict**: Next.jsの基本的なESLint設定に加えて、より厳しい[Core Web Vitalsルールセット](#core-web-vitals)が含まれています。ESLintを初めて設定する開発者に推奨される設定です。

  ```json title=".eslintrc.json"
  {
    "extends": "next/core-web-vitals"
  }
  ```

- **Base**: Next.jsの基本的なESLint設定が含まれています。

  ```json title=".eslintrc.json"
  {
    "extends": "next"
  }
  ```

- **Cancel**: どんなESLint設定も含まれません。このオプションを選ぶのは、独自のESLint設定をする予定がある場合のみです。

2つの設定オプションのいずれかが選択されると、Next.jsは自動的に`eslint`と`eslint-config-next`をアプリケーションの依存関係としてインストールし、選択した設定を含む`.eslintrc.json`ファイルをプロジェクトのルートに作成します。

ESLintを実行してエラーを見つけたいときは、毎回 `next lint` を実行できます。一度ESLintが設定されると、すべてのビルド（`next build`）中にも自動的に実行されます。エラーはビルドを中断させますが、警告はそうしません。

<AppOnly>

> `next build`中にESLintを実行したくない場合は、[Ignoring ESLint](/docs/app/api-reference/next-config-js/eslint)のドキュメントを参照してください。

</AppOnly>

<PagesOnly>

> `next build`中にESLintを実行したくない場合は、[Ignoring ESLint](/docs/pages/api-reference/next-config-js/eslint)のドキュメントを参照してください。

</PagesOnly>

開発中にコードエディタ内で警告やエラーを直接確認できるように、適切な[統合](https://eslint.org/docs/user-guide/integrations#editors)を使用することをお勧めします。

## ESLint Config

デフォルトの構成（`eslint-config-next`）には、Next.jsで最適なすぐに使えるエクスペリエンスを得るために必要なすべてが含まれています。アプリケーションでまだESLintを設定していない場合、この構成とともにESLintを設定するために`next lint`を使用することをお勧めします。

> `eslint-config-next`と他のESLint構成を一緒に使用したい場合、競合が起こらないようにする方法を学ぶために、[追加構成](#additional-configurations)のセクションを参照してください。

`eslint-config-next`内で使用されるESLintプラグインからの推奨ルールセットは以下です：

- [`eslint-plugin-react`](https://www.npmjs.com/package/eslint-plugin-react)
- [`eslint-plugin-react-hooks`](https://www.npmjs.com/package/eslint-plugin-react-hooks)
- [`eslint-plugin-next`](https://www.npmjs.com/package/@next/eslint-plugin-next)

これは`next.config.js`の構成よりも優先されます。

### カスタム設定

#### `rootDir`

Next.jsがルートディレクトリにインストールされていないプロジェクト（モノレポのような）で`eslint-plugin-next`を使用している場合、`.eslintrc`の`settings`プロパティを使用して、Next.jsアプリケーションの場所を`eslint-plugin-next`に指示することができます：

```json title=".eslintrc.json"
{
  "extends": "next",
  "settings": {
    "next": {
      "rootDir": "packages/my-app/"
    }
  }
}
```

`rootDir`はパス（相対または絶対）、グロブ（例: `"packages/*/"`）、またはパスと/またはグロブの配列にすることができます。

## カスタムディレクトリとファイルのリンティング

デフォルトでは、Next.jsは`pages/`、`app/`、`components/`、`lib/`、および`src/`ディレクトリ内のすべてのファイルに対してESLintを実行します。ただし、`next.config.js`のESLint構成で`dirs`オプションを指定することによって、どのディレクトリをビルドするかを指定できます：

```js title="next.config.js"
module.exports = {
  eslint: {
    dirs: ['pages', 'utils'], // 本番ビルド（next build）時に 'pages'と 'utils'ディレクトリにだけESLintを実行する
  },
}
```

同様に、固有のディレクトリとファイルをリンティングするために、`next lint`で`--dir`と`--file`フラグを使用できます：

```bash title="Terminal"
next lint --dir pages --dir utils --file bar.js
```

## キャッシュ

<AppOnly>

パフォーマンスを向上させるため、ESLintによって処理されたファイルの情報はデフォルトでキャッシュされます。これは`.next/cache`に保存されるか、定義された[ビルドディレクトリ](/docs/app/api-reference/next-config-js/distDir)に保存されます。単一のソースファイルの内容以外のものに依存するESLintルールを含める場合、キャッシュを無効にする必要がある場合は、`next lint`で`--no-cache`フラグを使用してください。

</AppOnly>

## ルールの無効化

サポートされているプラグイン（`react`、`react-hooks`、`next`）によって提供されるルールを変更または無効化したい場合、`.eslintrc`の`rules`プロパティを使用して直接変更できます：

```json title=".eslintrc.json"
{
  "extends": "next",
  "rules": {
    "react/no-unescaped-entities": "off",
    "@next/next/no-page-custom-font": "off"
  }
}
```

### Core Web Vitals

`next/core-web-vitals`ルールセットは、`next lint`が初めて実行されるときに**strict**オプションが選択された場合、有効になります。

```json title=".eslintrc.json"
{
  "extends": "next/core-web-vitals"
}
```

`next/core-web-vitals`は、[Core Web Vitals](https://web.dev/vitals/)に影響を与えるルールの場合、デフォルトで警告ではなくエラーになるように`eslint-plugin-next`を更新します。

> `next/core-web-vitals`のエントリーポイントは、[Create Next App](/docs/app/api-reference/cli/create-next-app)で作成された新しいアプリケーションに自動的に含まれます。

### TypeScript

Next.jsのESLintルールに加えて、`create-next-app --typescript`は`next/typescript`でTypeScript特有のリントルールも設定に追加します：

```json title=".eslintrc.json"
{
  "extends": ["next/core-web-vitals", "next/typescript"]
}
```

これらのルールは、[`plugin:@typescript-eslint/recommended`](https://typescript-eslint.io/linting/configs#recommended)に基づいています。詳細については、[typescript-eslint > Configs](https://typescript-eslint.io/linting/configs)を参照してください。

## 他のツールとの併用

### Prettier

ESLintにはコードフォーマッティングルールも含まれており、既存の[Prettier](https://prettier.io/)設定と衝突する可能性があります。ESLintとPrettierを一緒に動作させるために、[eslint-config-prettier](https://github.com/prettier/eslint-config-prettier)をESLint設定に含めることをお勧めします。

まず、依存関係をインストールします：

```bash title="Terminal"
npm install --save-dev eslint-config-prettier

yarn add --dev eslint-config-prettier

pnpm add --save-dev eslint-config-prettier

bun add --dev eslint-config-prettier
```

それから、既存のESLint設定に`prettier`を追加します：

```json title=".eslintrc.json"
{
  "extends": ["next", "prettier"]
}
```

### lint-staged

[lint-staged](https://github.com/okonet/lint-staged)を使ってステージされたGitファイルにリンターを実行するために`next lint`を使用したい場合、`--file`フラグの使用を指定するためにプロジェクトのルートにある`.lintstagedrc.js`ファイルに以下を追加する必要があります。

```js title=".lintstagedrc.js"
const path = require('path')

const buildEslintCommand = (filenames) =>
  `next lint --fix --file ${filenames
    .map((f) => path.relative(process.cwd(), f))
    .join(' --file ')}`

module.exports = {
  '*.{js,jsx,ts,tsx}': [buildEslintCommand],
}
```

## 既存の設定の移行

### 推奨プラグインルールセット

アプリケーションで既にESLintが設定されていて、次の条件のいずれかが真である場合：

- 以下のプラグインの1つ以上が既にインストールされている（`airbnb`や`react-app`などの別の設定を通じて）
  - `react`
  - `react-hooks`
  - `jsx-a11y`
  - `import`
- BabelがNext.js内で設定されている方法とは異なる特定の`parserOptions`を定義している（この設定は[自分のBabel設定をカスタマイズ](/docs/pages/building-your-application/configuring/babel)していない限り推奨されません）
- Node.jsおよび/またはTypeScriptのインストール済みの`eslint-plugin-import`を持ち、インポートを処理するための[解決ツール](https://github.com/benmosher/eslint-plugin-import#resolvers)が定義されている

この場合、これらの設定を削除し、[`eslint-config-next`](https://github.com/vercel/next.js/blob/canary/packages/eslint-config-next/index.js)内でこれらのプロパティがどのように設定されているかを好むか、またはNext.js ESLintプラグインから直接拡張することをお勧めします：

```js
module.exports = {
  extends: [
    //...
    'plugin:@next/next/recommended',
  ],
}
```

プラグインは、`next lint`を実行せずに通常通りプロジェクトにインストールできます：

```bash title="Terminal"
npm install --save-dev @next/eslint-plugin-next

yarn add --dev @next/eslint-plugin-next

pnpm add --save-dev @next/eslint-plugin-next

bun add --dev @next/eslint-plugin-next
```

これにより、同じプラグインやパーサを複数の構成でインポートすることによって発生する衝突やエラーのリスクが排除されます。

### 追加構成

別のESLint構成を既に使用していて`eslint-config-next`を含めたい場合、他の構成の後に最後に拡張されることを確認してください。例：

```json title=".eslintrc.json"
{
  "extends": ["eslint:recommended", "next"]
}
```

`next`構成は`parser`、`plugins`、および`settings`プロパティのデフォルト値を設定することを既に処理しています。これらのプロパティを手動で再宣言する必要はありませんが、ユースケースに応じて異なる構成が必要な場合を除きます。

他の共有可能な構成を含める場合、**これらのプロパティが上書きされたり変更されたりしないようにしなければなりません**。それ以外の場合は、`next`構成と動作を共有する構成を削除することをお勧めします。上記で述べたように、Next.js ESLintプラグインから直接拡張してください。
