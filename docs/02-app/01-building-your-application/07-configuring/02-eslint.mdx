Next.jsは、すぐに使える[ESLint](https://eslint.org/)の統合体験を提供します。`package.json` に `next lint` をスクリプトとして追加します：

```json title="package.json"
{
  "scripts": {
    "lint": "next lint"
  }
}
```

次に、`npm run lint` または `yarn lint` を実行します：

```bash title="Terminal"
yarn lint
```

まだアプリケーションでESLintが設定されていない場合は、インストールと設定プロセスに従いながら進めてください。

```bash title="Terminal"
yarn lint
```

> このようなプロンプトが表示されます：
>
> ? How would you like to configure ESLint?
>
> ❯ Strict (recommended)  
> Base  
> Cancel

以下の3つのオプションのうち1つを選択できます：

- **Strict**: Next.jsの基本ESLint設定と、より厳格な[Core Web Vitalsルールセット](#core-web-vitals)を含みます。ESLintを初めてセットアップする開発者に推奨される設定です。

  ```json title=".eslintrc.json"
  {
    "extends": "next/core-web-vitals"
  }
  ```

- **Base**: Next.jsの基本ESLint設定を含みます。

  ```json title=".eslintrc.json"
  {
    "extends": "next"
  }
  ```

- **Cancel**: ESLint設定を含みません。カスタムESLint設定を独自にセットアップする予定がある場合のみ、このオプションを選択してください。

これらの設定オプションのいずれかを選択すると、Next.jsは自動的にアプリケーションの依存関係として `eslint` と `eslint-config-next` をインストールし、プロジェクトのrootに選択した設定を含む `.eslintrc.json` ファイルを作成します。

ESLintを実行してエラーをキャッチするたびに `next lint` を実行できます。ESLintがセットアップされた後、すべてのビルド（`next build`）の際にも自動的に実行されます。エラーはビルドが失敗しますが、警告は失敗しません。

<AppOnly>

> `next build` 時にESLintを実行したくない場合は、[Ignoring ESLint](/docs/app/api-reference/next-config-js/eslint)のドキュメントを参照してください。

</AppOnly>

<PagesOnly>

> `next build` 時にESLintを実行したくない場合は、[Ignoring ESLint](https://nextjs.org/docs/canary/pages/api-reference/next-config-js/eslint)のドキュメントを参照してください。

</PagesOnly>

開発中にコードエディター内で警告やエラーを直接確認するために、適切な[統合](https://eslint.org/docs/user-guide/integrations#editors)を使用することをお勧めします。

## ESLint Config

デフォルトの設定（`eslint-config-next`）には、Next.jsでの最適なLint体験を得るために必要なすべてが含まれています。アプリケーションでまだESLintが設定されていない場合、この設定とともに `next lint` を使用してESLintを設定することをお勧めします。

> `eslint-config-next` を他のESLint設定と一緒に使用したい場合は、[Additional Configurations](#additional-configurations)セクションを参照して、競合を引き起こすことなく使用する方法を学んでください。

次のESLintプラグインからの推奨されるルールセットはすべて `eslint-config-next` 内で使用されています：

- [`eslint-plugin-react`](https://www.npmjs.com/package/eslint-plugin-react)
- [`eslint-plugin-react-hooks`](https://www.npmjs.com/package/eslint-plugin-react-hooks)
- [`eslint-plugin-next`](https://www.npmjs.com/package/@next/eslint-plugin-next)

これは `next.config.js` の設定よりも優先されます。

### カスタム設定

#### `rootDir`

Next.jsがrootディレクトリにインストールされていないプロジェクト（例：モノレポ）の場合、`.eslintrc`の`settings`プロパティを使用して `eslint-plugin-next` にNext.jsアプリケーションがどこにあるかを伝えることができます：

```json title=".eslintrc.json"
{
  "extends": "next",
  "settings": {
    "next": {
      "rootDir": "packages/my-app/"
    }
  }
}
```

`rootDir` はパス（相対または絶対）やグロブ（即ち `"packages/*/"`）またはパスやグロブの配列になり得ます。

## カスタムディレクトリおよびファイルのLint

デフォルトで、Next.jsは `pages/`、`app/`、`components/`、`lib/`、および `src/` ディレクトリ内のすべてのファイルに対してESLintを実行します。ただし、プロダクションビルド用の `next.config.js` の `eslint` 設定の `dirs` オプションを使用して、どのディレクトリを対象にするか指定できます：

```js title="next.config.js"
module.exports = {
  eslint: {
    dirs: ['pages', 'utils'] // プロダクションビルド（next build）の際にESLintを実行するのは 'pages' と 'utils' ディレクトリのみ
  }
}
```

同様に、`next lint` において特定のディレクトリおよびファイルをLintするためには、`--dir` および `--file` フラグを使用できます：

```bash title="Terminal"
next lint --dir pages --dir utils --file bar.js
```

## キャッシング

<AppOnly>

パフォーマンスを向上させるために、ESLintで処理されたファイルの情報はデフォルトでキャッシュされます。これは `.next/cache` または定義した[ビルドディレクトリ](/docs/app/api-reference/next-config-js/distDir)に保存されます。単一のソースファイルの内容以上に依存するESLintルールが含まれており、キャッシュを無効にする必要がある場合は、`next lint` に `--no-cache` フラグを使用してください。

</AppOnly>

<PagesOnly>

```bash title="Terminal"
next lint --no-cache
```

## ルールの無効化

サポートするプラグイン（`react`、`react-hooks`、`next`）が提供するルールを変更または無効化したい場合は、`.eslintrc` の `rules` プロパティを使用して直接変更できます：

```json title=".eslintrc.json"
{
  "extends": "next",
  "rules": {
    "react/no-unescaped-entities": "off",
    "@next/next/no-page-custom-font": "off"
  }
}
```

### Core Web Vitals

`next lint` が初めて実行され、「strict」オプションが選択された場合、`next/core-web-vitals` ルールセットが有効になります。

```json title=".eslintrc.json"
{
  "extends": "next/core-web-vitals"
}
```

`next/core-web-vitals` は、[Core Web Vitals](https://web.dev/vitals/) に影響を与えるとデフォルトで警告となるいくつかのルールにエラーを出すように、`eslint-plugin-next` をアップデートします。

> `next/core-web-vitals` エントリポイントは、新しいアプリケーションを [Create Next App](/docs/app/api-reference/cli/create-next-app) で構築する際に自動的に含まれます。

### TypeScript

Next.jsのESLintルールに加えて、`create-next-app --typescript` は `next/typescript` を使用して、TypeScript固有のLintルールも設定に追加します：

```json title=".eslintrc.json"
{
  "extends": ["next/core-web-vitals", "next/typescript"]
}
```

これらのルールは、[`plugin:@typescript-eslint/recommended`](https://typescript-eslint.io/linting/configs#recommended)に基づいています。
詳細は、[typescript-eslint > Configs](https://typescript-eslint.io/linting/configs)を参照してください。

## 他のツールとの使用

### Prettier

ESLintはコードフォーマットルールも含んでおり、既存の[Prettier](https://prettier.io/)セットアップと競合する可能性があります。ESLintとPrettierを一緒に機能させるために、[eslint-config-prettier](https://github.com/prettier/eslint-config-prettier)をESLint設定に含めることをお勧めします。

まず、依存関係をインストールします：

```bash title="Terminal"
npm install --save-dev eslint-config-prettier

yarn add --dev eslint-config-prettier

pnpm add --save-dev eslint-config-prettier

bun add --dev eslint-config-prettier
```

次に、既存のESLint設定に `prettier` を追加します：

```json title=".eslintrc.json"
{
  "extends": ["next", "prettier"]
}
```

### lint-staged

`next lint` を [lint-staged](https://github.com/okonet/lint-staged)と一緒に使用してステージされたgitファイルでリンタを実行する場合は、ルートプロジェクトに存在する `.lintstagedrc.js` ファイルに次を追加し、`--file` フラグの使用を指定する必要があります。

```js title=".lintstagedrc.js"
const path = require('path')

const buildEslintCommand = (filenames) =>
  `next lint --fix --file ${filenames
    .map((f) => path.relative(process.cwd(), f))
    .join(' --file ')}`

module.exports = {
  '*.{js,jsx,ts,tsx}': [buildEslintCommand],
}
```

## 既存の設定の移行

### 推奨プラグインルールセット

既にアプリケーションでESLintを設定しており、次のいずれかの条件が真である場合：

- 次のプラグインの1つ以上を既にインストールしている（`airbnb`や`react-app`のような異なる設定を通じてインストールしている場合を含む）：
  - `react`
  - `react-hooks`
  - `jsx-a11y`
  - `import`
- BabelがNext.jsで設定されている方法と異なる `parserOptions` を指定している（Babel設定をカスタマイズしていない限り、これは推奨されません）；
- `eslint-plugin-import` をNode.jsおよび/またはTypeScriptの[リゾルバー](https://github.com/benmosher/eslint-plugin-import#resolvers)を使ってインストールしており、インポートの処理を行う；

以上の場合、[`eslint-config-next`](https://github.com/vercel/next.js/blob/canary/packages/eslint-config-next/index.js) 内でこれらのプロパティが設定されている方法を好む場合はこれらの設定を削除するか、Next.js ESLintプラグインから直接拡張することを推奨します：

```js
module.exports = {
  extends: [
    //...
    'plugin:@next/next/recommended',
  ],
}
```

通常の手順でプロジェクトにプラグインをインストールし、`next lint` を実行する必要はありません：

```bash title="Terminal"
npm install --save-dev @next/eslint-plugin-next

yarn add --dev @next/eslint-plugin-next

pnpm add --save-dev @next/eslint-plugin-next

bun add --dev @next/eslint-plugin-next
```

これは、複数の設定にまたがって同じプラグインまたはパーサーをインポートすることによる衝突やエラーのリスクを排除します。

### 追加設定

別個のESLint設定を既に使用しており、`eslint-config-next` を含めたい場合は、他の設定の後にそれが最後に拡張されていることを確認してください。例えば：

```json title=".eslintrc.json"
{
  "extends": ["eslint:recommended", "next"]
}
```

`next` 設定は、`parser`、`plugins`、`settings` プロパティのデフォルト値を設定することを既に処理しています。使用例に対して異なる設定が必要でない限り、これらのプロパティを手動で再宣言する必要はありません。

他の共有可能な設定を含める場合、**これらのプロパティが上書きまたは変更されないことを確認する必要があります。** さもなければ、`next` 設定と挙動を共有する設定を削除することをお勧めしますか、上記のようにNext.js ESLintプラグインから直接拡張してください。
