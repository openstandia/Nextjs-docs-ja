認証を理解することは、アプリケーションのデータを保護するために非常に重要です。このページでは、ReactとNext.jsの機能を使用して認証を実装する方法を案内します。

始める前に、プロセスを次の三つの概念に分解することが役立ちます：

1. **[認証](#authentication)**：ユーザーが自分自身であることを確認します。これには、ユーザーがユーザー名やパスワードのようなものを使用して自分のIDを証明する必要があります。
2. **[セッション管理](#session-management)**：リクエストを通じてユーザーの認証状態を追跡します。
3. **[認可](#authorization)**：ユーザーがアクセスできるルートやデータを決定します。

### サインアップとログインの機能 {#sign-up-and-login-functionality}

[`<form>`](https://react.dev/reference/react-dom/components/form)要素とReactの[server actions](/docs/app/building-your-application/data-fetching/server-actions-and-mutations)、`useFormState`を使用して、ユーザーの資格情報をキャプチャし、フォームフィールドを検証し、認証プロバイダーのAPIまたはデータベースを呼び出すことができます。

server actionsは常にサーバーで実行されるため、認証ロジックを安全に処理する環境を提供します。

サインアップ/ログイン機能を実装する手順は次のとおりです：

#### 2. サーバーでフォームフィールドを検証する {#2-validate-form-fields-on-the-server}

Server actionを使用して、サーバーでフォームフィールドを検証します。認証プロバイダーがフォーム検証を提供していない場合は、[Zod](https://zod.dev/)や[Yup](https://github.com/jquense/yup)などのスキーマ検証ライブラリを使用できます。

Zodを例として、適切なエラーメッセージを伴うフォームスキーマを定義できます：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/lib/definitions.ts" switcher
import { z } from 'zod'

export const SignupFormSchema = z.object({
  name: z
    .string()
    .min(2, { message: '名前は2文字以上である必要があります。' })
    .trim(),
  email: z
    .string()
    .email({ message: '有効なメールアドレスを入力してください。' })
    .trim(),
  password: z
    .string()
    .min(8, { message: '8文字以上である必要があります' })
    .regex(/[a-zA-Z]/, { message: '少なくとも1文字を含めてください。' })
    .regex(/[0-9]/, { message: '少なくとも1つの数字を含めてください。' })
    .regex(/[^a-zA-Z0-9]/, {
      message: '少なくとも1つの特殊文字を含めてください。',
    })
    .trim(),
})

export type FormState =
  | {
      errors?: {
        name?: string[]
        email?: string[]
        password?: string[]
      }
      message?: string
    }
  | undefined
```

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/actions/auth.ts" switcher
import { SignupFormSchema, FormState } from '@/app/lib/definitions'

export async function signup(state: FormState, formData: FormData) {
  // フォームフィールドを検証します
  const validatedFields = SignupFormSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email'),
    password: formData.get('password'),
  })

  // フォームフィールドが無効な場合は、早期に戻ります
  if (!validatedFields.success) {
    return {
      errors: validatedFields.error.flatten().fieldErrors,
    }
  }

  // プロバイダーまたはDBに呼び出してユーザーを作成...
}
```

サインアップおよび/またはログインフォームを実装する手順は次のとおりです：

1. ユーザーがフォームを通じて自分の資格情報を送信します。
2. フォームはAPIルートで処理されるリクエストを送信します。
3. 認証の成功が確認されると、ユーザーの認証の成功を示すプロセスが完了します。
4. 認証が失敗した場合、エラーメッセージが表示されます。

```tsx title="pages/login.tsx" switcher
import { useRouter } from 'next/router'

export default function LoginPage() {
  const router = useRouter()

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault()

    const formData = new FormData(event.currentTarget)
    const email = formData.get('email')
    const password = formData.get('password')

    const response = await fetch('/api/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    })

    if (response.ok) {
      router.push('/profile')
    } else {
      // エラーを処理します
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input type="email" name="email" placeholder="Email" required />
      <input type="password" name="password" placeholder="Password" required />
      <button type="submit">Login</button>
    </form>
  )
}
```

```ts title="pages/api/auth/login.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'
import { signIn } from '@/auth'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  try {
    const { email, password } = req.body
    await signIn('credentials', { email, password })

    res.status(200).json({ success: true })

    if (error.type === 'CredentialsSignin') {
      res.status(401).json({ error: '無効な資格情報です。' })

}
```

## セッション管理 {#session-management}

セッション管理は、複数回のリクエストにわたってユーザーの認証状態を保存することを保証します。これは、セッションやトークンの作成、保存、更新、および削除を含みます。

セッションには二つのタイプがあります：

1. [**ステートレス**](#stateless-sessions)：セッションデータ（またはトークン）はブラウザのcookieに保存されます。cookieは各リクエストと共に送信され、セッションがサーバーで検証されます。この方法は簡単ですが、適切に実装されないとセキュリティが劣る可能性があります。
2. [**データベース**](#database-sessions)：セッションデータがデータベースに保存され、ユーザーのブラウザには暗号化されたセッションIDだけが送信されます。この方法はより安全ですが、複雑で、サーバーリソースを多く使用する場合があります。

> **Good to know:** どちらの方法も、または両方を使用することができますが、[iron-session](https://github.com/vvo/iron-session)や[Jose](https://github.com/panva/jose)などのセッション管理ライブラリを使用することをお勧めします。

### ステートレスセッション {#stateless-sessions}

<AppOnly>

ステートレスセッションを作成および管理するための手順は次のとおりです：

1. セッションを署名するために使用する秘密キーを生成し、[環境変数](/docs/app/building-your-application/configuring/environment-variables)として保存します。
2. セッションデータを暗号化/復号化するロジックをセッション管理ライブラリを使用して書きます。
3. Next.jsの[`cookies`](/docs/app/api-reference/functions/cookies) APIを使用してcookieを管理します。

上記に加え、ユーザーがアプリケーションに戻ったときにセッションを[更新（またはリフレッシュ）](#updating-or-refreshing-sessions)し、ユーザーがログアウトしたときに[削除](#deleting-the-session)する機能を追加することを検討してください。

> **Good to know:** お使いの[認証ライブラリ](#auth-libraries)にセッション管理が含まれているか確認してください。

#### １． 秘密キーを生成する {#1-generating-a-secret-key}

セッションに署名するために秘密キーを生成するためのいくつかの方法があります。例えば、ターミナルで`openssl`コマンドを使用することを選択するかもしれません：

```bash title="terminal"
openssl rand -base64 32
```

このコマンドは32文字のランダム文字列を生成し、[環境変数ファイル](/docs/app/building-your-application/configuring/environment-variables)に秘密キーとして保存することができます：

```bash title=".env"
SESSION_SECRET=your_secret_key
```

次に、このキーをセッション管理ロジックで参照することができます：

```js title="app/lib/session.js"
const secretKey = process.env.SESSION_SECRET
```

#### セッションを暗号化と復号化する {#2-encrypting-and-decrypting-sessions}

次に、任意の[セッション管理ライブラリ](#session-management-libraries)を使用してセッションを暗号化し、復号化することができます。先の例から続けて、[Jose](https://www.npmjs.com/package/jose) （[edge runtime](/docs/app/building-your-application/rendering/edge-and-nodejs-runtimes)と互換性があります）とReactの[`server-only`](https://www.npmjs.com/package/server-only)パッケージを使用して、セッション管理ロジックがサーバーでのみ実行されるようにします。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/session.ts" switcher
import 'server-only'
import { SignJWT, jwtVerify } from 'jose'
import { SessionPayload } from '@/app/lib/definitions'

const secretKey = process.env.SESSION_SECRET
const encodedKey = new TextEncoder().encode(secretKey)

export async function encrypt(payload: SessionPayload) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('7d')
    .sign(encodedKey)
}

export async function decrypt(session: string | undefined = '') {
  try {
    const { payload } = await jwtVerify(session, encodedKey, {
      algorithms: ['HS256'],
    })
    return payload
  } catch (error) {
    console.log('Failed to verify session')
  }
}
```

</TabItem>

#### cookieを設定する（推奨オプション） {#3-setting-cookies-recommended-options}

cookieにセッションを保存するには、Next.jsの`[`cookies`](/docs/app/api-reference/functions/cookies) APIを使用してください。cookieはサーバー上で設定され、推奨されるオプションを含むべきです：

- **HttpOnly**: クライアントサイドのJavaScriptがcookieにアクセスするのを防ぎます。
- **Secure**: HTTPSを使用してcookieを送信します。
- **SameSite**: cookieをクロスサイトのリクエストで送信できるかどうかを指定します。
- **Max-AgeまたはExpires**: 一定期間後にcookieを削除します。
- **Path**: cookieのURLパスを定義します。

これらのオプションの詳細は[MDN](https://developer.mozilla.org/en-US/docs/Web/HTTP/Cookies)を参照してください。

<TabItem value="ts" label="TypeScript">

```ts title="app/lib/session.ts" switcher
import 'server-only'
import { cookies } from 'next/headers'

export async function createSession(userId: string) {
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  const session = await encrypt({ userId, expiresAt })(await cookies()).set(
    'session',
    session,
    {
      httpOnly: true,
      secure: true,
      expires: expiresAt,
      sameSite: 'lax',
      path: '/',
    }
  )
}
```

### データベースセッション {#database-sessions}

データベースセッションを作成し管理するには、次の手順を実行する必要があります：

1. セッションおよびデータを保存するためのテーブルをデータベースに作成する（またはお使いの認証ライブラリがこれを処理するか確認します）。
2. セッションを挿入、更新、削除する機能を実装します。
3. ユーザーのブラウザでセッションIDを暗号化する前に、それを保存し、データベースとcookieを同期させます（これは任意ですが、[ミドルウェア](#optimistic-checks-with-middleware-optional)での楽観的な認証チェックのために推奨されます）。

<AppOnly>

例えば：

```ts title="app/lib/session.ts" switcher
  const session = await encrypt({ sessionId, expiresAt })

  const cookieStore = await cookies()
  cookieStore.set('session', session, {
    httpOnly: true,
    secure: true,
    expires: expiresAt,
    sameSite: 'lax',
    path: '/',
  })
}
```

## 認可 {#authorization}

ユーザーが認証され、セッションが作成された後、アプリケーション内でユーザーがアクセスできる内容や操作を制御するための認可を実装できます。

認可のチェックには主に二つのタイプがあります：

1. **楽観的**：cookieに保存されたセッションデータを使用して、ユーザーがルートにアクセスする権限があるか、またはアクションを実行する権限があるかを確認します。これらのチェックは、UI要素の表示/非表示や、権限や役割に基づくユーザーのリダイレクトなど、迅速な操作に役立ちます。
2. **安全**：データベースに保存されたセッションデータを使用して、ユーザーがルートにアクセスする権限があるか、またはアクションを実行する権限があるかを確認します。これらのチェックはより安全であり、機密情報や操作へのアクセスが必要なときに使用されます。

どちらのケースでも、以下を推奨します：

- 認可ロジックを集中管理するための[データアクセス層](#creating-a-data-access-layer-dal)を作成すること
- 必要なデータだけを返すための[データ転送オブジェクト（DTO）](#using-data-transfer-objects-dto)を使用すること
- 任意で[ミドルウェア](#optimistic-checks-with-middleware-optional)を使用して楽観的なチェックを行うことができます。

### 楽観的チェックとミドルウェア（任意） {#optimistic-checks-with-middleware-optional}

[ミドルウェア](/docs/app/building-your-application/routing/middleware)を使用し、権限に基づいてユーザーをリダイレクトする必要のある場合があります：

- 楽観的チェックを行うためには、[プレ回収](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)されたルートも含めて、ミドルウェアがすべてのルートで実行されるので、未承認のユーザーを事前にフィルタリングし、リダイレクトロジックを集中的に管理するための良い方法です。
- ユーザー間でデータを共有する静的ルートを保護するため（例えば、有料コンテンツ）。

ただし、ミドルウェアは[プレ回収](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)されたルートも含めてすべてのルートで実行されるため、パフォーマンスの問題を防ぐために、セッションからcookieを読み込む（楽観的なチェック）だけでデータベースチェックを避けることが重要です。

例えば：

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="middleware.ts" switcher
import { NextRequest, NextResponse } from 'next/server'
import { decrypt } from '@/app/lib/session'
import { cookies } from 'next/headers'

// 1. 保護されたルートと公開ルートを指定します
const protectedRoutes = ['/dashboard']
const publicRoutes = ['/login', '/signup', '/']

export default async function middleware(req: NextRequest) {
  // 2. 現在のルートが保護されたものか、公開されたものかを確認します
  const path = req.nextUrl.pathname
  const isProtectedRoute = protectedRoutes.includes(path)
  const isPublicRoute = publicRoutes.includes(path)

  // 3. Cookieからセッションを復号化します
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  // 4. ユーザーが認証されていない場合は、/loginにリダイレクトします
  if (isProtectedRoute && !session?.userId) {
    return NextResponse.redirect(new URL('/login', req.nextUrl))
  }

  // 5. ユーザーが認証されている場合は、/dashboardにリダイレクトします
  if (
    isPublicRoute &&
    session?.userId &&
    !req.nextUrl.pathname.startsWith('/dashboard')
  ) {
    return NextResponse.redirect(new URL('/dashboard', req.nextUrl))
  }

  return NextResponse.next()
}

// ミドルウェアが実行されないルート
export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

</TabItem>

```js title="middleware.js" switcher
    isPublicRoute &&
    session?.userId &&
    !req.nextUrl.pathname.startsWith('/dashboard')
  ) {
    return NextResponse.redirect(new URL('/dashboard', req.nextUrl))
  }

  return NextResponse.next()
}

// ミドルウェアが実行されないルート
export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

</TabItem>
</Tabs>

ミドルウェアは初期チェックに役立ちますが、データを保護するための唯一の防御手段として使用するべきではありません。セキュリティはデータソースにできるだけ近いところで実行されるべきであり、詳細は[データアクセスレイヤー](#creating-a-data-access-layer-dal)を参照してください。

> **Tips**:
>
> - ミドルウェアでは、`req.cookies.get('session').value`を使用してcookieを読むこともできます。
> - ミドルウェアは[edge runtime](/docs/app/building-your-application/rendering/edge-and-nodejs-runtimes)を使用しています。お使いの認証ライブラリとセッション管理ライブラリが互換性があるか確認してください。
> - ミドルウェアの`matcher`プロパティを使用して、ミドルウェアを実行するルートを指定できます。ただし、認証のためには、ミドルウェアがすべてのルートで実行されることが推奨されます。

### データアクセスレイヤー（DAL）の作成 {#creating-a-data-access-layer-dal}

データリクエストと認可ロジックを集中管理するためにDALを作成することをお勧めします。

DALには、アプリケーションと連携する際にユーザーのセッションを確認する関数を含むべきです。少なくとも、その関数はセッションが有効であるかどうかを確認し、その後、ユーザー情報を返すか、リダイレクトを行うべきです。

例えば、DALのために別途ファイルを作成し、`verifySession()`関数を含めてください。次に、関数の返り値をReactの[cache](https://react.dev/reference/react/cache)APIを使用してReactのレンダーパス中にメモ化します：

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/dal.ts" switcher
import 'server-only'

import { cookies } from 'next/headers'
import { decrypt } from '@/app/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  if (!session?.userId) {
    redirect('/login')
  }

  return { isAuth: true, userId: session.userId }
})
```

</TabItem>
<TabItem value="js" label="JavaScript">

```js title="app/lib/dal.js" switcher
import 'server-only'

import { cookies } from 'next/headers'

    redirect('/login')

```

</TabItem>
</Tabs>

```tsx title="app/lib/dal.ts" switcher
        id: true,

        email: true,
```

> **Tip**:
>
> - DALは、要求時にデータを保護するために使用できます。しかし、ユーザー間でデータを共有する静的ルートでは、データはビルド時に取得され、要求時には取得されません。[楽観的チェックとミドルウェア](#optimistic-checks-with-middleware-optional)を使用して静的ルートを保護してください。
> - 安全なチェックのために、セッションが有効かどうかをデータベースと比較して確認できます。重複した不要なデータベースへの要求を避けるために、Reactの[cache](https://react.dev/reference/react/cache)関数を使用してください。
> - 関連するデータリクエストをJavaScriptクラスに統合し、あらゆるメソッドを実行する前に`verifySession()`を実行することを検討するかもしれません。

### データ転送オブジェクト（DTO）の使用 {#using-data-transfer-objects-dto}

データを取得する際、アプリケーションで実際に使用される必要なデータのみを返し、全体のオブジェクトは返さないことが推奨されます。例えば、ユーザーデータを取得する場合、ユーザーのIDと名前だけを返すかもしれません。ユーザーオブジェクト全体には、パスワード、電話番号などが含まれている可能性があります。

ただし、返されるデータ構造を制御できない場合、またはチームで協力しつつ、全体のオブジェクトがクライアントに渡されないようにする場合、安全にクライアントに公開しても問題ないフィールドを指定するなどの戦略を使用できます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/dto.ts" switcher
import 'server-only'
import { getUser } from '@/app/lib/dal'

function canSeeUsername(viewer: User) {
  return true
}

function canSeePhoneNumber(viewer: User, team: string) {
  return viewer.isAdmin || team === viewer.team
}

export async function getProfileDTO(slug: string) {
  const data = await db.query.users.findMany({
    where: eq(users.slug, slug),
    // ここで特定のカラムを返す
  })
  const user = data[0]

  const currentUser = await getUser(user.id)

  // あるいはここでクエリに特化したデータを返す
  return {
    username: canSeeUsername(currentUser) ? user.username : null,
    phonenumber: canSeePhoneNumber(currentUser, user.team)
      ? user.phonenumber
      : null,
  }
}
```

</TabItem>
<TabItem value="js" label="JavaScript">


> - [次の記事](https://nextjs.org/blog/security-nextjs-server-components-actions)で、Next.jsにおけるセキュリティのベストプラクティスについて詳細を学ぶことができます。

### server component {#server-components}

[server component](/docs/app/building-your-application/rendering/server-components)の認証チェックは、役割ベースのアクセスに役立ちます。たとえば、ユーザーの役割に応じてコンポーネントを条件付きでレンダリングする場合：

### レイアウトと認証チェック {#layouts-and-auth-checks}

[Partial Rendering](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)により、[レイアウト](/docs/app/building-your-application/routing/layouts-and-templates)でチェックを行う際には注意が必要です。これらはナビゲーション時に再レンダリングされず、ユーザーセッションはすべてのルート変更時にチェックされません。

代わりに、データソースや条件付きでレンダリングされるコンポーネントに近いところでチェックを行うべきです。

たとえば、ユーザーデータを取得し、ナビゲーションにユーザー画像を表示する共有レイアウトを考えてみます。レイアウトで認証チェックを行う代わりに、レイアウト内でユーザーデータ(`getUser()`)を取得し、DALで認証チェックを行うべきです。

これにより、アプリケーション内で`getUser()`が呼び出されるところでは、常に認証チェックが実行され、データにアクセスする権限があることを開発者が忘れることを防ぎます。

### server action {#server-actions}

[server action](/docs/app/building-your-application/data-fetching/server-actions-and-mutations)は公開APIエンドポイントと同じセキュリティ上の考慮事項を持ち、ユーザーが変更を行うことが許可されるかどうかを確認する必要があります。

以下の例では、アクションを続行する前にユーザーの役割を確認しています：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/lib/actions.ts" switcher
'use server'
import { verifySession } from '@/app/lib/dal'

export async function serverAction(formData: FormData) {
  const session = await verifySession()
  const userRole = session?.user?.role

  // ユーザーがアクションを実行する権限がない場合は早期に戻ります
  if (userRole !== 'admin') {
    return null
  }

  // 許可されたユーザーのためにアクションを実行します
}
```

### Route Handlers {#route-handlers}

[Route Handlers](/docs/app/building-your-application/routing/route-handlers)も公開APIエンドポイントと同じセキュリティ上の考慮事項を持ち、ユーザーがRoute Handlerにアクセスできるかどうかを確認する必要があります。

```ts title="app/api/route.ts" switcher

  // ユーザー認証と役割の確認
  const session = await verifySession()

  // ユーザーが認証されているかどうかを確認します
  if (!session) {
    // ユーザーは認証されていません
    return new Response(null, { status: 401 })
  }

  // ユーザーが'admin'役割を持っているかどうかを確認します
  if (session.user.role !== 'admin') {
    // ユーザーは認証されているが、正しい権限を持っていません
    return new Response(null, { status: 403 })
  }

  // 許可されたユーザーのために続行します
}
```

#### APIルートの保護 {#protecting-api-routes}

Next.js での APIルートは、サーバーサイドのロジックとデータ管理を処理するために不可欠です。これらのルートをセキュアにすることは、特定の機能にアクセスできるのは認可されたユーザーだけであることを保証するために重要です。これは通常、ユーザーの認証状態とその役割に基づくパーミッションを検証することを含みます。

以下は、APIルートのセキュリティを強化する方法の例です：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/route.ts" switcher
import { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  const session = await getSession(req)

  // ユーザーが認証されているか確認します
  if (!session) {
    res.status(401).json({
      error: 'ユーザーが認証されていません。',
    })
    return
  }

  // ユーザーが'admin'役割を持っているかを確認します
  if (session.user.role !== 'admin') {
    res.status(401).json({
      error: '承認されていないアクセス：ユーザーには管理者権限がありません。',
    })
    return
  }

  // 許可されたユーザーのためにルートを続行します
  // ... APIルートの実装
}
```

</TabItem>
