## バージョン14から15へのアップグレード {#upgrading-from-14-to-15}

Next.js バージョン15にアップデートするには、`upgrade` codemod を使用します:

```bash title="Terminal"
npx @next/codemod@canary upgrade latest
```

手動で行う場合は、最新の Next と React RC をインストールしてください。例:

```bash title="Terminal"
npm i next@latest react@rc react-dom@rc eslint-config-next@latest
```

> **Good to know:**
>
> - ピア依存関係の警告が表示された場合は、`react` および `react-dom` を提案されたバージョンに更新するか、警告を無視するために `--force` または `--legacy-peer-deps` フラグを使用する必要があります。これは、Next.js 15 と React 19 が安定版になれば必要ありません。

## React 19 {#react-19}

- `react` および `react-dom` の最小バージョンは19になりました。
- `useFormState` は `useActionState` に置き換えられました。`useFormState` フックは React 19 でも利用可能ですが、非推奨となり、将来のリリースで削除されます。`useActionState` が推奨され、`pending` 状態の読み取りなど、追加のプロパティを含んでいます。[詳細はこちら](https://react.dev/reference/react/useActionState)
- `useFormStatus` は現在、`data`、`method`、`action` などの追加キーを含んでいます。React 19 を使用していない場合、`pending` キーのみが利用可能です。[詳細はこちら](https://react.dev/reference/react-dom/hooks/useFormStatus)
- [React 19 アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)で詳しく読むことができます。

> **Good to know:** TypeScript を使用している場合は、`@types/react` と `@types/react-dom` も最新バージョンにアップグレードしてください。

## 非同期リクエスト API（破壊的変更） {#async-request-apis-breaking-change}

ランタイム情報に依存する以前の同期的な Dynamic API は現在 **非同期** です：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`params`](/docs/app/api-reference/file-conventions/layout), [`page.js`](/docs/app/api-reference/file-conventions/page), [`route.js`](/docs/app/api-reference/file-conventions/route), [`default.js`](/docs/app/api-reference/file-conventions/default), [`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image), [`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image), [`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons), [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons).
- [`searchParams`](/docs/app/api-reference/file-conventions/page) in [`page.js`](/docs/app/api-reference/file-conventions/page)

移行の負担を軽減するために、プロセスを自動化するための [codemod が利用可能](/docs/app/building-your-application/upgrading/codemods#150) です。API は一時的に同期的にアクセス可能です。

### `cookies` {#cookies}

#### 推奨される非同期の使用方法 {#recommended-async-usage}

```tsx
import { cookies } from 'next/headers'

// 以前
const cookieStore = cookies()
const token = cookieStore.get('token')

// 以降
const cookieStore = await cookies()
const token = cookieStore.get('token')
```

## `runtime` 構成（破壊的変更） {#runtime-configuration-breaking-change}

以前は`runtime` [segment configuration](/docs/app/api-reference/file-conventions/route-segment-config#runtime) で、`edge` に加えて `experimental-edge` という値がサポートされていました。両方の構成は同じものを指しており、オプションを簡素化するために、今後は `experimental-edge` を使用するとエラーが発生します。修正するには、`runtime` 構成を `edge` に更新します。[codemod](/docs/app/building-your-application/upgrading/codemods#app-dir-runtime-config-experimental-edge) がこれを自動的に行うために利用可能です。

</AppOnly>

## `fetch`リクエスト {#fetch-requests}

[`fetch` リクエスト](/docs/app/api-reference/functions/fetch) はデフォルトではキャッシュされなくなりました。

特定の`fetch` リクエストをキャッシュにオプトインするには、`cache: 'force-cache'` オプションを指定します。

```js title="app/layout.js"
export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされない
  const b = await fetch('https://...', { cache: 'force-cache' }) // キャッシュされる

  // ...
}
```

レイアウトやページ内のすべての`fetch` リクエストをキャッシュにオプトインするには、`export const fetchCache = 'default-cache'` [segment config option](/docs/app/api-reference/file-conventions/route-segment-config) を使用できます。個々の`fetch` リクエストが`cache` オプションを指定している場合は、それが使用されます。

```js title="app/layout.js"
// これは root レイアウトなので、アプリ内のすべての fetch リクエスト
// に独自の cache オプションを設定しない限りキャッシュされます。
export const fetchCache = 'default-cache'

export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされる
  const b = await fetch('https://...', { cache: 'no-store' }) // キャッシュされない

  // ...
}
```

## ルートハンドラ {#route-handlers}

[ルートハンドラ](/docs/app/api-reference/file-conventions/route) の`GET` 関数はデフォルトではキャッシュされなくなりました。`GET` メソッドをキャッシュにオプトインするには、ルートハンドラファイル内で`export const dynamic = 'force-static'`などの[ルート構成オプション](/docs/app/api-reference/file-conventions/route-segment-config)を使用できます。

```js title="app/api/route.js"
export const dynamic = 'force-static'

export async function GET() {}
```

## クライアントサイドルーターキャッシュ {#client-side-router-cache}

`<Link>` または `useRouter` を介してページ間をナビゲートする際、クライアントサイドルーターキャッシュからは[page](/docs/app/api-reference/file-conventions/page)セグメントが再利用されなくなりました。ただし、ブラウザの後方および前方ナビゲーション、共有レイアウトでは引き続き再利用されます。

ページセグメントをキャッシュにオプトインするには、[`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 構成オプションを使用できます:

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
}

module.exports = nextConfig
```

[レイアウト](/docs/app/api-reference/file-conventions/layout) と[ローディング状態](/docs/app/api-reference/file-conventions/loading)は、ナビゲーション時にキャッシュされ、再利用されます。

## `next/font` {#next-font}

`@next/font` パッケージは組み込みの[`next/font`](/docs/app/api-reference/components/font)に置き換えられました。インポートを安全かつ自動的にリネームするための[codemod が利用可能](/docs/app/building-your-application/upgrading/codemods#built-in-next-font)です。

```js title="app/layout.js"
// 以前
import { Inter } from '@next/font/google'

// 以降
import { Inter } from 'next/font/google'
```

## bundlePagesRouterDependencies {#bundlepagesrouterdependencies}

`experimental.bundlePagesExternals` は安定し、`bundlePagesRouterDependencies` にリネームされました。

const nextConfig = {
  // 以前

## Speed Insights {#speed-insights}

Speed Insightsの自動計測はNext.js 15で削除されました。

Speed Insightsを使用し続けるには、[Vercel Speed Insights Quickstart](https://vercel.com/docs/speed-insights/quickstart)ガイドに従ってください。

## `NextRequest` ジオロケーション {#nextrequest-geolocation}

`NextRequest`の`geo`および`ip`プロパティは、ホスティングプロバイダーによって提供されるため削除されました。[codemod](/docs/app/building-your-application/upgrading/codemods#150)がこの移行を自動化するために利用可能です。

Vercelを使用している場合、代わりに[`@vercel/functions`](https://vercel.com/docs/functions/vercel-functions-package) から`geolocation`および`ipAddress`関数を使用することもできます:

```ts title="middleware.ts"
import { geolocation } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { city } = geolocation(request)

  // ...
}
```

```ts title="middleware.ts"
import { ipAddress } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const ip = ipAddress(request)

  // ...
}
```
