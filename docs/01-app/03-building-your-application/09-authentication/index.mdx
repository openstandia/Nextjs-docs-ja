認証を理解することは、アプリケーションのデータを保護するために重要です。このページでは、認証を実装するためにどのReactとNext.jsの機能を使用するかを案内します。

始める前に、プロセスを三つの概念に分解するのが役立ちます：

1. **[認証](#authentication)**: ユーザーが自身であることを確認します。これには、ユーザーが名前やパスワードなどの所有物を使って自身の身元を証明することが必要です。
2. **[セッション管理](#session-management)**: リクエスト全体でユーザーの認証状態を追跡します。
3. **[認可](#authorization)**: ユーザーがアクセスできるルートとデータを決定します。

このページの例では、教育目的で基本的なユーザー名とパスワードの認証を紹介しています。独自の認証ソリューションを実装することもできますが、セキュリティと簡便さを向上させるためには、認証ライブラリを使用することをお勧めします。これらは認証、セッション管理、認可のための組み込みソリューションを提供し、ソーシャルログイン、多要素認証、役割ベースのアクセス制御などの追加機能も含んでいます。[認証ライブラリ](#auth-libraries)セクションでリストを見つけることができます。

## 認証 {#authentication}

### サインアップとログイン機能 {#sign-up-and-login-functionality}

ユーザーの資格情報をキャプチャし、フォームフィールドを検証し、認証プロバイダーのAPIまたはデータベースを呼び出すために、Reactの[Server Actions](/docs/app/building-your-application/data-fetching/server-actions-and-mutations)と`useActionState`を使用することで、[`<form>`](https://react.dev/reference/react-dom/components/form)要素を利用できます。

Server Actionsは常にサーバー上で実行されるため、認証ロジックを扱うための安全な環境を提供します。

サインアップ/ログイン機能を実装する手順は次のとおりです。

#### 2. サーバーでフォームフィールドを検証する {#2-validate-form-fields-on-the-server}

Server Actionを使用して、サーバー上でフォームフィールドを検証します。認証プロバイダーがフォーム検証を提供していない場合は、[Zod](https://zod.dev/)や[Yup](https://github.com/jquense/yup)などのスキーマ検証ライブラリを使用できます。

Zodを使用する例として、適切なエラーメッセージを持つフォームスキーマを定義できます：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/lib/definitions.ts" switcher
import { z } from 'zod'

export const SignupFormSchema = z.object({
  name: z
    .string()
    .min(2, { message: '名前は少なくとも2文字必要です。' })
    .trim(),
  email: z
    .string()
    .email({ message: '有効なメールアドレスを入力してください。' })
    .trim(),
  password: z
    .string()
    .min(8, { message: '少なくとも8文字必要です' })
    .regex(/[a-zA-Z]/, {
      message: '少なくとも1つの文字を含める必要があります。',
    })
    .regex(/[0-9]/, { message: '少なくとも1つの数字を含める必要があります。' })
    .regex(/[^a-zA-Z0-9]/, {
      message: '少なくとも1つの特殊文字を含める必要があります。',
    })
    .trim(),
})

export type FormState =
  | {
      errors?: {
        name?: string[]
        email?: string[]
        password?: string[]
      }
      message?: string
    }
  | undefined
```

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/actions/auth.ts" switcher
import { SignupFormSchema, FormState } from '@/app/lib/definitions'

export async function signup(state: FormState, formData: FormData) {
  // フォームフィールドを検証
  const validatedFields = SignupFormSchema.safeParse({
    name: formData.get('name'),
    email: formData.get('email'),
    password: formData.get('password'),
  })

  // もしどのフォームフィールドも無効な場合、早めに戻る
  if (!validatedFields.success) {
    return {
      errors: validatedFields.error.flatten().fieldErrors,
    }
  }

  // プロバイダーまたはdbを呼び出してユーザーを作成する...
}
```

> **Good to know:**
>
> - React 19では、`useFormStatus`は、データ、メソッド、アクションなどの追加のキーを返されるオブジェクトに含みます。React 19を使用していない場合、利用可能なキーは `pending` のみです。

#### 3. ユーザーを作成するか、ユーザー資格情報を確認する {#3-create-a-user-or-check-user-credentials}

ユーザーアカウントの作成が成功するかユーザー資格情報の確認後、ユーザーの認証状態を管理するためにセッションを作成することができます。セッション管理の戦略によって、セッションはcookieやデータベースに、または両方に保管される場合があります。詳しくは[セッション管理](#session-management)セクションを参照してください。

>
> - 上記の例は、教育目的のために認証ステップを分解しているため冗長です。これは、独自の安全なソリューションを実装することが迅速に複雑になる可能性があることを示しています。プロセスを簡略化するために、[認証ライブラリ](#auth-libraries)の使用を検討してください。

サインアップやログインフォームを実装する手順は次のとおりです。

1. ユーザーがフォームを通して資格情報を送信します。
2. フォームはAPIルートによって処理されるリクエストを送信します。
3. 検証が成功すると、ユーザーの認証が成功したことが通知されます。
4. 検証が失敗した場合は、エラーメッセージが表示されます。

ユーザーが資格情報を入力できるログインフォームを考えてみましょう。

```tsx title="pages/login.tsx" switcher
import { useRouter } from 'next/router'

export default function LoginPage() {
  const router = useRouter()

  async function handleSubmit(event: FormEvent<HTMLFormElement>) {
    event.preventDefault()

    const formData = new FormData(event.currentTarget)
    const email = formData.get('email')
    const password = formData.get('password')

    const response = await fetch('/api/auth/login', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({ email, password }),
    })

    if (response.ok) {
      router.push('/profile')
    } else {
      // エラーを処理する
    }
  }

  return (
    <form onSubmit={handleSubmit}>
      <input type="email" name="email" placeholder="Email" required />
      <input type="password" name="password" placeholder="Password" required />
      <button type="submit">Login</button>
    </form>
  )
}
```

```ts title="pages/api/auth/login.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'
import { signIn } from '@/auth'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  try {
    const { email, password } = req.body
    await signIn('credentials', { email, password })

    res.status(200).json({ success: true })
  } catch (error) {
    if (error.type === 'CredentialsSignin') {
      res.status(401).json({ error: '資格情報が無効です。' })
    } else {
      res.status(500).json({ error: '何かが間違いました。' })

  }
}
```

## セッション管理 {#session-management}

セッション管理は、リクエスト全体でユーザーの認証された状態を保持することを保証します。これは、セッションやトークンの作成、保存、更新、および削除を含みます。

二つのタイプのセッションがあります：

1. [**ステートレス**](#stateless-sessions): セッションデータ（またはトークン）はブラウザのcookieに保存されます。cookieは各リクエストと共に送信され、サーバー上でセッションを検証することができます。この方法はシンプルですが、正しく実装されないと安全性が低くなる場合があります。
2. [**データベース**](#database-sessions): セッションデータはデータベースに保存され、ユーザーのブラウザには暗号化されたセッションIDのみが送信されます。この方法はより安全ですが、複雑であり、より多くのサーバーリソースを使用することがあります。

> **Good to know:** どちらの方法を使用するか、または両方を使用することができますが、[iron-session](https://github.com/vvo/iron-session) や [Jose](https://github.com/panva/jose) などのセッション管理ライブラリの使用をお勧めします。

### ステートレスセッション {#stateless-sessions}

<AppOnly>

ステートレスセッションを作成および管理するために、従うべき手順は次のとおりです：

1. セッションを署名するために使用される秘密鍵を生成し、[環境変数](/docs/app/building-your-application/configuring/environment-variables)として保存します。
2. セッション管理ライブラリを使用して、セッションデータを暗号化/復号化するロジックを記述します。
3. Next.jsの[`cookies`](/docs/app/api-reference/functions/cookies)APIを使用してcookieを管理します。

上記に加えて、ユーザーがアプリケーションに戻ったときにセッションを[更新(またはリフレッシュ)](#updating-or-refreshing-sessions)し、ユーザーがログアウトしたときにセッションを[削除](#deleting-the-session)する機能を追加することを検討してください。

> **Good to know:** 使用している[authライブラリ](#auth-libraries)がセッション管理を含んでいるかどうか確認してください。

#### 1. 秘密鍵を生成する {#1-generating-a-secret-key}

セッションを署名するための秘密鍵を生成する方法はいくつかあります。たとえば、ターミナルで`openssl`コマンドを使用することを選択するかもしれません：

```bash title="terminal"
openssl rand -base64 32
```

このコマンドは、秘密鍵として使用できる32文字のランダムな文字列を生成し、[環境変数ファイル](/docs/app/building-your-application/configuring/environment-variables)に保存できます：

```bash title=".env"
SESSION_SECRET=your_secret_key
```

その後、この鍵をセッション管理ロジックで参照することができます：

```js title="app/lib/session.js"
const secretKey = process.env.SESSION_SECRET
```

#### 2. セッションを暗号化および復号化する {#2-encrypting-and-decrypting-sessions}

次に、好みの[セッション管理ライブラリ](#session-management-libraries)を使用してセッションを暗号化および復号化できます。前の例を続けて、[Jose](https://www.npmjs.com/package/jose)（[Edge Runtime](/docs/app/building-your-application/rendering/edge-and-nodejs-runtimes)と互換性があります）とReactの[`server-only`](https://www.npmjs.com/package/server-only)パッケージを使用して、セッション管理ロジックがサーバー上のみに実行されるようにします。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/session.ts" switcher
import 'server-only'
import { SignJWT, jwtVerify } from 'jose'
import { SessionPayload } from '@/app/lib/definitions'

const secretKey = process.env.SESSION_SECRET
const encodedKey = new TextEncoder().encode(secretKey)

export async function encrypt(payload: SessionPayload) {
  return new SignJWT(payload)
    .setProtectedHeader({ alg: 'HS256' })
    .setIssuedAt()
    .setExpirationTime('7d')
    .sign(encodedKey)
}

export async function decrypt(session: string | undefined = '') {
  try {
    const { payload } = await jwtVerify(session, encodedKey, {
      algorithms: ['HS256'],
    })
    return payload
  } catch (error) {
    console.log('セッションの検証に失敗しました')
  }
}
```

#### 3. cookieを設定する（推奨されるオプション） {#3-setting-cookies-recommended-options}

セッションをcookieに保存するには、Next.jsの[`cookies`](/docs/app/api-reference/functions/cookies)APIを使用します。cookieはサーバー上で設定され、推奨されるオプションを含める必要があります：

- **HttpOnly**: クライアントサイドのJavaScriptがcookieにアクセスするのを防ぎます。
- **Secure**: cookieをhttpsで送信します。
- **SameSite**: cookieがクロスサイトリクエストと共に送信できるかどうかを指定します。
- **Max-Age or Expires**: 一定期間後にcookieを削除します。
- **Path**: cookieのためのURLパスを定義します。

これらのオプションの詳細については、[MDN](https://developer.mozilla.org/ja/docs/Web/HTTP/Cookies)を参照してください。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/lib/session.ts" switcher
import 'server-only'
import { cookies } from 'next/headers'

export async function createSession(userId: string) {
  const expiresAt = new Date(Date.now() + 7 * 24 * 60 * 60 * 1000)
  const session = await encrypt({ userId, expiresAt })
  const cookieStore = await cookies()

  cookieStore.set('session', session, {
    httpOnly: true,
    secure: true,
    expires: expiresAt,
    sameSite: 'lax',
    path: '/',
  })
}
```

### データベースセッション {#database-sessions}

データベースセッションを作成および管理するために、次の手順を実行する必要があります：

1. セッションとデータを保存するためのテーブルをデータベースに作成する（またはAuthライブラリがこれを処理しているかどうか確認します）。
2. セッションを挿入、更新、削除する機能を実装する
3. セッションIDをユーザーブラウザに保存する前に暗号化し、データベースとcookieを同期させる（[ミドルウェア](#optimistic-checks-with-middleware-optional)で楽観的な認証チェックを行うためにオプションですが推奨されます）。

<AppOnly>

たとえば：

```ts title="app/lib/session.ts" switcher
    httpOnly: true,
    secure: true,
    expires: expiresAt,
    sameSite: 'lax',
    path: '/',
  })
}
```

## 認可 {#authorization}

ユーザーが認証され、セッションが作成されたら、アプリケーション内でユーザーがアクセスできる内容と実行できる内容を制御するための認可を実装できます。

認可チェックには二つの主要なタイプがあります：

1. **楽観的**: cookieに保存されたセッションデータを使用して、ユーザーがルートをアクセスするかアクションを実行する権限があるかどうかをチェックします。これらのチェックは、UI要素を表示/非表示にしたり、権限や役割に基づいてユーザーをリダイレクトするような迅速な操作に有用です。
2. **セキュア**: データベースに保存されたセッションデータを使用して、ユーザーがルートをアクセスするかアクションを実行する権限があるかどうかをチェックします。これらのチェックはより安全で、機密データやアクションへのアクセスが必要な操作に使用されます。

どちらの場合でも、以下を推奨します：

- 認可ロジックを集約するための[データアクセスレイヤー(DAL)](#creating-a-data-access-layer-dal)を作成する
- 必要なデータのみを返すための[データ転送オブジェクト(DTO)](#using-data-transfer-objects-dto)を使用する
- 楽観的チェックを行うために[ミドルウェア](#optimistic-checks-with-middleware-optional)を使用するオプションがあります。

### ミドルウェアを使用した楽観的なチェック（オプション） {#optimistic-checks-with-middleware-optional}

いくつかの場合、[ミドルウェア](/docs/app/building-your-application/routing/middleware)を使用して、ユーザーをリダイレクトすることを検討することがあります：

- 楽観的なチェックを行うためです。ミドルウェアはすべてのルートで実行されるため、リダイレクトロジックを集約し、許可されていないユーザーをあらかじめフィルタリングするのによい方法です。
- ユーザー間でデータを共有する静的ルートを保護するためです（例：有料ウォールの裏側のコンテンツ）。

ただし、ミドルウェアは[プリフェッチ](/docs/app/building-your-application/routing/linking-and-navigating#2-prefetching)されたルートを含め、すべてのルート上で実行されるため、性能問題を防ぐために データベースチェックを避け、cookieからセッションを読み取る(楽観的なチェック）ことが重要です。

以下は例です：

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="middleware.ts" switcher
import { NextRequest, NextResponse } from 'next/server'
import { decrypt } from '@/app/lib/session'
import { cookies } from 'next/headers'

// 1. 保護されているルートと公開のルートを指定する
const protectedRoutes = ['/dashboard']
const publicRoutes = ['/login', '/signup', '/']

export default async function middleware(req: NextRequest) {
  // 2. 現在のルートが保護されたルートか公開のルートかをチェックする
  const path = req.nextUrl.pathname
  const isProtectedRoute = protectedRoutes.includes(path)
  const isPublicRoute = publicRoutes.includes(path)

  // 3. cookieからセッションを復号する
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  // 4. ユーザーが認証されていない場合、/login にリダイレクトする
  if (isProtectedRoute && !session?.userId) {
    return NextResponse.redirect(new URL('/login', req.nextUrl))
  }

  // 5. ユーザーが認証されている場合、/dashboard にリダイレクトする
  if (
    isPublicRoute &&
    session?.userId &&
    !req.nextUrl.pathname.startsWith('/dashboard')
  ) {
    return NextResponse.redirect(new URL('/dashboard', req.nextUrl))
  }

  return NextResponse.next()
}

// ミドルウェアは実行されるべきではないルート
export const config = {
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

</TabItem>

```js title="middleware.js" switcher
  matcher: ['/((?!api|_next/static|_next/image|.*\\.png$).*)'],
}
```

</TabItem>
</Tabs>

ミドルウェアは最初のチェックには役立つかもしれませんが、データを保護する上での唯一の防御の手段であるべきではありません。ほとんどのセキュリティチェックは、データソースにできるだけ近い位置で実施するべきです。詳細については[データアクセスレイヤー](#creating-a-data-access-layer-dal)を参照してください。

> **Tips**:
>
> - ミドルウェアでは、`req.cookies.get('session').value`を使用して cookieを読むこともできます。
> - ミドルウェアは[Edge Runtime](/docs/app/building-your-application/rendering/edge-and-nodejs-runtimes)を使用しています。Authライブラリとセッション管理ライブラリが互換性があるかどうか確認してください。
> - ミドルウェア内の`matcher`プロパティを使用して、ミドルウェアが実行されるべきルートを指定できます。ただし、認証のためには、ミドルウェアがすべてのルートで実行されることをお勧めします。

### データアクセスレイヤー(DAL)の作成 {#creating-a-data-access-layer-dal}

データリクエストと認可ロジックを集約するためのDALを作成することをお勧めします。

DALには、アプリケーションと対話する際にユーザーのセッションを検証する関数を含むべきです。その最小限として、関数はセッションが有効かどうかを確認し、次のリクエストを行うためにユーザー情報をリダイレクトまたは返します。

たとえば、DAL用に別ファイルを作成し、`verifySession()`関数を含めます。そして、Reactの[キャッシュ](https://react.dev/reference/react/cache)APIを使用して、Reactのレンダー中に関数の戻り値をメモ化します：

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/dal.ts" switcher
import 'server-only'

import { cookies } from 'next/headers'
import { decrypt } from '@/app/lib/session'

export const verifySession = cache(async () => {
  const cookie = (await cookies()).get('session')?.value
  const session = await decrypt(cookie)

  if (!session?.userId) {
    redirect('/login')
  }

  return { isAuth: true, userId: session.userId }
})
```

</TabItem>
<TabItem value="js" label="JavaScript">

```js title="app/lib/dal.js" switcher
import 'server-only'

```

</TabItem>
</Tabs>

`verifySession()`関数は、データリクエスト、Server Actions、 Route Handlersで呼び出すことができます：

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/dal.ts" switcher
export const getUser = cache(async () => {
  const session = await verifySession()
  if (!session) return null

  try {
    const data = await db.query.users.findMany({
      where: eq(users.id, session.userId),
      // 必要なカラムを明示的に返す、ユーザーオブジェクト全体ではなく

        id: true,
        name: true,
        email: true,
      },
    })

    const user = data[0]

    return user
  } catch (error) {
    console.log('ユーザーの取得に失敗しました')
    return null
  }
})
```

</TabItem>

> **Tip**:
>
> - DALは、リクエスト時にデータを保護するために使用できます。ただし、ユーザー間でデータを共有する静的ルートの場合、データはビルド時に取得され、リクエスト時には取得されません。[ミドルウェア](#optimistic-checks-with-middleware-optional)を使用して静的ルートを保護します。
> - セキュアなチェックのために、データベースとセッションIDを比較してセッションが有効かどうかを確認できます。Reactの[キャッシュ](https://react.dev/reference/react/cache)関数を使用して、レンダリング中にデータベースへの重複リクエストを避けます。
> - 関連するデータリクエストをJavaScriptクラスにまとめ、`verifySession()`を実行してからメソッドを実行することを検討するかもしれません。

### データ転送オブジェクト（DTO）の使用 {#using-data-transfer-objects-dto}

データを取得する際には、アプリケーションで使用される必要最小限のデータのみを返し、オブジェクト全体ではなく、たとえば、ユーザーデータを取得する際、ユーザーのIDと名前のみを返します。これは、パスワード、電話番号などを含む可能性があるユーザーオブジェクト全体ではない。

ただし、データの構造を制御できない場合、または、クライアントに渡されることのないオブジェクト全体を避けたい場合は、DTOを使用して、クライアントに公開することが安全なフィールドを特定するような戦略を使用することができます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/lib/dto.ts" switcher
import 'server-only'
import { getUser } from '@/app/lib/dal'

function canSeeUsername(viewer: User) {
  return true
}

function canSeePhoneNumber(viewer: User, team: string) {
  return viewer.isAdmin || team === viewer.team
}

export async function getProfileDTO(slug: string) {
  const data = await db.query.users.findMany({
    where: eq(users.slug, slug),
    // ここに特定のカラムを返す
  })
  const user = data[0]

  const currentUser = await getUser(user.id)

  // または、クエリに固有のものを返す
  return {
    username: canSeeUsername(currentUser) ? user.username : null,
    phonenumber: canSeePhoneNumber(currentUser, user.team)
      ? user.phonenumber
      : null,
  }
}
```

</TabItem>
<TabItem value="js" label="JavaScript">


> **Good to know**:
>
> - DTOを定義する方法はいくつかあり、`toJSON()`を使用することもでき、上記の例のように個別の関数を使用することもでき、JSクラスも利用できます。これらはJavaScriptのパターンであり、ReactまたはNext.jsの機能ではないため、あなたのアプリケーションに最適なパターンを見つけるためにリサーチを行うことをお勧めします。
> - [Next.js ecurityに関する記事](https://nextjs.org/blog/security-nextjs-server-components-actions)で、セキュリティのベストプラクティスについて詳しく学びましょう。

### Server Component {#server-components}

[Server Component](/docs/app/building-your-application/rendering/server-components)での認証チェックは、役割ベースのアクセスに便利です。たとえば、ユーザーの役割に基づいてコンポーネントを条件付きでレンダリングするために：

### レイアウトと認証チェック {#layouts-and-auth-checks}

[部分的なレンダリング](/docs/app/building-your-application/routing/linking-and-navigating#4-partial-rendering)のために[レイアウト](/docs/app/building-your-application/routing/layouts-and-templates)でのチェックには注意が必要です。これらはナビゲーションで再レンダリングされないため、ユーザーセッションはすべてのルート変更でチェックされません。

代わりに、データソースや条件付きレンダリングされるコンポーネントに近い場所でチェックを行うべきです。

たとえば、共有レイアウトを考えてみてください。これがユーザーデータを取得し、ナビゲーションにユーザー画像を表示します。認証チェックをレイアウト内で行うのではなく、レイアウト内でユーザーデータ（`getUser()`）を取得し、DALで認証チェックを行うべきです。

このことは、アプリケーション内で`getUser()`が呼び出された場所であればどこでも認証チェックが行われることを保証し、ユーザーがデータにアクセスする権限があることを確認し忘れることを防ぎます。

### Server Actions {#server-actions}

[Server Actions](/docs/app/building-your-application/data-fetching/server-actions-and-mutations)はパブリックなAPIエンドポイントと同じセキュリティの考慮を持ち、ユーザーがミューテーションを実行することが許可されているかどうかを確認します。

下記の例では、アクションを進行許可する前にユーザーの役割をチェックしています：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/lib/actions.ts" switcher
'use server'
import { verifySession } from '@/app/lib/dal'

export async function serverAction(formData: FormData) {
  const session = await verifySession()
  const userRole = session?.user?.role

  // ユーザーがアクションを実行する権限がない場合、早期に終了する
  if (userRole !== 'admin') {
    return null
  }

  // 許可されたユーザーに対してアクションを実行
}
```

### Route Handlers {#route-handlers}

[ルートハンドラ](/docs/app/building-your-application/routing/route-handlers)は、パブリックなAPIエンドポイントと同じセキュリティの考慮を持ち、ユーザーがルートハンドラへのアクセスが許可されているかどうかを確認します。

例：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/api/route.ts" switcher
import { verifySession } from '@/app/lib/dal'

export async function GET() {
  // ユーザー認証と役割の検証
  const session = await verifySession()

  // ユーザーが認証されているかどうかを確認します
  if (!session) {
    // ユーザーは認証されていません
    return new Response(null, { status: 401 })
  }

  // ユーザーが「admin」役割を持っているかどうかを確認します
  if (session.user.role !== 'admin') {
    // ユーザーは認証されていますが、適切な権限がありません
    return new Response(null, { status: 403 })
  }

  // 許可されたユーザーに対して続行する
}
```

#### APIルートの保護 {#protecting-api-routes}

Next.jsのAPIルートは、サーバーサイドのロジックとデータ管理を扱うために不可欠です。これらのルートを保護し、特定の機能へのアクセスを許可されているユーザーのみが行うことを確に重要です。これは通常、ユーザーの認証ステータスと役割ベースの権限を検証することを伴います。

以下は、APIルートを保護する例です：

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/route.ts" switcher
import { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  const session = await getSession(req)

  // ユーザーが認証されているかを確認します
  if (!session) {
    res.status(401).json({
      error: 'ユーザーが認証されていません',
    })
    return
  }

  // ユーザーが 'admin' 役割を持っているかを確認します
  if (session.user.role !== 'admin') {
    res.status(401).json({
      error: '未承認のアクセス：ユーザーはadminの権限を持っていません。',
    })
    return
  }

  // 許可されたユーザーのためにルートを続行する
  // ... APIルートの実装
}
```
