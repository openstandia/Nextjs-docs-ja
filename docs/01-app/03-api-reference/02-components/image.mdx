export default function Page() {
  return <Image src="/profile.png" width={500} height={500} alt="著者の写真" />
}

## 必須Props {#required-props}

画像コンポーネントには次のプロパティが必要です：`src`、`alt`、`width`と`height`（もしくは`fill`）

```jsx title="app/page.js"
import Image from 'next/image'

export default function Page() {
  return (
    <div>
      <Image src="/profile.png" width={500} height={500} alt="著者の写真" />
    </div>
  )
}
```

### `src` {#src}

次のいずれかでなければなりません：

- [静的にインポートされた](/docs/app/building-your-application/optimizing/images#local-images)画像ファイル
- パス文字列｡これは、絶対外部URLまたは内部パスのいずれかであり、[loader](#loader) propによります｡

デフォルトの[loader](#loader)を使用する場合、ソース画像について次の点も考慮してください：

- srcが外部URLの場合、[remotePatterns](#remotepatterns)も設定する必要があります
- srcが[アニメーション](#animated-images)または既知の形式（JPEG、PNG、WebP、AVIF、GIF、TIFF）でない場合、画像はそのまま提供されます
- srcがSVG形式の場合、[`unoptimized`](#unoptimized)や[`dangerouslyAllowSVG`](#dangerouslyallowsvg)が有効でない限りブロックされます

### `width` {#width}

`width`プロパティはピクセル単位で画像の*元の*幅を表します｡このプロパティは、画像の正しいアスペクト比を推測し、読み込み中のレイアウトシフトを防ぐために使用されます｡これは、画像の表示される実際のサイズを決定するものではなく、HTMLの`<img>`タグの`width`属性と同様にCSSによって制御されます｡

[静的にインポートされた画像](/docs/app/building-your-application/optimizing/images#local-images)か[`fill`プロパティ](#fill)を持つ画像以外で必須です｡

### `height` {#height}

`height`プロパティはピクセル単位で画像の*元の*高さを表します｡このプロパティは、画像の正しいアスペクト比を推測し、読み込み中のレイアウトシフトを防ぐために使用されます｡これは、画像の表示される実際のサイズを決定するものではなく、HTMLの`<img>`タグの`height`属性と同様にCSSによって制御されます｡

[静的にインポートされた画像](/docs/app/building-your-application/optimizing/images#local-images)か[`fill`プロパティ](#fill)を持つ画像以外で必須です｡

> **知っておくと便利です：**
>
> - `width`と`height`プロパティを組み合わせて、画像のアスペクト比を決定し、ブラウザが画像を読み込む前にそのスペースを確保するために使用されます｡
> - 元のサイズはブラウザでの表示サイズと必ずしも同じではなく、親コンテナによって決まります｡例えば、親コンテナが元のサイズよりも小さい場合、画像はコンテナに合わせて縮小されます｡
> - 幅と高さが不明な場合、[`fill`プロパティ](#fill)を使用できます｡

### `alt` {#alt}

`alt`プロパティは、スクリーンリーダーや検索エンジンのために画像を説明するために使用されます｡また、画像が無効になっている場合や、読み込み中にエラーが発生した場合の代替テキストとしても使用されます｡

このプロパティは、画像を[ページの意味を変更せずに置き換えることができるテキスト](https://html.spec.whatwg.org/multipage/images.html#general-guidelines)を含むべきです｡これは画像を補完することを意図したものではなく、画像の上または下にあるキャプションですでに提供されている情報を繰り返すべきではありません｡

もし画像が[純粋に装飾的なもの](https://html.spec.whatwg.org/multipage/images.html#a-purely-decorative-image-that-doesn't-add-any-information)である場合、または[ユーザーを対象としない](https://html.spec.whatwg.org/multipage/images.html#an-image-not-intended-for-the-user)場合、`alt`プロパティは空の文字列（`alt=""`）にするべきです｡

[詳細を学ぶ](https://html.spec.whatwg.org/multipage/images.html#alt)

## オプションのProps {#optional-props}

`<Image />`コンポーネントは、必須のプロパティ以外にいくつかの追加のプロパティを受け入れます｡このセクションでは、最も一般的に使用される画像コンポーネントのプロパティについて説明します｡あまり使用されないプロパティの詳細については[高度なProps](#advanced-props)セクションをご覧ください｡

### `loader` {#loader}

画像URLを解決するために使用されるカスタム関数です｡

`loader`は、以下のパラメータを与えるURL文字列を返す関数です：

- [`src`](#src)
- [`width`](#width)
- [`quality`](#quality)

ここでは、カスタムローダーを使用する例を示します：

<AppOnly>

```js
'use client'

import Image from 'next/image'

const imageLoader = ({ src, width, quality }) => {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`
}

export default function Page() {
  return (
    <Image
      loader={imageLoader}
      src="me.png"
      alt="著者の写真"
      width={500}
      height={500}
    />
  )
}
```

> **知っておくと便利です**: `loader`のような関数を受け入れるpropsを使用する場合、提供された関数をシリアライズするために[Client Components](/docs/app/building-your-application/rendering/client-components)を使用する必要があります｡

</AppOnly>

<PagesOnly>

```js
import Image from 'next/image'

const imageLoader = ({ src, width, quality }) => {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`
}

export default function Page() {
  return (
    <Image
      loader={imageLoader}
      src="me.png"
      alt="著者の写真"
      width={500}
      height={500}
    />
  )
}
```

</PagesOnly>

また、`next.config.js`で[loaderFile](#loaderfile)の設定を使用して、アプリケーション内の`next/image`のインスタンスを個別にpropsを渡さずに設定することもできます｡

### `fill` {#fill}

```js
fill={true} // {true} | {false}
```

画像が親要素を埋めるようにするブール値で、[`width`](#width)と[`height`](#height)が不明な場合に便利です｡

親要素は`position: "relative"`, `position: "fixed"`, もしくは`position: "absolute"`スタイルを割り当てる*必要が*あります｡

デフォルトで，img要素は自動的に`position: "absolute"`スタイルが割り当てられます｡

画像にスタイルが適用されていない場合、画像は
コンテナに合わせて拡張されます｡コンテナに合わせてレターボックス化された画像を保持したアスペクト比で表示するために、`object-fit: "contain"`を設定するのをお勧めします｡また`object-fit: "cover"`は画像全体をコンテナに収め、アスペクト比を維持するためにクロップされます｡

詳細については以下もご参照ください：

- [`position`](https://developer.mozilla.org/docs/Web/CSS/position)
- [`object-fit`](https://developer.mozilla.org/docs/Web/CSS/object-fit)
- [`object-position`](https://developer.mozilla.org/docs/Web/CSS/object-position)

### `sizes` {#sizes}

異なるブレークポイントで画像がどれほどの幅になるかどうかの情報を提供するメディアクエリに似た文字列｡画像に[`fill`](#fill)が使用されている場合や[レスポンシブサイズにスタイリングされている](#responsive-images)画像に対して、`sizes`の値は性能に大きく影響します｡

`sizes`プロパティは、画像のパフォーマンスに関する2つの重要な目的を果たします：

- 第一に、`sizes`の値は、ブラウザが`next/image`が自動生成した`srcset`からダウンロードする
  画像のサイズを決定するために使用されます｡ブラウザが選ぶときは画像のページ上のサイズをまだ知っていないので、ビューポートと同じかそれ以上のサイズの画像を選択します｡`sizes`プロパティを使用して、画像が実際には画面全体よりも小さくなることをブラウザに伝えることができます｡フィルプロパティを使用した画像に`sizes`値を指定しない場合、デフォルト値の`100vw`（全画面幅）が使用されます｡
- 第二に、`sizes`プロパティが自動生成された`srcset`値の動作を変更します｡もし`sizes`値が存在しない場合、固定サイズの画像（1x/2x/etc）に適した小さな`srcset`が生成されます｡もし`sizes`が定義されていれば、レスポンシブ画像（640w/750w/etc）に適した大きな`srcset`が生成されます｡sizesプロパティが`50vw`のようなビューポート幅のパーセンテージを含んでいる場合、srcsetに含まれる値が必要以上に小さくならないように、`srcset`がトリミングされます｡

例えば、スタイリングがモバイルデバイスで画像を全幅、タブレットで2列レイアウト、デスクトップディスプレイで3列レイアウトになることを知っている場合は、以下のようなsizesプロパティを含むべきです：

```jsx
import Image from 'next/image'

export default function Page() {
  return (
    <div className="grid-element">
      <Image
        fill
        src="/example.png"
        sizes="(max-width: 768px) 100vw, (max-width: 1200px) 50vw, 33vw"
      />
    </div>
  )
}
```

この`sizes`の例は、パフォーマンスメトリクスに劇的な影響を与える可能性があります｡`33vw`のsizesがなければ、サーバーから選択された画像は必要な幅の3倍になるでしょう｡ファイルサイズは幅の二乗に比例するため、それなしではユーザーは必要なサイズの9倍の大きさの画像をダウンロードすることになります｡

`srcset`と`sizes`の詳細を学ぶ：

- [web.dev](https://web.dev/learn/design/responsive-images/#sizes)
- [mdn](https://developer.mozilla.org/docs/Web/HTML/Element/img#sizes)

### `quality` {#quality}

```js
quality={75} // {number 1-100}
```

最適化された画像の品質で、1から100の整数で、100が最良の品質であり、したがって最大のファイルサイズになります｡デフォルトは`75`です｡

### `priority` {#priority}

```js
priority={false} // {false} | {true}
```

`true`の場合、画像は高い優先度を持つと見なされ
[プリロード](https://web.dev/preload-responsive-images/)されます｡`priority`が使用されている画像の遅延読み込みは自動的に無効化されます｡`priority`プロパティが必要である場合、[`loading`](#loading)プロパティを使用できません｡`loading`プロパティは、高度なユースケース向だけです｡`priority`が必要な時は`loading`を削除してください｡

優先プロパティは、[Largest Contentful Paint (LCP)](https://nextjs.org/learn/seo/web-performance/lcp)要素として検出された画像に使用するべきです｡異なるビューポートサイズの場合、異なる画像がLCP要素になることがあるため、複数の優先画像が適切かもしれません｡

優先プロパティは、折りたたまれる前に画像が表示されている場合にのみ使用するべきです｡デフォルトでは`false`です｡

### `placeholder` {#placeholder}

```js
placeholder = 'empty' // "empty" | "blur" | "data:image/..."
```

画像が読み込まれている間に使用するプレースホルダー｡可能な値は`blur`、`empty`、または`data:image/...`
であり、デフォルトは`empty`です｡

`blur`の場合、[`blurDataURL`](#blurdataurl)プロパティがプレースホルダーとして使用されます｡もし`src`が[静的インポート](/docs/app/building-your-application/optimizing/images#local-images)からのオブジェクトであり、インポートされた画像が`.jpg`, `.png`, `.webp`, または`.avif`であれば、`blurDataURL`は自動的に設定されます｡例外は画像がアニメーション化されていると検出された場合です｡

動的画像の場合、[`blurDataURL`](#blurdataurl)プロパティを提供する必要があります｡ [Plaiceholder](https://github.com/joe-bell/plaiceholder)のようなソリューションは`base64`の生成に役立ちます｡

`data:image/...`の場合、[データURL](https://developer.mozilla.org/docs/Web/HTTP/Basics_of_HTTP/Data_URIs)が
画像が読み込まれている間のプレースホルダーとして使用されます｡

`empty`の場合、画像が読み込まれている間にプレースホルダーは使用されず、空白のみです｡

試してみる：

- [`blur`プレースホルダーをデモ](https://image-component.nextjs.gallery/placeholder)
- データURL`プレースホルダー`プロプでのシマーエフェクトをデモ](https://image-component.nextjs.gallery/shimmer)
- [`blurDataURL` プロプでの色エフェクトをデモ](https://image-component.nextjs.gallery/color)

## 高度なProps {#advanced-props}

場合によっては、より高度な使用が必要になるかもしれません｡`<Image />` コンポーネントは以下の高度なプロパティを受け入れます｡

### `style` {#style}

基礎の画像要素にCSSスタイルを渡すことを可能にします｡

```jsx title="components/ProfileImage.js"
const imageStyle = {
  borderRadius: '50%',
  border: '1px solid #fff',
}

export default function ProfileImage() {
  return <Image src="..." style={imageStyle} />
}
```

必須のwidthとheightプロパティがスタイリングに影響を与えることも覚えておいてください｡画像の幅を変更するためにスタイリングを使用する場合は、その画像の高さも`auto`にして、その固有のアスペクト比を保持するようにスタイリングするべきです。でないと、画像が歪んで表示されます｡

### `onLoadingComplete` {#onloadingcomplete}

<AppOnly>

```jsx
'use client'

<Image onLoadingComplete={(img) => console.log(img.naturalWidth)} />
```

</AppOnly>

<PagesOnly>

```jsx
<Image onLoadingComplete={(img) => console.log(img.naturalWidth)} />
```

</PagesOnly>

> **警告**: Next.js 14から`onLoad`を優先し廃止されました｡

画像が完全に読み込まれ、[プレースホルダー](#placeholder)が削除されたときに呼び出されるコールバック関数｡

このコールバック関数は、1つの引数である元の`<img>`要素への参照とともに呼び出されます｡

<AppOnly>

> **知っておくと便利です**: `onLoadingComplete`のような関数を受け入れるpropsを使用する場合、提供された関数をシリアライズするために[Client Components](/docs/app/building-your-application/rendering/client-components)を使用する必要があります｡

</AppOnly>

### `onLoad` {#onload}

```jsx
<Image onLoad={(e) => console.log(e.target.naturalWidth)} />
```

画像が完全に読み込まれ、[プレースホルダー](#placeholder)が削除されたときに呼び出されるコールバック関数｡

このコールバック関数は、`target`が元の`<img>`要素を参照しているイベントとともに呼び出されます｡

<AppOnly>

> **知っておくと便利です**: `onLoad`のような関数を受け入れるpropsを使用する場合、提供された関数をシリアライズするために[Client Components](/docs/app/building-your-application/rendering/client-components)を使用する必要があります｡

</AppOnly>

### `onError` {#onerror}

```jsx
<Image onError={(e) => console.error(e.target.id)} />
```

画像の読み込みが失敗した場合に呼び出されるコールバック関数｡

<AppOnly>

> **知っておくと便利です**: `onError`のような関数を受け入れるpropsを使用する場合、提供された関数をシリアライズするために[Client Components](/docs/app/building-your-application/rendering/client-components)を使用する必要があります｡

</AppOnly>

### `loading` {#loading}

```js
loading = 'lazy' // {lazy} | {eager}
```

画像の読み込み動作｡デフォルトは`lazy`｡

`lazy`の場合、画像が視界から一定の距離に到達するまで画像の読み込みを延期します｡

`eager`の場合、画像を即座に読み込みます｡

[`loading`属性について学ぶ](https://developer.mozilla.org/docs/Web/HTML/Element/img#loading)

### `blurDataURL` {#blurdataurl}

[srcの画像が正常にロードされる前のプレースホルダー画像として使用される[データURL](https://developer.mozilla.org/docs/Web/HTTP/Basics_of_HTTP/Data_URIs)｡[`placeholder="blur"`](#placeholder)と組み合わせた場合のみ効果を発揮します｡

base64エンコードされた画像である必要があります｡拡大されてぼかされるため、非常に小さい画像（10px以下）が推奨されます｡大きな画像をプレースホルダーとして含めると、アプリケーションのパフォーマンスに悪影響を与える可能性があります｡

試してみる：

- デフォルトの`blurDataURL`プロパティをデモ](https://image-component.nextjs.gallery/placeholder)
- [`blurDataURL` プロプの色エフェクトをデモ](https://image-component.nextjs.gallery/color)

画像に一致するソリッドカラーのデータURLを生成することもできます｡

### `unoptimized` {#unoptimized}

```js
unoptimized = {false} // {false} | {true}
```

`true`の場合、画像の品質、サイズ、形式を変更するのではなく、`src`から元の画像がそのまま提供されます｡デフォルトは`false`です｡

これは、1KB未満の小さな画像、SVGのようなベクター画像、GIFのようなアニメーション画像など、最適化の恩恵を受けない画像に便利です｡

```js
import Image from 'next/image'

const UnoptimizedImage = (props) => {
  return <Image {...props} unoptimized />
}
```

Next.js 12.3.0以降、このプロパティは次の設定で`next.config.js`を更新することにより、すべての画像に割り当てることができます：

```js title="next.config.js"
module.exports = {
  images: {
    unoptimized: true,
  },
}
```

### `overrideSrc` {#overridesrc}

`<Image>` コンポーネントに`src` プロップを提供する場合、`srcset` 属性と`src` 属性の両方が生成されます｡

```jsx title="input.js"
<Image src="/me.jpg" />
```

```html title="output.html"
<img
  srcset="
    /_next/image?url=%2Fme.jpg&w=640&q=75 1x,
    /_next/image?url=%2Fme.jpg&w=828&q=75 2x
  "
  src="/_next/image?url=%2Fme.jpg&w=828&q=75"
/>
```

場合によっては、生成された`src`属性が望ましくないことがあり、`overrideSrc`プロップを使用して上書きしたい場合があります｡

たとえば、既存のウェブサイトを`<img>`から`<Image>`にアップグレードする際、画像ランキングや再クロール回避などのSEO目的で同じ`src`属性を維持したい場合があります｡

```jsx title="input.js"
<Image src="/me.jpg" overrideSrc="/override.jpg" />
```

```html title="output.html"
<img
  srcset="
    /_next/image?url=%2Fme.jpg&w=640&q=75 1x,
    /_next/image?url=%2Fme.jpg&w=828&q=75 2x
  "
  src="/override.jpg"
/>
```

### decoding {#decoding}

ブラウザに、他のコンテンツの更新を示す前に画像をデコードするべきか待つべきかのヒントを与える｡デフォルトは`async`｡

可能な値は以下です：

- `async` - 非同期で画像をデコードし、完了する前に他のコンテンツをレンダリングします｡
- `sync` - 他のコンテンツとコミットで表示するために同期で画像をデコードします｡
- `auto` - デコードモードに対する明確な優先はなく、ブラウザが最適な方法を決定します｡

[`decoding`属性について学ぶ](https://developer.mozilla.org/docs/Web/HTML/Element/img#decoding)

`<Image />`コンポーネントの他のプロパティは、以下を除いて、基礎となる
`img`要素に渡されます：

- `srcSet`. [デバイスサイズ](#devicesizes)を使用してください｡

## 設定オプション {#configuration-options}

propsに加えて、`next.config.js`で画像コンポーネントを設定することができます｡以下のオプションが利用可能です：

## `localPatterns` {#localpatterns}

特定のパスを最適化し、他の全てのパスをブロックするために、`next.config.js`ファイル内で`localPatterns`をオプショナルに設定することができます｡

```js title="next.config.js"
module.exports = {
  images: {
    localPatterns: [
      {
        pathname: '/assets/images/**',
        search: '',
      },
    ],
  },
}
```

> **知っておくと便利です**: 上記の例は、`next/image`の`src`プロパティが`/assets/images/`で始まり、クエリ文字列が含まれていない場合に限り確保されます｡異なるパスを最適化しようとすると、400 Bad Requestで応答します｡

### `remotePatterns` {#remotepatterns}

悪意のあるユーザーからアプリケーションを保護するために、外部画像を使用するためには設定が必要です｡これによりアカウントからの外部画像のみがNext.js画像最適化APIを介して提供されることが保証されます｡これらの外部画像は、以下のように`next.config.js`ファイルの`remotePatterns`プロパティで設定されることができます：

```js title="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'example.com',
        port: '',
        pathname: '/account123/**',
        search: '',
      },
    ],
  },
}
```

> **知っておくと便利です**: 上記の例は、`next/image`の`src`プロパティが`https://example.com/account123/`で始まり、クエリ文字列が含まれていない場合に限り確保されます｡異なるプロトコル、ホスト名、ポート、または一致しないパスは、400 Bad Requestで応答します｡

以下に`hostname`でワイルドカードパターンを使用した`next.config.js`ファイル内の`remotePatterns`プロパティの例を示します：

```js title="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: '**.example.com',
        port: '',
        search: '',
      },
    ],
  },
}
```

> **知っておくと便利です**: 上記の例は、`next/image`の`src`プロパティが`https://img1.example.com`または`https://me.avatar.example.com`、または任意のサブドメインで始まる場合に限り確保されます｡ポートやクエリ文字列が含まれていないこと｡異なるプロトコルまたは一致しないホスト名は、400 Bad Requestで応答します｡

ワイルドカードパターンは`pathname`と`hostname`の両方に使用でき、以下の構文を持ちます：

- `*` は単一のパスセグメントまたはサブドメインに一致します
- `**` は最後の任意の数のパスセグメントまたは先頭のサブドメインに一致します

`**`構文はパターンの中央では機能しません｡

> **知っておくと便利です：** `protocol`、`port`、`search`を省略する場合、ワイルドカード `**` が暗黙のうちに適用されます｡あまり推奨されませんが、それにより悪意のあるアクターが意図しないURLを最適化する可能性があります｡

以下に`search`を使用した`next.config.js`ファイル内の`remotePatterns`プロパティの例を示します：

```js title="next.config.js"
module.exports = {
  images: {
    remotePatterns: [
      {
        protocol: 'https',
        hostname: 'assets.example.com',
        search: '?v=1727111025337',
      },
    ],
  },
}
```

> **知っておくと便利です**: 上記の例は、`next/image`の`src`プロパティが`https://assets.example.com`から始まり、正確に`?v=1727111025337`というクエリ文字列を持つ場合に限り確保されます｡異なるプロトコルまたはクエリ文字列は、400 Bad Requestで応答します｡

### `domains` {#domains}

> **警告**: 悪意のあるユーザーからアプリケーションを保護するために、厳密な [`remotePatterns`](#remotepatterns)を優先してNext.js 14から廃止されました｡`domains`構成を使用するのは、ドメインから提供されるコンテンツをすべて所有している場合のみです｡

[`remotePatterns`](#remotepatterns)と同様に、外部画像の許可されたホスト名リストを提供するために`domains`構成が使用できます｡

ただし、`domains`構成はワイルドカードパターンマッチングをサポートしておらず、プロトコル、ポート、パス名を制限することはできません｡

以下に`next.config.js`ファイル内の`domains`プロパティの例を示します：

```js title="next.config.js"
module.exports = {
  images: {
    domains: ['assets.acme.com'],
  },
}
```

### `loaderFile` {#loaderfile}

Next.jsの組み込み画像最適化APIを使用せずにクラウドプロバイダーを使用して画像を最適化したい場合、次のように`next.config.js`で`loaderFile`を構成できます：

```js title="next.config.js"
module.exports = {
  images: {
    loader: 'custom',
    loaderFile: './my/image/loader.js',
  },
}
```

これはNext.jsアプリケーションのルートから相対的なファイルを指す必要があります｡そのファイルは文字列を返すデフォルト関数をエクスポートする必要があり、例えば次のように：

<AppOnly>

```js title="my/image/loader.js"
'use client'

export default function myImageLoader({ src, width, quality }) {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`
}
```

</AppOnly>

<PagesOnly>

```js title="my/image/loader.js"
export default function myImageLoader({ src, width, quality }) {
  return `https://example.com/${src}?w=${width}&q=${quality || 75}`
}
```

</PagesOnly>

あるいは、[`loader`プロップ](#loader)を使用して`next/image`の各インスタンスを設定できます｡

例：

- [カスタム画像ローダーの設定](/docs/app/api-reference/config/next-config-js/images#example-loader-configuration)

<AppOnly>

> **知っておくと便利です**: 関数を受け入れる画像のローダーファイルをカスタマイズするには、提供された関数をシリアライズするために[Client Components](/docs/app/building-your-application/rendering/client-components)を使用する必要があります｡

</AppOnly>

## 高度な設定 {#advanced}

以下の設定は高度なユースケース向けで、通常は必要ありません｡以下のプロパティを設定する場合、将来のアップデートでNext.jsのデフォルトの変更を上書きします｡

### `deviceSizes` {#devicesizes}

ユーザーの予想されるデバイス幅を知っている場合、`next.config.js`で`deviceSizes`プロパティを使用してデバイスの幅のブレークポイントリストを指定できます｡これらの幅は、`next/image`コンポーネントが[`sizes`](#sizes)プロップを使用する際に、ユーザーのデバイスに正しい画像が提供されることを保証するために使用されます｡

設定が提供されていない場合、以下のデフォルトが使用されます｡

```js title="next.config.js"
module.exports = {
  images: {
    deviceSizes: [640, 750, 828, 1080, 1200, 1920, 2048, 3840],
  },
}
```

### `imageSizes` {#imagesizes}

`next.config.js`ファイルで`images.imageSizes`プロパティを使用して画像幅のリストを指定できます｡これらの幅は、[deviceSizes](#devicesizes)との連結により、画像[srcset](https://developer.mozilla.org/docs/Web/API/HTMLImageElement/srcset)を生成する際に使用されるサイズの完全なリストを形成します｡

分かれている2つのリストの理由は、imageSizesは[`sizes`](#sizes)プロップを提供する画像のみで使用され、それが画像が画面全体の幅よりも小さいことを示しているからです｡**したがって、imageSizes内のサイズは、deviceSizesの最小サイズよりも小さいサイズにする必要があります｡**

設定が提供されていない場合、以下のデフォルトが使用されます｡

```js title="next.config.js"
module.exports = {
  images: {
    imageSizes: [16, 32, 48, 64, 96, 128, 256, 384],
  },
}
```

### `formats` {#formats}

デフォルトの[画像最適化API]（#loader）は、リクエストの`Accept`ヘッダーを介してブラウザがサポートしている画像フォーマットを自動検出し、最適な出力形式を決定します｡

もし`Accept`ヘッダーが指定されたフォーマットの一つ以上に一致する場合、配列内の最初の一致が使われます｡したがって、配列の順序は重要です｡マッチがない場合（もしくは、ソース画像が[アニメーション化されている場合](#animated-images))、画像最適化APIは元の画像のフォーマットに戻ります｡

設定が提供されていない場合、以下のデフォルトが使用されます｡

```js title="next.config.js"
module.exports = {
  images: {
    formats: ['image/webp'],
  },
}
```

以下の設定でAVIFサポートを有効にしてWebPにフォールバックすることもできます｡

```js title="next.config.js"
module.exports = {
  images: {
    formats: ['image/avif', 'image/webp'],
  },
}
```

> **知っておくと便利です：**
>
> - AVIFはエンコードに通常比50％長くかかりますが、WebPと比較して20％小さく圧縮されます｡これは、画像が最初にリクエストされるときは通常よりも遅くなり、キャッシュされて次のリクエストが行われると速くなります。
> - Proxy/CDNを Next.js の前に置いて自己ホスティングする場合、Proxy に `Accept` ヘッダーを転送するよう構成する必要があります。

## キャッシング動作 {#caching-behavior}

デフォルトの[loader](#loader)のためのキャッシングアルゴリズムを以下に説明します｡その他のローダーについては、クラウドプロバイダーのドキュメントを参照してください｡

images はリクエスト時に動的に最適化され、`/distDir>/cache/images` ディレクトリに保存されます｡この最適化された画像ファイルは、期限が満了するまでその後のリクエストに対して提供されます｡キャッシュされたが期限切れのファイルと一致するリクエストが行われた場合、期限切れの画像はすぐに古い状態 (STALE) として提供されます。その後、画像はバックグラウンドで再度最適化され（再検証とも呼ばれる）、新しい有効期限でキャッシュに保存されます｡

画像のキャッシュステータスは、`x-nextjs-cache`レスポンスヘッダーの値を読むことで確認できます｡可能な値は以下です：

- `MISS` - キャッシュにパスが存在しない（初回訪問時に一度だけ発生）
- `STALE` - キャッシュにパスが存在するが再検証時間を超えたため、バックグラウンドで更新される
- `HIT` - キャッシュにパスが存在し、再検証時間を超えていない

期限切れ（または正確には最大保存期間）は、[`minimumCacheTTL`](#minimumcachettl) 設定または上流画像のキャッシュ制御ヘッダーのどちらか大きい方によって定義されます｡具体的には、キャッシュ制御ヘッダーの`max-age`値が使用されます｡`s-maxage`と`max-age`の両方が見つかった場合、`s-maxage`が優先されます｡`max-age`は、CDNやブラウザを含む下流のクライアントにも通過されます｡

- 元画像にキャッシュ制御ヘッダーが含まれていない、もしくは非常に低い場合に期限を延ばすために、[`minimumCacheTTL`](#minimumcachettl)を設定することが出来ます｡
- [`deviceSizes`](#devicesizes)と[`imageSizes`](#imagesizes)を設定して、生成される可能性のある画像の総数を減らすことができます｡
- [formats](#formats)を設定して、複数の形式を無効化し、単一の画像形式に変更できます｡

### `minimumCacheTTL` {#minimumcachettl}

キャッシュされた最適化された画像のTTL（存続期間）を秒で構成できます｡多くの場合、ファイルコンテンツを自動的にハッシュし、「cache-control」ヘッダを「immutable」で永遠にキャッシュする[静的画像インポート](/docs/app/building-your-application/optimizing/images#local-images)の方が良いでしょう｡

```js title="next.config.js"
module.exports = {
  images: {
    minimumCacheTTL: 60,
  },
}
```

最適化された画像の消滅（あるいは正確には最大年齢）は、[`minimumCacheTTL`](#minimumcachettl)または上流画像のキャッシュ制御ヘッダーのいずれか大きい方によって定義されます｡

画像ごとにキャシュ動作を変更する必要がある場合、[`headers`](/docs/app/api-reference/config/next-config-js/headers)を設定して、上流の画像（例えば`/some-asset.jpg`、画像自体ではなく）に`Cache-Control`ヘッダーを設定できます｡

現在、キャッシュを無効化するメカニズムは用意されていないため、`minimumCacheTTL`を低く設定することが好ましいです｡そうでない場合、手動で[`src`](#src)属性や`<distDir>/cache/images`を変更する必要がある場合があります｡

### `disableStaticImages` {#disablestaticimages}

デフォルトの動作では、`import icon from './icon.png'`のように静的ファイルをインポートし、それを`src`プロパティに渡すことができます｡

場合によっては、この機能を無効にして、インポートを異なる方法で処理させることを期待する他のプラグインと競合する場合があります｡

`next.config.js`内で静的画像のインポートを無効にすることができます｡

```js title="next.config.js"
module.exports = {
  images: {
    disableStaticImages: true,
  },
}
```

### `dangerouslyAllowSVG` {#dangerouslyallowsvg}

デフォルトの[loader](#loader)は、いくつかの理由でSVG画像を最適化しません｡第一に、SVGはベクトル形式であり、損失なくサイズを変更できるからです｡第二に、SVGはHTML/CSSと同じ機能を持っているため、適切な[コンテンツセキュリティポリシー（CSP）ヘッダ](/docs/app/api-reference/config/next-config-js/headers#content-security-policy)を設定しないと、脆弱性が発生する可能性があります｡

したがって、[`src`](#src)プロパティがSVGであると分かっている場合は、[unoptimized](#unoptimized)プロ兵を使用することをお勧めします｡これは、`src`が`".svg"`で終わる場合に自動的に実行されます｡

しかしながら、デフォルト画像最適化APIでSVG画像を提供する必要がある場合、`next.config.js`内で`dangerouslyAllowSVG`を設定することができます｡

```js title="next.config.js"
module.exports = {
  images: {
    dangerouslyAllowSVG: true,
    contentDispositionType: 'attachment',
    contentSecurityPolicy: "default-src 'self'; script-src 'none'; sandbox;",
  },
}
```

加えて、画像に埋め込まれたスクリプトの実行を防ぐために、画像を強制的にブラウザにダウンロードさせるため、`contentDispositionType`を設定し、強く`内容セキュリティポリシー`を設定することをお勧めします｡

### `contentDispositionType` {#contentdispositiontype}

デフォルトの[loader](#loader)は、APIが任意のリモート画像を提供できるため、保護を強化するために[`Content-Disposition`](https://developer.mozilla.org/en-US/docs/Web/HTTP/Headers/Content-Disposition#as_a_response_header_for_the_main_body)ヘッダーを`attachment`に設定します｡

デフォルト値は`attachment`であり、これによりブラウザが直接訪問時に画像をダウンロードすることを強制します｡[`dangerouslyAllowSVG`](#dangerouslyallowsvg)がtrueである場合、これは特に重要です｡

画像をダウンロードせずに直接訪問時にブラウザに画像を表示することを許可するために、独立して設定できます｡

```js title="next.config.js"
module.exports = {
  images: {
    contentDispositionType: 'inline',
  },
}
```

## アニメーション画像 {#animated-images}

デフォルトの[loader](#loader)は、アニメーション画像に対して画像最適化を自動的にバイパスし、そのまま画像を提供します｡

アニメーションファイルの自動検出はベストエフォートで行われ、GIF、APNG、およびWebPがサポートされます｡特定のアニメーション画像の画像最適化を明示的にバイパスしたい場合、[unoptimized](#unoptimized)プロップを使用してください｡

## レスポンシブ画像 {#responsive-images}

デフォルトで生成された`srcset`には、異なるデバイスピクセル比をサポートするための`1x`と`2x`の画像が含まれています｡ただし、ビューポートに応じて伸縮するレスポンシブ画像をレンダリングしたい場合があります｡その場合、`sizes`だけでなく`style`（または`className`）を設定する必要があります｡

以下の方法でレスポンシブ画像をレンダリングできます｡

### 静的インポートを使用したレスポンシブ画像 {#responsive-image-using-a-static-import}

もしソース画像が動的でない場合、レスポンシブ画像を作成するために静的インポートすることができます：

```jsx title="components/author.js"
import Image from 'next/image'
import me from '../photos/me.jpg'

export default function Author() {
  return (
    <Image
      src={me}
      alt="著者の写真"
      sizes="100vw"
      style={{
        width: '100%',
        height: 'auto',
      }}
    />
  )
}
```

試してみる：

- [ビューポートに応じたレスポンシブ画像のデモ](https://image-component.nextjs.gallery/responsive)

### アスペクト比を持つレスポンシブ画像 {#responsive-image-with-aspect-ratio}

もしソース画像が動的またはリモートURLである場合、レスポンシブ画像の正しいアスペクト比を設定するために`width`と`height`を提供する必要があります：

```jsx title="components/page.js"
import Image from 'next/image'

export default function Page({ photoUrl }) {
  return (
    <Image
      src={photoUrl}
      alt="著者の写真"
      sizes="100vw"
      style={{
        width: '100%',
        height: 'auto',
      }}
      width={500}
      height={300}
    />
  )
}
```

試してみる：

- [ビューポートに応じたレスポンシブ画像のデモ](https://image-component.nextjs.gallery/responsive)

### `fill`を使用したレスポンシブ画像 {#responsive-image-with-fill}

アスペクト比が不明な場合は、親要素に[`fill`](#fill)プロップを設定し、`position: relative`を設定する必要があります｡必要に応じて、目的のストレッチとクロップの動作に応じて`object-fit`スタイルを設定できます：

```jsx title="app/page.js"
import Image from 'next/image'

export default function Page({ photoUrl }) {
  return (
    <div style={{ position: 'relative', width: '300px', height: '500px' }}>
      <Image
        src={photoUrl}
        alt="著者の写真"
        sizes="300px"
        fill
        style={{
          objectFit: 'contain',
        }}
      />
    </div>
  )
}
```

- [`fill`プロップのデモ](https://image-component.nextjs.gallery/fill)

## getImageProps {#getimageprops}

より高度なユースケース向けには、`getImageProps()`を呼び出して、基礎となる`<img>`要素に渡されるプロップを取得し、それを別のコンポーネント、スタイル、キャンバスなどに渡すことができます｡

React `useState()` の呼び出しも回避するため、パフォーマンスの向上につながる可能性がありますが、プレースホルダーが削除されることはないため[`placeholder`](#placeholder)プロップと一緒に使用することはできません｡

### テーマ検出用の画像 {#theme-detection-picture}

ライトモードとダークモードで異なる画像を表示したい場合、ユーザーの[好ましいカラースキーム](https://developer.mozilla.org/en-US/docs/Web/CSS/@media/prefers-color-scheme)に基づいた異なる画像を表示するには、[`<picture>`](https://developer.mozilla.org/docs/Web/HTML/Element/picture)要素を使用することができます｡

```jsx title="app/page.js"
import { getImageProps } from 'next/image'

export default function Page() {
  const common = { alt: 'テーマの例', width: 800, height: 400 }
  const {
    props: { srcSet: dark },
  } = getImageProps({ ...common, src: '/dark.png' })
  const {
    props: { srcSet: light, ...rest },
  } = getImageProps({ ...common, src: '/light.png' })

  return (
    <picture>
      <source media="(prefers-color-scheme: dark)" srcSet={dark} />
      <source media="(prefers-color-scheme: light)" srcSet={light} />
      <img {...rest} />
    </picture>
  )
}
```

### アートディレクション {#art-direction}

モバイルとデスクトップで異なる画像を表示したい場合、[アートディレクション](https://developer.mozilla.org/en-US/docs/Learn/HTML/Multimedia_and_embedding/Responsive_images#art_direction)とも呼ばれることがありますが、異なる`src`, `width`, `height`, と`quality`プロップを`getImageProps()`に提供することができます｡

```jsx title="app/page.js"
import { getImageProps } from 'next/image'

export default function Home() {
  const common = { alt: 'アートディレクション例', sizes: '100vw' }
  const {
    props: { srcSet: desktop },
  } = getImageProps({
    ...common,
    width: 1440,
    height: 875,
    quality: 80,
    src: '/desktop.jpg',
  })
  const {
    props: { srcSet: mobile, ...rest },
  } = getImageProps({
    ...common,
    width: 750,
    height: 1334,
    quality: 70,
    src: '/mobile.jpg',
  })

  return (
    <picture>
      <source media="(min-width: 1000px)" srcSet={desktop} />
      <source media="(min-width: 500px)" srcSet={mobile} />
      <img {...rest} style={{ width: '100%', height: 'auto' }} />
    </picture>
  )
}
```

### 背景CSS {#background-css}

`srcSet`文字列を[`image-set()`](https://developer.mozilla.org/en-US/docs/Web/CSS/image/image-set) CSS関数に変換して、背景画像を最適化することもできます｡

```jsx title="app/page.js"
import { getImageProps } from 'next/image'

function getBackgroundImage(srcSet = '') {
  const imageSet = srcSet
    .split(', ')
    .map((str) => {
      const [url, dpi] = str.split(' ')
      return `url("${url}") ${dpi}`
    })
    .join(', ')
  return `image-set(${imageSet})`
}

export default function Home() {
  const {
    props: { srcSet },
  } = getImageProps({ alt: '', width: 128, height: 128, src: '/img.png' })
  const backgroundImage = getBackgroundImage(srcSet)
  const style = { height: '100vh', width: '100vw', backgroundImage }

  return (
    <main style={style}>
      <h1>Hello World</h1>
    </main>
  )
}
```

## 知られているブラウザの問題 {#known-browser-bugs}

この`next/image`コンポーネントはブラウザのネイティブな[遅延ロード](https://caniuse.com/loading-lazy-attr)を使用し、Safari 15.4以前の古いブラウザに対しては、熱心なロードにフォールバックすることがあります｡ぼかしプレースホルダーを使用する場合、Safari 12以前の古いブラウザは空白プレースホルダーにフォールバックします｡`auto`の`width`と`height`を使用するスタイルを指定すると、Safari 15以前の古いブラウザで[レイアウトシフト](https://web.dev/cls/)を引き起こす可能性があります。これに関しては[このMDNのビデオ](https://www.youtube.com/watch?v=4-d_SoCHeWE)を参照してください｡

- Safari 15 - 16.3は読み込み中に灰色の境界線を表示します｡ Safari 16.4では[この問題が解決しました](https://webkit.org/blog/13966/webkit-features-in-safari-16-4/#:~:text=Now%20in%20Safari%2016.4%2C%20a%20gray%20line%20no%20longer%20appears%20to%20mark%20the%20space%20where%20a%20lazy%2Dloaded%20image%20will%20appear%20once%20it%E2%80%99s%20been%20loaded.). 可能な解決法：
  - CSS `@supports (font: -apple-system-body) and (-webkit-appearance: none) { img[loading="lazy"] { clip-path: inset(0.6px) } }` を使用
  - 画像がスクロールを超えている場合、[`priority`](#priority)を使用
- Firefox 67+ は読み込み中に白い背景を表示します｡ 可能な解決策：
  - AVIF [`formats`](#formats)を有効に
  - [`placeholder`](#placeholder)を使用
