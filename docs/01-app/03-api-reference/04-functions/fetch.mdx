Next.jsは、サーバー上の各リクエストがそれ自身の永続的なキャッシュと再検証の意味を設定できるように、[Web `fetch()` API](https://developer.mozilla.org/docs/Web/API/Fetch_API)を拡張しています。

ブラウザでは、`cache`オプションはfetchリクエストが _ブラウザの_ HTTPキャッシュとどのように相互作用するかを示します。この拡張では、`cache`は _サーバーサイドの_ fetchリクエストがフレームワークの永続的な[Data Cache](/docs/app/building-your-application/caching#data-cache)とどのように相互作用するかを示します。

`async`と`await`を使って、Server Component内で直接`fetch`を呼び出せます。

export default async function Page() {

## `fetch(url, options)` {#fetch-url-options}

Next.jsは[Web `fetch()` API](https://developer.mozilla.org/docs/Web/API/Fetch_API)を拡張しているため、[ネイティブなオプション](https://developer.mozilla.org/docs/Web/API/fetch#parameters)のいずれも使用できます。

### `options.cache` {#options-cache}

Next.jsの[Data Cache](/docs/app/building-your-application/caching#data-cache)とどのように相互作用するかをリクエストに設定します。

```ts
fetch(`https://...`, { cache: 'force-cache' | 'no-store' })
```

- **`auto no cache`** (デフォルト)：Next.jsは開発中のリモートサーバーからのリソースを毎回フェッチしますが、ルートが静的に事前レンダリングされるため、`next build`時には一度だけフェッチします。ルートで[Dynamic APIs](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)が検出された場合、Next.jsは毎回リソースをフェッチします
- **`no-store`**: Dynamic APIがルートで検出されなくても、Next.jsは毎回リモートサーバーからリソースをフェッチします
- **`force-cache`**: Next.jsはそのData Cacheで一致するリクエストを探します。
  - 一致があり、それが新鮮であれば、キャッシュから返されます
  - 一致がないか、一致が古い場合、Next.jsはリモートサーバーからリソースをフェッチし、ダウンロードしたリソースでキャッシュを更新します

### `options.next.revalidate` {#options-next-revalidate}

```ts
fetch(`https://...`, { next: { revalidate: false | 0 | number } })
```

リソースのキャッシュの有効期間を（秒単位で）設定します。

- **`false`** - リソースを無期限にキャッシュします。`revalidate: Infinity`とセマンティックに等価です。HTTPキャッシュは、時間の経過とともに古いリソースを取り除くことがあります
- **`0`** - リソースをキャッシュしないようにします
- **`number`** - （秒単位）リソースに最大`n`秒のキャッシュの有効期間があるべきであることを指定します

> **Good to know**:
>
> - 個別の`fetch()`リクエストがあるルートの[デフォルトの`revalidate`](/docs/app/api-reference/file-conventions/route-segment-config#revalidate)より低い`revalidate`数を設定すると、ルート全体の再検証間隔が短くなります
> - 同じURLで異なる`revalidate`値を持つ2つのfetchリクエストが同じルートにある場合、低い方の値が使用されます
> - 便宜上、`revalidate`が数値に設定されている場合は`cache`オプションを設定する必要はありません
> - `{ revalidate: 3600, cache: 'no-store' }`のような矛盾するオプションはエラーを引き起こします

### `options.next.tags` {#options-next-tags}

```ts
fetch(`https://...`, { next: { tags: ['collection'] } })
```

リソースのキャッシュタグを設定します。データをオンデマンドで再検証できるようになり、[`revalidateTag`](https://nextjs.org/docs/app/api-reference/functions/revalidateTag)を使用できます。カスタムタグの最大長は256文字で、最大タグ項目数は128です。

## トラブルシューティング {#troubleshooting}

### Fetchデフォルトの`auto no store`と`cache: 'no-store'`が開発中に新しいデータを表示しない場合 {#fetch-default-auto-no-store-and-cache-no-store-not-showing-fresh-data-in-development}

Next.jsは、ローカル開発における高速な応答と請求されるAPI呼び出しのコスト削減のために、Server ComponentでのHMR（ホットモジュール交換）を跨いで`fetch`応答をキャッシュします。

デフォルトでは、[HMRキャッシュ](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache)は、デフォルトの`auto no cache`および`cache: 'no-store'`オプションを含むすべてのfetchリクエストに適用されます。これは、キャッシュされていないリクエストがHMRリフレッシュの間に新しいデータを表示しないことを意味します。ただし、キャッシュはナビゲーションやフルページのリロード時にクリアされます。

詳細は[`serverComponentsHmrCache`](/docs/app/api-reference/config/next-config-js/serverComponentsHmrCache)ドキュメントを参照してください。
