## 14 から 15 へのアップグレード {#upgrading-from-14-to-15}

Next.js バージョン 15 に更新するには、`upgrade` codemod を使用できます：

```bash title="Terminal"
npx @next/codemod@canary upgrade latest
```

手動で行う場合は、最新の Next と React の RC をインストールしていることを確認してください。例：

```bash title="Terminal"
npm i next@latest react@rc react-dom@rc eslint-config-next@latest
```

> **Good to know:**
>
> - ピア依存関係の警告が表示された場合は、`react` および `react-dom` を提案されたバージョンに更新するか、警告を無視するために `--force` または `--legacy-peer-deps` フラグを使用する必要があります。Next.js 15 と React 19 が安定すれば、これは必要なくなります。

## React 19 {#react-19}

- `react` と `react-dom` の最小バージョンは19です。
- `useFormState` は `useActionState` に置き換えられました。`useFormState` フックは React 19 でも利用可能ですが、非推奨であり、将来のリリースで削除されます。`useActionState` が推奨され、`pending` ステータスを直接読み取るなどの追加のプロパティが含まれています。[詳しくはこちら](https://react.dev/reference/react/useActionState)。
- `useFormStatus` には `data`、`method`、`action` などの追加キーが含まれるようになりました。React 19 を使用していない場合は、`pending` キーのみが利用可能です。[詳しくはこちら](https://react.dev/reference/react-dom/hooks/useFormStatus)。
- 詳しくは [React 19 アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)を参照してください。

> **Good to know:** TypeScript を使用している場合は、`@types/react` および `@types/react-dom` も最新バージョンにアップグレードしてください。

## 非同期リクエストAPI （重大な変更） {#async-request-apis-breaking-change}

以前は同期的に実行されていた実行時情報に依存する動的 API は、現在 **非同期** になっています：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout) の `params`、[`page.js`](/docs/app/api-reference/file-conventions/page)、[`route.js`](/docs/app/api-reference/file-conventions/route)、[`default.js`](/docs/app/api-reference/file-conventions/default)、[`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image)、[`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons)、および [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons)。
- [`page.js`](/docs/app/api-reference/file-conventions/page) の `searchParams`

移行の負担を軽減するために、プロセスを自動化するための [codemod が利用可能です](/docs/app/building-your-application/upgrading/codemods#150) とAPIは一時的に同期的にアクセスできます。

### `cookies` {#cookies}

#### 推奨される非同期使用法 {#recommended-async-usage}

```tsx
import { cookies } from 'next/headers'

// Before
const cookieStore = cookies()
const token = cookieStore.get('token')

// After
const cookieStore = await cookies()
const token = cookieStore.get('token')
```

## `runtime` 構成 （重大な変更） {#runtime-configuration-breaking-change}

`runtime` [segment config](/docs/app/api-reference/file-conventions/route-segment-config#runtime) は以前は `edge` に加えて `experimental-edge` の値をサポートしていました。 両方の構成は同じものを指していますが、オプションを簡素化するため、`experimental-edge` を使用した場合はエラーが発生します。これを修正するには、`runtime` 設定を `edge` に更新します。 自動でこれを行うための [codemod](/docs/app/building-your-application/upgrading/codemods#app-dir-runtime-config-experimental-edge) が利用可能です。

</AppOnly>

## `fetch` リクエスト {#fetch-requests}

[`fetch` リクエスト](/docs/app/api-reference/functions/fetch) はデフォルトではキャッシュされなくなりました。

特定の `fetch` リクエストをキャッシュに対応させるには、`cache: 'force-cache'` オプションを渡すことができます。

```js title="app/layout.js"
export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされない
  const b = await fetch('https://...', { cache: 'force-cache' }) // キャッシュされる

  // ...
}
```

レイアウトまたはページ内のすべての `fetch` リクエストをキャッシュに対応させるには、`export const fetchCache = 'default-cache'` [segment config opotion](/docs/app/api-reference/file-conventions/route-segment-config) を使用できます。個々の `fetch` リクエストが `cache` オプションを指定している場合は、それが使用されます。

```js title="app/layout.js"
// ルートレイアウトであるため、キャッシュオプションを設定しない
// アプリ内のすべての fetch リクエストはキャッシュされます。
export const fetchCache = 'default-cache'

export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされる
  const b = await fetch('https://...', { cache: 'no-store' }) // キャッシュされない

  // ...
}
```

## ルート ハンドラー {#route-handlers}

[ルート ハンドラー](/docs/app/api-reference/file-conventions/route) の `GET` 関数は、デフォルトではキャッシュされなくなりました。 `GET` メソッドをキャッシュに対応させるには、`export const dynamic = 'force-static'` などの [ルート config オプション](/docs/app/api-reference/file-conventions/route-segment-config) をルート ハンドラー ファイルに使用できます。

```js title="app/api/route.js"
export const dynamic = 'force-static'

export async function GET() {}
```

## クライアントサイドのルーターキャッシュ {#client-side-router-cache}

`<Link>` または `useRouter` を介してページ間を移動する際には、[ページ](/docs/app/api-reference/file-conventions/page) セグメントはもはやクライアントサイドのルーター キャッシュから再利用されません。ただし、ブラウザの後方および前方のナビゲーション中および共有レイアウトでは引き続き再利用されます。

ページセグメントをキャッシュに対応させるには、[`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes) 設定オプションを使用できます：

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
}

module.exports = nextConfig
```

[レイアウト](/docs/app/api-reference/file-conventions/layout) と [ロード状態](/docs/app/api-reference/file-conventions/loading) は依然としてキャッシュされ、移動時に再利用されます。

## `next/font` {#next-font}

`@next/font` パッケージは組み込みの [`next/font`](/docs/app/api-reference/components/font) に置き換えられました。インポート名を安全に自動でリネームするための [codemod が利用可能です](/docs/app/building-your-application/upgrading/codemods#built-in-next-font) 。

```js title="app/layout.js"
// Before
import { Inter } from '@next/font/google'

// After
import { Inter } from 'next/font/google'
```

## bundlePagesRouterDependencies {#bundlepagesrouterdependencies}

`experimental.bundlePagesExternals` が安定しており、`bundlePagesRouterDependencies` に名前が変更されました。

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  // Before
  experimental: {
    bundlePagesExternals: true,
  },

  // After
  bundlePagesRouterDependencies: true,
}

module.exports = nextConfig
```

## serverExternalPackages {#serverexternalpackages}

`experimental.serverComponentsExternalPackages` が安定しており、`serverExternalPackages` に名前が変更されました。

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  // Before
  experimental: {
    serverComponentsExternalPackages: ['package-name'],
  },

  // After
  serverExternalPackages: ['package-name'],

module.exports = nextConfig
```

## Speed Insights {#speed-insights}

Speed Insights の自動インストゥルメンテーションは Next.js 15 で削除されました。

Speed Insights を引き続き使用するには、[Vercel Speed Insights クイックスタート](https://vercel.com/docs/speed-insights/quickstart) ガイドを参照してください。

## `NextRequest` のジオロケーション {#nextrequest-geolocation}

`NextRequest` の `geo` および `ip` プロパティは、これらの値はホスティングプロバイダーによって提供されるため削除されました。 この移行を自動化するための [codemod が利用可能です](/docs/app/building-your-application/upgrading/codemods#150)。

Vercel を使用している場合は、代わりに [`@vercel/functions`](https://vercel.com/docs/functions/vercel-functions-package) の `geolocation` および `ipAddress` 関数を使用できます：

```ts title="middleware.ts"
import { geolocation } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { city } = geolocation(request)

  // ...
}
```

```ts title="middleware.ts"
import { ipAddress } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const ip = ipAddress(request)

  // ...
}
```
