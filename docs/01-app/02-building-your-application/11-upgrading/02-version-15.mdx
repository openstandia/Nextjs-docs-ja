## 14 から 15 にアップグレードする {#upgrading-from-14-to-15}

Next.js バージョン 15 に更新するには、`upgrade` codemodを使用できます：

```bash title="Terminal"
npx @next/codemod@canary upgrade latest
```

手動で行う場合は、次のように、最新のNextとReactのRCをインストールしていることを確認してください：

```bash title="Terminal"
npm i next@latest react@rc react-dom@rc eslint-config-next@latest
```

> **Good to know:**
>
> - ピア依存関係の警告が表示される場合は、`react`や`react-dom`を提案されたバージョンに更新するか、`--force`や`--legacy-peer-deps`フラグを使用して警告を無視してください。Next.js 15とReact 19が安定版になると、これは不要になります。
> - TypeScriptを使用している場合、一時的にReact型を上書きする必要があります。詳細は[React 19 RC アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide#installing)を参照してください。

## React 19 {#react-19}

- `react`と`react-dom`の最小バージョンは19になりました。
- `useFormState`は`useActionState`に置き換えられました。`useFormState`フックはReact 19ではまだ利用可能ですが、非推奨となっており、将来のリリースで削除される予定です。`useActionState`の使用が推奨され、`pending`状態を直接読むなど、追加のプロパティが含まれています。[詳細を学ぶ](https://react.dev/reference/react/useActionState)。
- `useFormStatus`は、`data`、`method`、`action`などの追加キーを含むようになりました。React 19を使用していない場合は、`pending`キーのみが利用可能です。[詳細を学ぶ](https://react.dev/reference/react-dom/hooks/useFormStatus)。
- [React 19 アップグレードガイド](https://react.dev/blog/2024/04/25/react-19-upgrade-guide)に詳細があります。

## 非同期リクエストAPI（破壊的変更） {#async-request-apis-breaking-change}

以前は実行時情報に依存する同期的な動的APIが、現在は**非同期**になっています：

- [`cookies`](/docs/app/api-reference/functions/cookies)
- [`headers`](/docs/app/api-reference/functions/headers)
- [`draftMode`](/docs/app/api-reference/functions/draft-mode)
- [`layout.js`](/docs/app/api-reference/file-conventions/layout), [`page.js`](/docs/app/api-reference/file-conventions/page), [`route.js`](/docs/app/api-reference/file-conventions/route), [`default.js`](/docs/app/api-reference/file-conventions/default), [`opengraph-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image), [`twitter-image`](/docs/app/api-reference/file-conventions/metadata/opengraph-image), [`icon`](/docs/app/api-reference/file-conventions/metadata/app-icons), および [`apple-icon`](/docs/app/api-reference/file-conventions/metadata/app-icons) での `params`
- [`page.js`](/docs/app/api-reference/file-conventions/page) での `searchParams`

移行の負担を軽減するために、自動化する[コードモッド](/docs/app/building-your-application/upgrading/codemods#150)が利用可能で、APIは一時的に同期的にアクセスすることができます。

### `cookies` {#cookies}

#### 推奨される非同期の使用法 {#recommended-async-usage}

```tsx
import { cookies } from 'next/headers'

// 以前
const cookieStore = cookies()
const token = cookieStore.get('token')

// その後
const cookieStore = await cookies()
const token = cookieStore.get('token')
```

## `runtime` 構成（破壊的変更） {#runtime-configuration-breaking-change}

`runtime` [segment構成](/docs/app/api-reference/file-conventions/route-segment-config#runtime)は以前、`experimental-edge`という値をサポートしていましたが、`edge`と同じものを意味します。選択肢を簡素化するために、`experimental-edge`の使用はエラーになります。これを修正するには、`runtime` 構成を `edge`に更新してください。[codemod](/docs/app/building-your-application/upgrading/codemods#app-dir-runtime-config-experimental-edge) が利用可能でこれを自動的に行います。

</AppOnly>

## `fetch` リクエスト {#fetch-requests}

[`fetch` リクエスト](/docs/app/api-reference/functions/fetch) はデフォルトでキャッシュされなくなりました。

特定の`fetch`リクエストをキャッシュにオプトインするには、`cache: 'force-cache'`オプションを渡すことができます。

```js title="app/layout.js"
export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされない
  const b = await fetch('https://...', { cache: 'force-cache' }) // キャッシュされる

  // ...
}
```

レイアウトやページ内のすべての`fetch`リクエストをキャッシュにオプトインするには、`export const fetchCache = 'default-cache'` [segment config オプション](/docs/app/api-reference/file-conventions/route-segment-config) を使用できます。個別の`fetch`リクエストが`cache`オプションを指定している場合、それが使用されます。

```js title="app/layout.js"
// これはルートレイアウトであるため、独自のキャッシュオプションを設定しないアプリ内すべてのfetchリクエストがキャッシュされます。
export const fetchCache = 'default-cache'

export default async function RootLayout() {
  const a = await fetch('https://...') // キャッシュされる
  const b = await fetch('https://...', { cache: 'no-store' }) // キャッシュされない

  // ...
}
```

## ルートハンドラ {#route-handlers}

[ルートハンドラ](/docs/app/api-reference/file-conventions/route)での`GET`関数はデフォルトでキャッシュされなくなりました。`GET`メソッドをキャッシュにオプトインするには、`export const dynamic = 'force-static'`のような[ルート構成オプション](/docs/app/api-reference/file-conventions/route-segment-config)を使用できます。

```js title="app/api/route.js"
export const dynamic = 'force-static'

export async function GET() {}
```

## クライアントサイドルーターキャッシュ {#client-side-router-cache}

`<Link>`または`useRouter`を使用してページ間を移動する際、[page](/docs/app/api-reference/file-conventions/page)セグメントはクライアントサイドルーターキャッシュから再利用されなくなりました。ただし、ブラウザの戻る進む操作中や共有されたレイアウトの場合は引き続き再利用されます。

ページセグメントをキャッシュにオプトインするには、[`staleTimes`](/docs/app/api-reference/config/next-config-js/staleTimes)構成オプションを使用できます：

```js title="next.config.js"
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    staleTimes: {
      dynamic: 30,
      static: 180,
    },
  },
}

module.exports = nextConfig
```

[レイアウト](/docs/app/api-reference/file-conventions/layout)と[ロード状態](/docs/app/api-reference/file-conventions/loading)はナビゲーションで引き続きキャッシュおよび再利用されます。

## `next/font` {#next-font}


## Speed Insights {#speed-insights}

Speed Insightsの自動インスツルメンテーションはNext.js 15で削除されました。

Speed Insightsを引き続き使用するには、[Vercel Speed Insights クイックスタート](https://vercel.com/docs/speed-insights/quickstart)ガイドを参照してください。

## `NextRequest` ジオロケーション {#nextrequest-geolocation}

`NextRequest`の`geo`と`ip`プロパティは、ホスティングプロバイダーによって提供されるため削除されました。自動的にこのマイグレーションを行う[コードモッド](/docs/app/building-your-application/upgrading/codemods#150)が利用可能です。

Vercelを使用する場合、代わりに[`@vercel/functions`](https://vercel.com/docs/functions/vercel-functions-package)からの`geolocation`と`ipAddress`関数を使用できます：

```ts title="middleware.ts"
import { geolocation } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const { city } = geolocation(request)

  // ...
}
```

```ts title="middleware.ts"
import { ipAddress } from '@vercel/functions'
import type { NextRequest } from 'next/server'

export function middleware(request: NextRequest) {
  const ip = ipAddress(request)

  // ...
}
```
