---
title: 'Codemods'
description: '新機能がリリースされた際に、Next.jsのコードベースをアップグレードするためにCodemodsを使用する方法。'
---

Codemodsは、コードベースに対してプログラム的に実行される変換です。これにより、大量の変更を手動で各ファイルを確認することなくプログラム的に適用することができます。

Next.jsは、APIの更新や非推奨時にコードベースをアップグレードするのを助けるためにCodemod変換を提供しています。

## 使用法 {#usage}

ターミナルでプロジェクトフォルダに移動して（`cd`）、次のように実行してください：

```bash title="Terminal"
npx @next/codemod <transform> <path>
```

`<transform>` と `<path>` を適切な値に置き換えます。

- `transform` - 変換の名前
- `path` - 変換するファイルまたはディレクトリ
- `--dry` 本番実行せずに結果を確認する（コードは編集されません）
- `--print` 変更された出力を比較のために表示

## Codemods {#codemods}

### 15.0 {#15-0}

#### App Router Route Segment Config の `runtime` 値を `experimental-edge` から `edge` に変換 {#transform-app-router-route-segment-config-runtime-value-from-experimental-edge-to-edge}

##### `app-dir-runtime-config-experimental-edge` {#app-dir-runtime-config-experimental-edge}

> **Note**: このCodemodはApp Router専用です。

```bash title="Terminal"
npx @next/codemod@latest app-dir-runtime-config-experimental-edge .
```

このCodemodは、[Route Segment Config `runtime`](https://nextjs.org/docs/app/api-reference/file-conventions/route-segment-config#runtime) の値 `experimental-edge` を `edge` に変換します。

例：

```ts
export const runtime = 'experimental-edge'
```

次のように変換：

```ts
export const runtime = 'edge'
```

#### 非同期Dynamic APIへの移行 {#migrate-to-async-dynamic-apis}

動的レンダリングにopt-inしていたAPIが、以前は同期アクセスをサポートしていましたが、今や非同期です。この破壊的変更についての詳細は、[アップグレードガイド](/docs/app/building-your-application/upgrading/version-15)をお読みください。

##### `next-async-request-api` {#next-async-request-api}

```bash title="Terminal"
npx @next/codemod@latest next-async-request-api .
```

このCodemodは、非同期として正しく待機されるか、適用可能な場合は`React.use()`でラップされるDynamic API（`cookies()`、`headers()`、および`draftMode()` from `next/headers`）を変換します。
自動変換が不可能な場合、このCodemodは、（もしTypeScriptファイルの場合）型キャストを追加するか、手動で確認＆更新が必要であることを知らせるコメントを追加します。

例：

```tsx
import { cookies, headers } from 'next/headers'
const token = cookies().get('token')

function useToken() {
  const token = cookies().get('token')
  return token
}

export default function Page() {
  const name = cookies().get('name')
}

function getHeader() {
  return headers().get('x-foo')
}
```

次のように変換：

```tsx
import { use } from 'react'
import {
  cookies,
  headers,
  type UnsafeUnwrappedCookies,
  type UnsafeUnwrappedHeaders,
} from 'next/headers'
const token = (cookies() as unknown as UnsafeUnwrappedCookies).get('token')

function useToken() {
  const token = use(cookies()).get('token')
  return token
}

export default async function Page() {
  const name = (await cookies()).get('name')
}

function getHeader() {
  return (headers() as unknown as UnsafeUnwrappedHeaders).get('x-foo')
}
```

このCodemodは、`params`または`searchParams`のpropsがページ/ルートエントリ（`page.js`、`layout.js`、`route.js`、または`default.js`）または`generateMetadata` / `generateViewport`APIでプロパティアクセスが行われていることを検出すると、同期から非同期関数へと呼び出し元を変換し、プロパティアクセスを待機するようにします。クライアントコンポーネントのように、非同期にできない場合は、`React.use`を使用してpromiseをアンラップします。

例：

```tsx
// page.tsx
export default function Page({
  params,
  searchParams,
}: {
  params: { slug: string }
  searchParams: { [key: string]: string | string[] | undefined }
}) {
  const { value } = searchParams
  if (value === 'foo') {
    // ...
  }
}

export function generateMetadata({ params }: { params: { slug: string } }) {
  const { slug } = params
  return {
    title: `My Page - ${slug}`,
  }
}
```

次のように変換：

```tsx
// page.tsx
export default async function Page(props: {
  params: Promise<{ slug: string }>
  searchParams: Promise<{ [key: string]: string | string[] | undefined }>
}) {
  const searchParams = await props.searchParams
  const { value } = searchParams
  if (value === 'foo') {
    // ...
  }
}

export async function generateMetadata(props: {
  params: Promise<{ slug: string }>
}) {
  const params = await props.params
  const { slug } = params
  return {
    title: `My Page - ${slug}`,
  }
}
```

> **Good to know:** このCodemodは手動による介入が必要な箇所を特定した際に、具体的な修正法を決定できない場合、コードにコメントや型キャストを追加して、手動での更新が必要であることをユーザーに伝えます。これらのコメントは**@next/codemod**で始まり、型キャストは`UnsafeUnwrapped`で始まります。
> このコメントが明示的に削除されるまで、ビルドはエラーになります。[詳細はこちら](https://nextjs.org/docs/messages/sync-dynamic-apis).

#### `NextRequest` の `geo` および `ip` プロパティを `@vercel/functions` で置き換え {#replace-geo-and-ip-properties-of-nextrequest-with-vercel-functions}

##### `next-request-geo-ip` {#next-request-geo-ip}

```bash title="Terminal"
npx @next/codemod@latest next-request-geo-ip .
```

このCodemodは`@vercel/functions`をインストールし、`NextRequest`の`geo`と`ip`プロパティを対応する`@vercel/functions`の機能で変換します。

例：

```ts
import type { NextRequest } from 'next/server'

export function GET(req: NextRequest) {
  const { geo, ip } = req
}
```

次のように変換：

```ts
import type { NextRequest } from 'next/server'
import { geolocation, ipAddress } from '@vercel/functions'

export function GET(req: NextRequest) {
  const geo = geolocation(req)
  const ip = ipAddress(req)
}
```

#### `viewport` エクスポートを使用 {#use-viewport-export}

##### `metadata-to-viewport-export` {#metadata-to-viewport-export}

```bash title="Terminal"
npx @next/codemod@latest metadata-to-viewport-export .
```

このCodemodは特定のビューポートメタデータを`viewport`エクスポートに移行します。

例：

```js
export const metadata = {
  title: 'My App',
  themeColor: 'dark',
  viewport: {
    width: 1,
  },
}
```

次のように変換：

```js
export const metadata = {
  title: 'My App',

export const viewport = {
  width: 1,

}
```

### 13.0 {#13-0}

#### Next Image インポートの名前変更 {#rename-next-image-imports}

##### `next-image-to-legacy-image` {#next-image-to-legacy-image}

```bash title="Terminal"
npx @next/codemod@latest next-image-to-legacy-image .
```

Next.js 10, 11, または 12で存在する`next/image`インポートをNext.js 13で`next/legacy/image`に安全にリネームします。また、`next/future/image`を`next/image`にリネームします。

例：

```jsx title="pages/index.js"
import Image1 from 'next/image'
import Image2 from 'next/future/image'

export default function Home() {
  return (
    <div>
      <Image1 src="/test.jpg" width="200" height="300" />
      <Image2 src="/test.png" width="500" height="400" />
    </div>
  )
}
```

次のように変換：

```jsx title="pages/index.js"
// 'next/image' becomes 'next/legacy/image'
import Image1 from 'next/legacy/image'
// 'next/future/image' becomes 'next/image'
import Image2 from 'next/image'

export default function Home() {
  return (
    <div>
      <Image1 src="/test.jpg" width="200" height="300" />
      <Image2 src="/test.png" width="500" height="400" />
    </div>
  )
}
```

#### 新しいImageコンポーネントへの移行 {#migrate-to-the-new-image-component}

##### `next-image-experimental` {#next-image-experimental}

```bash title="Terminal"
npx @next/codemod@latest next-image-experimental .
```

インラインスタイルを追加し、未使用のpropsを削除することで、`next/legacy/image`から新しい`next/image`への危険な移行を行います。

- `layout` propsを削除して`style`を追加します
- `objectFit` propsを削除して`style`を追加します
- `objectPosition` propsを削除して`style`を追加します
- `lazyBoundary` propsを削除します
- `lazyRoot` propsを削除します

#### Linkコンポーネントから`<a>`タグを削除 {#remove-a-tags-from-link-components}

##### `new-link` {#new-link}

```bash title="Terminal"
npx @next/codemod@latest new-link .
```

<AppOnly>

[Link Components](/docs/app/api-reference/components/link) の内部の`<a>`タグを削除するか、自動修正できないリンクには`legacyBehavior` propsを追加します。

</AppOnly>

<PagesOnly>

[Link Components](https://nextjs.org/docs/canary/pages/api-reference/components/link) の内部の`<a>`タグを削除するか、自動修正できないリンクには`legacyBehavior` propsを追加します。

</PagesOnly>

例：

```jsx
<Link href="/about">
  <a>About</a>
</Link>
// transforms into
<Link href="/about">
  About
</Link>

<Link href="/about">
  <a onClick={() => console.log('clicked')}>About</a>
</Link>
// transforms into
<Link href="/about" onClick={() => console.log('clicked')}>
  About
</Link>
```

自動修正が適用できない場合、`legacyBehavior` propsが追加されます。これにより、その特定のリンクで旧動作を使用してアプリが機能し続けます。

```jsx
const Component = () => <a>About</a>

<Link href="/about">
  <Component />
</Link>
// becomes
<Link href="/about" legacyBehavior>
  <Component />
</Link>
```

### 11 {#11}

#### CRAからの移行 {#migrate-from-cra}

##### `cra-to-next` {#cra-to-next}

```bash title="Terminal"
npx @next/codemod cra-to-next
```

Create React AppプロジェクトをNext.jsに移行し、Pages Routerと必要な設定を作成して挙動を一致させます。SSR中の`window`使用による互換性の損失を防ぐために、初めはクライアントサイドのみのレンダリングを利用し、Next.js特有の機能の段階的な採用を容易にします。

この変換に関連するフィードバックは[こちらのディスカッション](https://github.com/vercel/next.js/discussions/25858)で共有してください。

### 10 {#10}

#### Reactインポートを追加 {#add-react-imports}

##### `add-missing-react-import` {#add-missing-react-import}

```bash title="Terminal"
npx @next/codemod add-missing-react-import
```

`React`をインポートしていないファイルを変換し、新しい[React JSX変換](https://reactjs.org/blog/2020/09/22/introducing-the-new-jsx-transform.html)が動作するようにインポートを追加します。

例：

```jsx title="my-component.js"
export default class Home extends React.Component {
  render() {
    return <div>Hello World</div>
  }
}
```

次のように変換：

```jsx title="my-component.js"
import React from 'react'
export default class Home extends React.Component {
  render() {
    return <div>Hello World</div>
  }
}
```

### 9 {#9}

#### 匿名コンポーネントを名前付きコンポーネントに変換 {#transform-anonymous-components-into-named-components}

##### `name-default-component` {#name-default-component}

```bash title="Terminal"
npx @next/codemod name-default-component
```

**バージョン9以上。**

匿名コンポーネントを名前付きコンポーネントに変換し、[Fast Refresh](https://nextjs.org/blog/next-9-4#fast-refresh)と互換性を持たせます。

例：

```jsx title="my-component.js"
export default function () {
  return <div>Hello World</div>
}
```

次のように変換：

```jsx title="my-component.js"
export default function MyComponent() {
  return <div>Hello World</div>
}
```

コンポーネントはファイル名に基づいてキャメルケースの名前を持ち、箭頭関数にも対応しています。

### 8 {#8}

#### AMP HOC をページ設定に変換 {#transform-amp-hoc-into-page-config}

##### `withamp-to-config` {#withamp-to-config}

```bash title="Terminal"
npx @next/codemod withamp-to-config
```

`withAmp` HOCをNext.js 9のページ設定に変換します。

例：

```js
// Before
import { withAmp } from 'next/amp'

function Home() {
  return <h1>My AMP Page</h1>
}

export default withAmp(Home)
```

```js
// After
export default function Home() {
  return <h1>My AMP Page</h1>
}

export const config = {
  amp: true,
}
```

### 6 {#6}

#### `withRouter` を使用 {#use-withrouter}

##### `url-to-withrouter` {#url-to-withrouter}

```bash title="Terminal"
npx @next/codemod url-to-withrouter
```

ページレベルで自動的にインジェクトされた`url`プロパティの使用を非推奨とし、`withRouter`およびそれがインジェクトする`router`プロパティを使用するように変換します。詳しくはこちらをお読みください：[https://nextjs.org/docs/messages/url-deprecated](https://nextjs.org/docs/messages/url-deprecated)

例：

```js title="From"
import React from 'react'
export default class extends React.Component {
  render() {
    const { pathname } = this.props.url
    return <div>Current pathname: {pathname}</div>
  }
}
```

```js title="To"
import React from 'react'
import { withRouter } from 'next/router'
export default withRouter(
  class extends React.Component {
    render() {
      const { pathname } = this.props.router
      return <div>Current pathname: {pathname}</div>
    }
  }
)
```

これは一例です。変換される（そしてテストされた）すべてのケースは、[`__testfixtures__`ディレクトリ](https://github.com/vercel/next.js/tree/canary/packages/next-codemod/transforms/__testfixtures__/url-to-withrouter)にあります。
