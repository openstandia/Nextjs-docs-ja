Incremental Static Regeneration (ISR)は次のことを可能にします：

- サイト全体を再構築せずに静的コンテンツを更新する
- 大半のリクエストに対してプリレンダリングされた静的ページを提供しサーバー負荷を軽減する
- 適切な`cache-control`ヘッダーがページに自動的に追加されることを保証する
- `next build`時間が長くなることなく、大量のコンテンツページを処理する

以下は最小限の例です：

```tsx title="app/blog/[id]/page.tsx" switcher
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // ビルド時にこれらのパスのみをプリレンダリングします。
  // { fallback: 'blocking' }は、パスが存在しない場合に、
  // オンデマンドでページをサーバーレンダリングします。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // リクエストが来た時にNext.jsはキャッシュを無効化し、
    // 最大で60秒に1回行います。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

この例の動作の仕組み：

1. `next build`の間に、すべての既知のブログ記事が生成されます（この例では25あります）
2. これらのページに対するすべてのリクエスト（例：`/blog/1`）はキャッシュされ、瞬時に応答します
3. 60秒が経過した後の次のリクエストは、キャッシュされた（陳腐化した）ページを依然として表示します
4. キャッシュが無効化され、新しいバージョンのページがバックグラウンドで生成され始めます
5. 成功裏に生成された場合、Next.jsは更新されたページを表示し、キャッシュします
6. `/blog/26`がリクエストされた場合、Next.jsはオンデマンドでこのページを生成し、キャッシュします

### 時間ベースのリバリデーション {#time-based-revalidation}

これは`/blog`でブログ記事のリストを取得して表示します。1時間後、このページのキャッシュはページが次に訪問されたときに無効化されます。そしてバックグラウンドで、最新のブログ記事を使用して新しいバージョンのページが生成されます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/blog/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

export const revalidate = 3600 // 1時間ごとに無効化する

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

</TabItem>

リバリデーション時間は長く設定することを推奨します。例えば、1秒ではなく1時間にするなどです。もしより精密に制御したい場合は、オンデマンドリバリデーションを検討してください。リアルタイムデータが必要な場合は、[dynamic rendering](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)への切り替えを検討してください。

### `revalidatePath`を使用したオンデマンドリバリデーション {#on-demand-revalidation-with-revalidatepath}

リバリデーションをより精密に制御する方法として、`revalidatePath`関数を使用してページをオンデマンドで無効化します。

例えば、新しい投稿を追加した後にこのServer Actionが呼び出されるとします。`fetch`を使用するかデータベースに接続してデータを取得するかに関係なく、この操作はルート全体のキャッシュをクリアし、Server Componentに新しいデータを取得させることが可能です。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/actions.ts" switcher
export async function createPost() {
  // キャッシュ内の /postsルートを無効化します
  revalidatePath('/posts')
}
```

### `res.revalidate()`を使用したオンデマンドバリデーション {#on-demand-validation-with-res-revalidate}

リバリデーションをより精密に制御する方法として、APIルータからページをオンデマンドで生成するために`res.revalidate`を使用します。

例えば、このAPIルートは`/api/revalidate?secret=<token>`で呼び出され、指定されたブログ記事をリバリデーションすることができます。Next.jsアプリのみに知られるシークレットトークンを作成してください。このシークレットは、リバリデーションAPIルートへの不正アクセスを防ぐために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // シークレットを確認して、このリクエストが有効であることを確認します
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // これが実際のパスであり、書き換えられたパスではないことを確認してください
    // "/posts/[id]"の場合、"/posts/1"になるべきです
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーが発生した場合、Next.jsは最後に成功したページを表示し続けます
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### キャッチされていない例外の処理 {#handling-uncaught-exceptions}

<AppOnly>

データのリバリデーションを試みてエラーがスローされた場合でも、最後に成功したデータが引き続きキャッシュから提供されます。次のリクエスト時に、Next.jsはデータの再リバリデーションを試みます。[エラー処理についてもっと学ぶ](/docs/app/building-your-application/routing/error-handling)

</AppOnly>

<PagesOnly>

バックグラウンドリジェネレーション時に`getStaticProps`内でエラーが発生した場合、または手動でエラーをスローした場合、最後に成功したページが引き続き表示されます。次のリクエスト時に、Next.jsは再度`getStaticProps`を呼び出そうとします。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストがキャッチされていないエラーを投げた場合、Next.jsは
  // 現在表示されているページを無効化せず、
  // 次のリクエストの際にgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーが発生した場合、エラーをスローした方が良いでしょう
    // 成功したリクエストが来るまで、キャッシュは更新されません。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // リクエストが来たときにNext.jsはキャッシュを無効化し、
    // 最大でも60秒ごとに1回行います。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュの位置をカスタマイズする {#customizing-the-cache-location}

ページのキャッシュおよびリバリデーション（Incremental Static Regenerationを使用）は、同じ共有キャッシュを使用します。[Vercelにデプロイする](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)場合、ISRキャッシュは自動的に耐久性のあるストレージに永続化されます。

セルフホスティングでは、ISRキャッシュはNext.jsサーバー上のファイルシステム（ディスク）に保存されます。この作業は、PagesおよびApp Routerの両方を使用したセルフホスティングでも自動的に行われます。

もしキャッシュされたページやデータを耐久性のあるストレージに永続化したり、Next.jsアプリケーションの複数のコンテナやインスタンス間でキャッシュを共有したい場合、Next.jsのキャッシュ位置を設定できます。[もっと学ぶ](/docs/app/building-your-application/deploying#caching-and-isr)

## トラブルシューティング {#troubleshooting}

### ローカル開発でキャッシュされたデータをデバッグする {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、リクエストがキャッシュされているのかキャッシュされていないのかを理解するために追加のログを追加することができます。[`logging`オプションについてもっと学ぶ](/docs/app/api-reference/config/next-config-js/logging)

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しい本番設定を検証する {#verifying-correct-production-behavior}

本番環境でページが正しくキャッシュされリバリデーションされることを確認するには、`next build`を実行した後に`next start`を実行して、プロダクションNext.jsサーバーをローカルでテストすることができます。

これにより、プロダクション環境でのISRの動作をテストすることができます。さらにデバッグするために、次の環境変数を`.env`ファイルに追加します：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーはISRキャッシュのヒットとミスをコンソールログに記録します。`next build`中にどのページが生成されるか、オンデマンドでアクセスされる際にページがどのように更新されるかを出力を検査することができます。

## 注意事項 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用している場合にのみサポートされています。
- ISRは[静的エクスポート](/docs/app/building-your-application/deploying/static-exports)を作成する際にはサポートされていません。
- 静的にレンダリングされたルートに複数の`fetch`リクエストがあり、それぞれ異なる`revalidate`頻度を持つ場合、ISRでは最も低い時間が使用されます。ただし、これらのリバリデーション頻度は[データキャッシュ](/docs/app/building-your-application/caching#data-cache)によって依然として尊重されます。
- ルートで使用されている`fetch`リクエストのいずれかが`revalidate`時間を`0`または明示的に`no-store`に指定されている場合、ルートは[動的にレンダリングされます](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)。
- オンデマンドISRリクエストではミドルウェアが実行されないため、ミドルウェアでのパスの書き換えやロジックが適用されることはありません。正確なパスをリバリデーションしていることを確認してください。たとえば、書き換えられた`/post-1`ではなく`/post/1`を使用してください。

</AppOnly>
