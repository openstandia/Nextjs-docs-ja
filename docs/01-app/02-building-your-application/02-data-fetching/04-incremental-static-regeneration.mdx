Incremental Static Regeneration（ISR）は以下を可能にします：

- サイト全体を再構築することなく静的コンテンツを更新
- 事前レンダリングされた静的ページをほとんどのリクエストに対して提供することでサーバー負荷を軽減
- 適切な`cache-control`ヘッダーがページに自動的に追加されることを保証
- 大量のコンテンツページを長い`next build`時間なしで処理

ここに最小限の例があります：

  title: string
  content: string

// リクエストが入ると、60秒ごとに最大で1回。

// ビルド時に`generateStaticParams`からのみパラメータを事前レンダリングします。
// 生成されていないパスにリクエストが入ると、
// Next.jsはオンデマンドでページをサーバーレンダリングします。

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}

import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // ビルド時にはこれらのパスのみを事前レンダリングします。
  // { fallback: false } 時は、存在しないパスのページを
  // オンデマンドでサーバーレンダリングします。
  return { paths, fallback: false }
}

  params,

}) => {

    props: { post },
    // Next.jsはキャッシュを無効にします。

export default function Page({ post }: Props) {

この例はどのように機能するか：

1. `next build`中に、すべての既知のブログ投稿が生成されます（この例では25件あります）
2. これらのページへのすべてのリクエスト（例：`/blog/1`）はキャッシュされ、即時に応答されます
3. 60秒が経過した後、次のリクエストでもキャッシュされた（古い）ページが表示されます
4. キャッシュが無効になり、ページの新しいバージョンがバックグラウンドで生成され始めます
5. 正常に生成されると、Next.jsは更新されたページを表示し、キャッシュします
6. `/blog/26`がリクエストされた場合、Next.jsはオンデマンドでこのページを生成し、キャッシュします

高い再検証時間を設定することをお勧めします。例えば、1秒ではなく1時間。より精密な検証が必要な場合は、オンデマンド再検証をご検討ください。リアルタイムデータが必要な場合は、[dynamic rendering](/docs/app/building-your-application/rendering/server-components#dynamic-rendering) に切り替えることを検討してください。

### `revalidatePath`を使用したオンデマンド再検証 {#on-demand-revalidation-with-revalidatepath}

より正確な再検証方法には、`revalidatePath`関数を使用してオンデマンドでページを無効化します。

たとえば、このServer Actionは、新しい投稿を追加した後に呼び出されます。`fetch`を使用してデータを取得するか、データベースに接続するかに関係なく、この操作によりルート全体のキャッシュがクリアされ、Server Componentが新しいデータを取得できます。

### `res.revalidate()`を使用したオンデマンド検証 {#on-demand-validation-with-res-revalidate}

より正確な再検証方法のために、`res.revalidate`を使用してAPI Routerからオンデマンドで新しいページを生成します。

たとえば、このAPI Routeは`/api/revalidate?secret=<token>`で呼び出され、指定されたブログ投稿を再検証します。Next.jsアプリのみが知っている秘密のトークンを作成してください。この秘密は再検証API Routeへの不正なアクセスを防止します。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // このリクエストが有効であることを確認するための秘密を確認
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // これは書き換えられたパスではなく実際のパスでなければなりません
    // たとえば、「/posts/[id]」の場合は「/posts/1」
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーがあった場合、Next.jsは引き続き
    // 最後に正常に生成されたページを表示し続けます
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### 未捕捉例外の取り扱い {#handling-uncaught-exceptions}

<AppOnly>

データを再検証しようとしてエラーが発生した場合、最後に正常に生成されたデータがキャッシュから提供され続けます。次のリクエスト時に、Next.jsはデータの再検証を試みます。 [エラー処理について学ぶ](/docs/app/building-your-application/routing/error-handling)。

</AppOnly>

<PagesOnly>

バックグラウンド再生成を処理するときに`getStaticProps`内でエラーが発生した場合、または手動でエラーをスローした場合、最後に正常に生成されたページが引き続き表示されます。次のリクエスト時に、Next.jsは`getStaticProps`を再試行します。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストがキャッチされていないエラーをスローした場合、Next.jsは
  // 現在表示されているページを無効にせず、
  // 次のリクエストでgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーがある場合、次の成功したリクエストまで
    // キャッシュが更新されないようにエラーをスローすることを
    // 考慮することもできます。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.jsはキャッシュを無効にします。
    // リクエストが入ると、60秒ごとに最大で1回。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュの場所をカスタマイズする {#customizing-the-cache-location}

インクリメンタルスタティックリジェネレーションを使用したページのキャッシュと再検証には、同じ共有キャッシュが使用されます。 [Vercelにデプロイする](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)と、ISRキャッシュは自動的に耐久ストレージに永続化されます。

セルフホスティングの場合、ISRキャッシュはNext.jsサーバーでファイルシステム（ディスク上）に保存されます。これはPages RouterとApp Routerの両方を使用してセルフホスティングする場合に自動的に機能します。

キャッシュされたページとデータを耐久ストレージに永続化したり、Next.jsアプリケーションの複数のコンテナやインスタンス間でキャッシュを共有したい場合は、Next.jsのキャッシュ場所を設定できます。[詳細を学ぶ](/docs/app/building-your-application/deploying#caching-and-isr)。

## トラブルシューティング {#troubleshooting}

### ローカル開発でのキャッシュされたデータのデバッグ {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、キャッシュされたリクエストと未キャッシュのリクエストを理解するために追加のログを追加できます。[`logging`オプションの詳細を学ぶ](/docs/app/api-reference/config/next-config-js/logging)。

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しい本番環境の動作を確認する {#verifying-correct-production-behavior}

本番環境でページが正しくキャッシュされ再検証されているかを確認するために、`next build`を実行し、その後`next start`を実行して本番Next.jsサーバーを実行してローカルでテストできます。

これにより、本番環境でのISRの動作をテストできます。さらにデバッグするためには、次の環境変数を`.env`ファイルに追加します：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーのコンソールでISRキャッシュのヒットとミスがログ出力されます。どのページが`next build`中に生成され、どのようにオンデマンドでアクセスされたパスが更新されるかを出力から確認できます。

## 注意事項 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用する場合にのみサポートされています。
- ISRは[静的エクスポート](/docs/app/building-your-application/deploying/static-exports)を作成するときはサポートされていません。
- 静的にレンダリングされたルートに複数の`fetch`リクエストがあり、それぞれに異なる`revalidate`頻度がある場合、最も短い時間がISRに使用されます。ただし、それらの再検証頻度は[データキャッシュ](/docs/app/building-your-application/caching#data-cache)によって引き続き尊重されます。
- ルートで使用されているどの`fetch`リクエストにも`revalidate`時間が`0`または明示的な`no-store`がある場合、そのルートは[動的にレンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)されます。
- オンデマンドISRリクエストにはMiddlewareは実行されません。つまり、Middlewareの中のすべてのパスのリライトやロジックは適用されません。正確なパスを無効化していることを確認してください。例：`/post/1`の代わりにリライトされた`/post-1`。

</AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用する場合にのみサポートされています。
- ISRは[静的エクスポート](/docs/app/building-your-application/deploying/static-exports)を作成するときはサポートされていません。
- オンデマンドISRリクエストにはMiddlewareは実行されません。つまり、Middlewareの中のすべてのパスのリライトやロジックは適用されません。正確なパスを無効化していることを確認してください。例：`/post/1`の代わりにリライトされた`/post-1`。

</PagesOnly>
