Incremental Static Regeneration (ISR) により以下が可能になります：

- サイト全体を再構築することなく静的コンテンツを更新します
- ほとんどのリクエストに対して事前レンダリングされた静的ページを提供することでサーバー負荷を軽減します
- 正しい `cache-control` ヘッダがページに自動的に追加されるようにします
- 大量のコンテンツページを扱う際の `next build` 時間を短縮します

次に、最小限の例を示します：

import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // これらのパスのみをビルド時に事前レンダリングします。
  // { fallback: 'blocking' } を使うことで、パスが存在しない場合
  // サーバーレンダリングでページを生成します。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // Next.js は、リクエストが届くとキャッシュを無効化し、
    // 最大で60秒ごとに無効化します。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}

この例の動作は次のとおりです：

1. `next build` の間に、すべての既知のブログ記事が生成されます（この例では25個の記事があります）
2. これらのページ（例：`/blog/1`）へのすべてのリクエストはキャッシュされており、瞬時に応答します
3. 60秒が経過した後でも、次のリクエストは依然としてキャッシュされている（陳腐化した）ページを表示します
4. キャッシュが無効化され、バックグラウンドで新しいバージョンのページが生成を開始します
5. 成功裏に生成されたら、Next.js は更新されたページを表示しキャッシュします
6. `/blog/26` がリクエストされた場合、Next.js はこのページをオンデマンドで生成しキャッシュします

### Time-based revalidation {#time-based-revalidation}

これは `/blog` でブログ記事のリストを取得して表示します。1時間後、このページのキャッシュが次回の訪問時に無効化されます。その後、バックグラウンドで最新のブログ記事でページの新しいバージョンが生成されます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/blog/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

export const revalidate = 3600 // 毎時無効化します

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

</TabItem>
<TabItem value="jsx" label="JavaScript">

```jsx title="app/blog/page.js" switcher

再検証の時間は高めに設定することをお勧めします。例えば、1秒ではなく1時間です。精度が必要な場合はオンデマンドでの再検証を考慮し、リアルタイムデータが必要な場合は [動的レンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering) に切り替えてください。

### On-demand revalidation with `revalidatePath` {#on-demand-revalidation-with-revalidatepath}

より正確な再検証方法として、`revalidatePath` 関数を使用してページをオンデマンドで無効化します。

たとえば、この Server Action は新しい投稿を追加した後に呼び出されます。Server Component でのデータ取得方法にかかわらず、`fetch` を使用しているか、データベースに接続しているかにかかわらず、これによりルート全体のキャッシュがクリアされ、Server Component が新しいデータを取得できるようになります。

<Tabs>
<TabItem value="ts" label="TypeScript">

  // /posts ルートをキャッシュで無効化します
  revalidatePath('/posts')
}
```

</TabItem>

### On-demand validation with `res.revalidate()` {#on-demand-validation-with-res-revalidate}

より正確な再検証方法として、`res.revalidate` を使用して API Router からオンデマンドで新しいページを生成することができます。

たとえば、この API Route は `/api/revalidate?secret=<token>` で呼び出され、指定されたブログ記事を再検証します。 Next.js アプリによってのみ知られるシークレットトークンを作成してください。このシークレットは、再検証 API Route への不正アクセスを防ぐために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // シークレットを確認して、これが有効なリクエストであることを確認します
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // ここは書き換えられたパスではなく実際のパスである必要があります
    // 例えば、"/posts/[id]" の場合は "/posts/1" です
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーが発生した場合、Next.js は継続して
    // 最後に正常に生成されたページを表示し続けます
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### 未処理の例外を処理する {#handling-uncaught-exceptions}

<AppOnly>

データの再検証を試みた際にエラーが発生した場合、最後に正常に生成されたデータがキャッシュから提供され続けます。次回のリクエスト時に、Next.js は再度そのデータの再検証を試みます。 [エラーハンドリングについて詳しく学びます](/docs/app/building-your-application/routing/error-handling)。

</AppOnly>

<PagesOnly>

バックグラウンド再生成を処理する際、`getStaticProps` 内でエラーが発生した場合、または手動でエラーをスローした場合、最後に正常に生成されたページが表示され続けます。次回のリクエスト時に、Next.js は再び `getStaticProps` を呼び出すことを試みます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストが未処理のエラーを発生させた場合、Next.js は
  // 現在表示しているページを無効化せず、次回のリクエスト時に
  // getStaticProps を再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーがある場合、次の成功するリクエストまで
    // キャッシュが更新されないようにエラーをスローすることを検討
    // する必要があります。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.js は、リクエストが届くとキャッシュを無効化し、
    // 最大で60秒ごとに無効化します。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュ場所のカスタマイズ {#customizing-the-cache-location}

ページ（Incremental Static Regeneration で）のキャッシュと再検証は、同じ共有のキャッシュを使用します。 [Vercel へのデプロイ時](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)に、ISR キャッシュは自動的に永続ストレージに保存されます。

自己ホストの場合、ISR キャッシュは Next.js サーバーのファイルシステム（ディスク上）に保存されます。これは、Pages Router と App Router の両方を使用する自己ホストの際に自動的に機能します。

キャッシュされたページとデータを永続ストレージに保存したり、Next.js アプリケーションの複数のコンテナやインスタンス間でキャッシュを共有したりする場合は、Next.js キャッシュの場所を構成できます。 [詳細はこちら](/docs/app/building-your-application/deploying#caching-and-isr)。

## トラブルシューティング {#troubleshooting}

### ローカル開発でキャッシュデータをデバッグする {#debugging-cached-data-in-local-development}

「fetch」API を使用している場合、キャッシュされているリクエストとキャッシュされていないリクエストを理解するために追加のログを追加することができます。 [`logging` オプションについてさらに学びます](/docs/app/api-reference/config/next-config-js/logging)。

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しい本番環境での動作確認 {#verifying-correct-production-behavior}

あなたのページが本番環境で正しくキャッシュされ再検証されるかを確認するために、`next build` を実行し、その後 `next start` を実行して本番 Next.js サーバーを実行することで、ローカルでテストできます。

これにより、ISR の動作が本番環境でどのように機能するかをテストできます。さらなるデバッグを行うには、次の環境変数を `.env` ファイルに追加してください：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.js サーバーは ISR キャッシュのヒットとミスをコンソールログに記録します。これにより、`next build` の間に生成されたページ、およびオンデマンドでパスにアクセスするとページがどのように更新されるかを確認できます。

## 注意点 {#caveats}

<AppOnly>

- ISR は Node.js ランタイム（デフォルト）を使用する場合にのみサポートされています。
- [Static Export](/docs/app/building-your-application/deploying/static-exports) を作成する際には、ISR はサポートされていません。
- 静的にレンダリングされたルートに複数の `fetch` リクエストが存在し、それぞれが異なる `revalidate` 周期を持っている場合、ISR のために最も低い時間が使用されます。ただし、これらの再検証頻度は[データキャッシュ](/docs/app/building-your-application/caching#data-cache)によっても保持されます。
- ルートで使用されている `fetch` リクエストのいずれかが `revalidate` 時間 `0`、または明示的な `no-store` を持つ場合、そのルートは[動的にレンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)されます。
- オンデマンド ISR リクエストに対してミドルウェアは実行されません。つまり、ミドルウェアの中でのパス書き換えやロジックは適用されません。必ず、正確なパスを再検証してください。たとえば、書き換えられた `/post-1` ではなく、`/post/1` を再検証してください。

</AppOnly>
