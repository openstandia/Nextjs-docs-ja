Incremental Static Regeneration (ISR)を使用すると、以下のことが可能になります：

- サイト全体を再構築することなく静的コンテンツを更新
- ほとんどのリクエストに対してプリレンダリングされた静的ページを提供することでサーバー負荷を削減
- 適切な`cache-control`ヘッダーが自動的にページに追加されることを保証
- 長時間の`next build`なしで多くのコンテンツページを処理

以下は最小限の例です：

```tsx title="app/blog/[id]/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

// Next.jsは、リクエストが到着するとキャッシュを最長で60秒ごとに無効にします。
export const revalidate = 60

// ビルド時間に`generateStaticParams`からのパラメータのみをプリレンダリングします。
// 生成されていないパスでリクエストが来た場合、Next.jsはページをオンデマンドでサーバーレンダリングします。
export const dynamicParams = true // もしくはfalseは、未知のパスで404を返します。

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // ビルド時間にこれらのパスのみをプリレンダリングします。
  // { fallback: 'blocking' }は、パスが存在しない場合にオンデマンドでページをサーバーレンダリングします。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // Next.jsは、リクエストが到着するとキャッシュを最長で60秒ごとに無効にします。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

この例がどのように動作するかを説明します：

1. `next build`時に、すべての既知のブログ投稿が生成されます（この例では25件）
2. これらのページに対するすべてのリクエスト（例：`/blog/1`）はキャッシュされ、即座に応答されます
3. 60秒経過後でも、次のリクエストはキャッシュされた（古い）ページを表示します
4. キャッシュが無効化され、新しいバージョンのページがバックグラウンドで生成され始めます
5. 成功裏に生成されると、Next.jsはページを更新してキャッシュします
6. `/blog/26`がリクエストされた場合、Next.jsはこのページをオンデマンドで生成してキャッシュします

### 時間に基づくリバリデーション {#time-based-revalidation}

これは`/blog`でブログ投稿のリストを取得して表示します。1時間後に、このページのキャッシュは次の訪問で無効化されます。続いて、バックグラウンドで最新のブログ投稿を使用して新しいバージョンのページが生成されます。

```tsx title="app/blog/page.tsx" switcher
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

リバリデーションの時間を長く設定することを推奨します。例えば、1秒ではなく1時間です。より精緻な精度が必要な場合は、オンデマンドリバリデーションを使用することを検討してください。リアルタイムデータが必要な場合は、[動的レンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)に切り替えることを検討してください。

### `revalidatePath`を使用したオンデマンドリバリデーション {#on-demand-revalidation-with-revalidatepath}

より精緻なリバリデーション手法として、`revalidatePath`関数を使用してオンデマンドでページを無効化します。

例えば、新しい投稿を追加した後に呼び出されるServer Actionです。Server Componentでデータを取得する方法に関わらず、`fetch`を使用するかデータベースに接続するかに関わらず、これによりルート全体のキャッシュがクリアされ、Server Componentが新しいデータを取得できるようになります。

```ts title="app/actions.ts" switcher
export async function createPost() {
  // キャッシュ内の/postsルートを無効化
  revalidatePath('/posts')
}
```

</TabItem>

### `res.revalidate()`を使用したオンデマンド検証 {#on-demand-validation-with-res-revalidate}

より精緻なリバリデーション手法として、APIルーターからオンデマンドで新しいページを生成するために`res.revalidate`を使用します。

例えば、特定のブログ投稿を再検証するために`/api/revalidate?secret=<token>`で呼び出せるこのAPIルートを作成します。Next.jsアプリケーションのみに知られる秘密のトークンを作成してください。この秘密は、リバリデーションAPIルートへの不正アクセスを防ぐために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // 秘密を確認して、有効なリクエストであることを確認します
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // これは書き換えられたパスではなく実際のパスであるべきです
    // 例: "/posts/[id]" の場合は "/posts/1"
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーがあった場合、Next.jsは引き続き
    // 最後に成功したページを表示し続けます
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### キャッチされない例外の処理 {#handling-uncaught-exceptions}

<AppOnly>

データを再検証しようとする際にエラーが発生した場合、最後に正常に生成されたデータがキャッシュから引き続き提供されます。次のリクエストでは、Next.jsはデータの再検証を試みます。[エラーハンドリングについて詳しくはこちら](/docs/app/building-your-application/routing/error-handling)。

</AppOnly>

<PagesOnly>

バックグラウンド再生成を処理する際に`getStaticProps`内でエラーが発生した場合、またはエラーを手動でスローした場合、最後に正常に生成されたページが引き続き表示されます。次のリクエストでは、Next.jsは再度`getStaticProps`を呼び出そうとします。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストがキャッチされないエラーをスローすると、Next.jsは
  // 現在表示されているページを無効化せず、
  // 次のリクエスト時にgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーが発生した場合、キャッシュが次の成功するリクエストまで
    // 更新されないようにエラーをスローすることをお勧めします。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.jsは、リクエストが到着するとキャッシュを最長で60秒ごとに無効にします。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュの保存場所のカスタマイズ {#customizing-the-cache-location}

ページのキャッシングおよびリバリデーション（Incremental Static Regenerationを使用）は、同じ共有キャッシュを使用します。[Vercelにデプロイする際](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)、ISRキャッシュは耐久性のあるストレージに自動的に保存されます。

自己ホストの場合、ISRキャッシュはNext.jsサーバー上のファイルシステム（ディスク）に保存されます。これは、PagesルーターとApp Routerの両方を使用して自己ホストする場合に自動的に動作します。

キャッシュされたページとデータを耐久性のあるストレージに保存したり、Next.jsアプリケーションの複数のコンテナやインスタンス間でキャッシュを共有したい場合は、Next.jsキャッシュの保存場所を設定できます。[詳しくはこちら](/docs/app/building-your-application/deploying#caching-and-isr)。

## トラブルシューティング {#troubleshooting}

### ローカル開発におけるキャッシュデータのデバッグ {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、キャッシュ済みまたはキャッシュされていないリクエストを理解するために追加のログを追加できます。 [`logging`オプションについて詳しくはこちら](/docs/app/api-reference/config/next-config-js/logging)。

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しい本番動作の確認 {#verifying-correct-production-behavior}

本番環境でページが正しくキャッシュされてリバリデートされることを確認するには、`next build`を実行し、次に`next start`を実行して本番のNext.jsサーバーをローカルでテストできます。

これにより、本番環境でのISRの動作をテストできます。さらなるデバッグを行うために、次の環境変数を`.env`ファイルに追加します：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーのコンソールにISRキャッシュのヒットとミスがログ出力されます。`next build`中に生成されるページと、オンデマンドでアクセスされるパスに応じてページが更新される様子を確認できます。

## 注意点 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）でのみサポートされています。
- [静的エクスポート](/docs/app/building-your-application/deploying/static-exports)を作成する際にはISRはサポートされていません。
- 静的にレンダリングされたルート内に複数の`fetch`リクエストがあり、それぞれ異なる`revalidate`頻度がある場合、ISRには最も低い時間が使用されます。ただし、これらのリバリデート頻度は[データキャッシュ](/docs/app/building-your-application/caching#data-cache)によって引き続き考慮されます。
- ルートで使用される`fetch`リクエストのいずれかに`revalidate`時間が`0`または`no-store`が明示的に設定されている場合、そのルートは[動的レンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)されます。
- オンデマンドISRリクエストのミドルウェアは実行されないため、ミドルウェア内のパスの書き換えやロジックは適用されません。正確なパスを再検証していることを確認してください。例えば、書き換えられた`/post-1`ではなく`/post/1`です。

</AppOnly>

- ISRはNode.jsランタイム（デフォルト）でのみサポートされています。
- [静的エクスポート](/docs/app/building-your-application/deploying/static-exports)を作成する際にはISRはサポートされていません。
- オンデマンドISRリクエストのミドルウェアは実行されないため、ミドルウェア内のパスの書き換えやロジックは適用されません。正確なパスを再検証していることを確認してください。例えば、書き換えられた`/post-1`ではなく`/post/1`です。

</PagesOnly>
