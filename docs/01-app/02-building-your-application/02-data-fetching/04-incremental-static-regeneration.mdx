Incremental Static Regeneration（ISR）は以下を可能にします：

- サイト全体をビルドし直さずに静的コンテンツを更新する
- ほとんどのリクエストに対して事前レンダリングされた静的ページを配信することでサーバーロードを削減する
- 適切な`cache-control`ヘッダーをページに自動的に追加することを保証する
- 長い`next build`時間なしで膨大な量のコンテンツページを処理する

以下は最小の例です：

interface Post {
  id: string
  title: string
  content: string
}

// Next.jsは、リクエストが来たときにキャッシュを無効にします。
// 最大60秒に1度までです。
export const revalidate = 60

// ビルド時に`generateStaticParams`からのみparamsを事前レンダリングします。
// 生成されていないパスへのリクエストが来た場合、
// Next.jsはオンデマンドでページをサーバーレンダリングします。
export const dynamicParams = true // またはfalseにすると、未知のパスで404になります

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}

import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // ビルド時にのみこれらのパスを事前レンダリングします。
  // { fallback: 'blocking' } は、パスが存在しない場合にオンデマンドでサーバーレンダリングします。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // Next.jsは、リクエストが来たときにキャッシュを無効にします。
    // 最大60秒に1度までです。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}

この例の仕組みは以下のとおりです：

1. `next build`中に、既知のブログ投稿がすべて生成されます（この例では25件あります）
2. これらのページへのすべてのリクエスト（例：`/blog/1`）はキャッシュされ、瞬時に応答されます
3. 60秒が経過した後、次のリクエストは依然としてキャッシュされた（古い）ページを表示します
4. キャッシュが無効になり、ページの新しいバージョンがバックグラウンドで生成され始めます
5. 正常に生成されると、Next.jsは更新されたページを表示し、キャッシュします
6. `/blog/26`が要求されると、Next.jsはオンデマンドでこのページを生成してキャッシュします

### 時間ベースのリバリデーション {#time-based-revalidation}

これは `/blog` にブログ投稿のリストを取得して表示します。1時間後、このページのキャッシュは次回の訪問で無効化されます。次に、バックグラウンドで最新のブログ投稿を使ってページの新しいバージョンが生成されます。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/blog/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

export const revalidate = 3600 // 1時間ごとに無効化されます

export default async function Page() {
  const data = await fetch('https://api.vercel.app/blog')
  const posts: Post[] = await data.json()
  return (
    <main>
      <h1>Blog Posts</h1>
      <ul>
        {posts.map((post) => (
          <li key={post.id}>{post.title}</li>
        ))}
      </ul>
    </main>
  )
}
```

</TabItem>

高いリバリデーション時間を設定することをお勧めします。例えば、1秒ではなく1時間です。より高い精度が必要な場合は、オンデマンドリバリデーションを使用することを検討してください。リアルタイムデータが必要な場合は、[dynamic rendering](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)に切り替えることを検討してください。

### `revalidatePath`を使用したオンデマンドリバリデーション {#on-demand-revalidation-with-revalidatepath}

オンデマンドでページを無効化するリバリデーションのより正確な方法として、`revalidatePath`関数を使用します。

たとえば、新しい投稿を追加した後にこのServer Actionが呼び出されます。`fetch`を使用するか、データベースに接続するかに関わらず、Server Componentでどのようにデータを取得するかに関係なく、これはルート全体のキャッシュをクリアし、Server Componentが新鮮なデータを取得できるようにします。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="app/actions.ts" switcher

import { revalidatePath } from 'next/cache'

export async function createPost() {
  // キャッシュ内の/postsルートを無効化します
  revalidatePath('/posts')
}
```

</TabItem>

### `res.revalidate()`を使用したオンデマンドリバリデーション {#on-demand-validation-with-res-revalidate}

リバリデーションをより正確に行う方法として、`res.revalidate`を使用してAPI Routerからオンデマンドで新しいページを生成します。

たとえば、このAPI Routeは`/api/revalidate?secret=<token>` で特定のブログ投稿をリバリデートするように呼び出すことができます。Next.jsアプリだけが知っているシークレットトークンを作成します。このシークレットは、リバリデーションAPI Routeへの不正アクセスを防止するために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // シークレットを確認してこのリクエストが有効であることを確認
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // これは書き換えられたパスではなく、実際のパスであるべきです
    // 例："/posts/[id]"の場合は、"/posts/1"とします
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーがあった場合、Next.jsは
    // 最後に正常に生成されたページを表示し続けます
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### キャッチされない例外の処理 {#handling-uncaught-exceptions}

<AppOnly>

データのリバリデートを試みてエラーがスローされた場合、最後に成功した生成データがキャッシュから提供され続けます。次回以降のリクエスト時に、Next.jsはデータのリバリデートを再試行します。[エラー処理について詳しく学ぶ](/docs/app/building-your-application/routing/error-handling)。

</AppOnly>

<PagesOnly>

背景再生成を処理する際に`getStaticProps`内でエラーがある場合、または例外を手動でスローした場合、最後に正常に生成されたページが引き続き表示されます。次回以降のリクエスト時に、Next.jsは`getStaticProps`の呼び出しを再試行します。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストが捕捉されないエラーをスローした場合、Next.jsは
  // 現在表示されているページを無効にせず
  // 次回のリクエストでgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーがある場合、キャッシュが更新されないように
    // エラーをスローする代わりに返すことを検討してください
    // 次に正常なリクエストが来るまで。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.jsは、リクエストが来たときにキャッシュを無効にします。
    // 最大60秒に1度までです。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュの保存場所のカスタマイズ {#customizing-the-cache-location}

キャッシングとページのリバリデーション（Incremental Static Regenerationを使用）は同じ共有キャッシュを使用します。[Vercelへのデプロイ時](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)、ISRキャッシュは自動的に耐久性のあるストレージに保持されます。

セルフホスティング時には、ISRキャッシュはNext.jsサーバー上のファイルシステム（ディスク上）に保存されます。これは、Pages RouterとApp Routerの両方を使用してセルフホスティング時に自動的に機能します。

キャッシュされたページとデータを耐久性のあるストレージに保存するか、Next.jsアプリケーションの複数のコンテナまたはインスタンス間でキャッシュを共有したい場合は、Next.jsのキャッシュの場所を設定できます。[詳細を学ぶ](/docs/app/building-your-application/deploying#caching-and-isr).

## トラブルシューティング {#troubleshooting}

### ローカル開発におけるキャッシュデータのデバッグ {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、リクエストがキャッシュされているかどうかを理解するために追加のログを追加できます。[`logging`オプションについてもっと知る](/docs/app/api-reference/config/next-config-js/logging)。

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しいプロダクション動作の確認 {#verifying-correct-production-behavior}

ページがプロダクション環境で正しくキャッシュされ、リバリデーションされることを確認するには、`next build`を実行し、その後`next start`を実行してプロダクションのNext.jsサーバーをローカルでテストできます。

これにより、プロダクション環境でISRの動作を確認することができます。さらなるデバッグのために、次の環境変数を`.env`ファイルに追加します：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーのコンソールログでISRキャッシュヒットとミスが表示されます。どのページが`next build`中に生成されるか、およびパスがオンデマンドでアクセスされるとページがどのように更新されるかを出力で確認できます。

## 注意事項 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用している場合にのみサポートされます。
- ISRは[Static Export](/docs/app/building-your-application/deploying/static-exports)を作成する場合にはサポートされません。
- 複数の`fetch`リクエストが静的レンダリングされたルートで行われ、それぞれが異なる`revalidate`頻度を持つ場合、ISRには最短の時間が使用されます。ただし、[Data Cache](/docs/app/building-your-application/caching#data-cache)によってこれらのリバリデート頻度は引き続き尊重されます。
- ルートで使用される`fetch`リクエストのいずれかが`0`の`revalidate`時間を持つ場合、または明示的な`no-store`の場合、ルートは[動的にレンダリングされます](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)。
- オンデマンドISRリクエストのためにミドルウェアは実行されません。したがって、ミドルウェア内のパス書き換えやロジックは適用されません。正確なパスをリバリデートしていることを確認してください。たとえば、書き換えられた`/post-1`ではなく、`/post/1`です。

</AppOnly>
