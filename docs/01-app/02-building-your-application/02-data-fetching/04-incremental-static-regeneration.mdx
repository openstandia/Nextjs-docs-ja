Incremental Static Regeneration (ISR)を使用すると、次のことができます：

- サイト全体を再構築せずに静的コンテンツを更新する
- ほとんどのリクエストに対して事前レンダリングされた静的ページを提供することによりサーバーの負荷を軽減する
- 適切な`cache-control`ヘッダーが自動的にページに追加されることを保証する
- 長い`next build`時間を避けて大量のコンテンツページを処理する

以下に最小限の例を示します：

```tsx title="app/blog/[id]/page.tsx" switcher
interface Post {
  id: string
  title: string
  content: string
}

// キャッシュを無効化しますが、最大で60秒に1回だけです。
export const revalidate = 60

// `generateStaticParams`からのparamsのみをビルド時に事前レンダリングします。
// 生成されていないパスのリクエストがある場合、
// Next.jsはオンデマンドでページをサーバーレンダリングします。
export const dynamicParams = true // またはfalseに設定して未知のパスで404を返します

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // ビルド時にこれらのパスのみを事前レンダリングします。
  // { fallback: 'blocking' }はパスが存在しない場合に
  // ページをオンデマンドでサーバーレンダリングします。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // Next.jsはリクエストが来たときに
    // キャッシュを無効化しますが、最大で60秒に1回だけです。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (
    <main>
      <h1>{post.title}</h1>

}
```

この例の動作の仕組みは次のとおりです：

1. `next build`中に、すべての既知のブログ投稿が生成されます（この例では25個あります）
2. これらのページ（例：`/blog/1`）へのすべてのリクエストがキャッシュされ、即時に返されます
3. 60秒が経過した後でも、次のリクエストはキャッシュされた（古い）ページを表示し続けます
4. キャッシュが無効化され、バックグラウンドで新しいページの生成が始まります
5. 成功すると、新しいページが表示され、キャッシュされます
6. `/blog/26`がリクエストされた場合、Next.jsはこのページをオンデマンドで生成し、キャッシュします

### 時間に基づいた再検証 {#time-based-revalidation}

これは`/blog`でのブログ投稿のリストを取得して表示します。1時間後、このページのキャッシュは次回のページ訪問時に無効化されます。その後、バックグラウンドで最新のブログ投稿を含む新しいバージョンのページが生成されます。

再検証時間を高く設定することをおすすめします。例えば、1秒間隔ではなく1時間間隔にします。さらに精密さが必要な場合は、オンデマンドでの再検証を検討してください。リアルタイムデータが必要な場合は、[dynamic rendering](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)への切り替えを検討してください。

### `revalidatePath`によるオンデマンド再検証 {#on-demand-revalidation-with-revalidatepath}

より精密な再検証方法として、`revalidatePath`関数を用いてページをオンデマンドで無効化します。

例えば、新しい投稿を追加した後に呼び出されるServer Actionがあります。データの取得方法にかかわらず、Server Componentで`fetch`やデータベース接続を使用する場合でも、これによりルート全体のキャッシュがクリアされ、Server Componentが新しいデータを取得できます。

```ts title="app/actions.ts" switcher
  // キャッシュ内の/postsルートを無効化

}
```

### `res.revalidate()`によるオンデマンド再検証 {#on-demand-validation-with-res-revalidate}

より精密な再検証方法として、`res.revalidate`を使用してAPIルーターからオンデマンドで新しいページを生成します。

例えば、このAPIルートは`/api/revalidate?secret=<token>`で呼び出すことで指定されたブログ投稿を再検証できます。Next.jsアプリでのみ知られているシークレットトークンを作成します。このシークレットは再検証APIルートへの不正アクセスを防ぐために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // 正当なリクエストであることを確認するためのシークレットのチェック
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // 実際のパスでなければなりません
    // 例：`/posts/[id]`の場合、`/posts/1`である必要があります
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーが発生した場合、Next.jsは最後に
    // 正常に生成されたページを引き続き表示します
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### キャッチされていない例外の処理 {#handling-uncaught-exceptions}

<AppOnly>

データ再検証を試みる際にエラーが発生した場合、最後に成功したデータがキャッシュから提供され続けます。次の継続的なリクエストでは、Next.jsはデータ再検証を再試行します。[エラー処理の詳細はこちら](/docs/app/building-your-application/routing/error-handling)をご覧ください。

</AppOnly>

<PagesOnly>

バックグラウンド再生成を処理する際に`getStaticProps`内でエラーが発生した場合、または意図的にエラーを投げた場合、最後に生成に成功したページが引き続き表示されます。次回のリクエスト時に、Next.jsは`getStaticProps`の呼び出しを再試行します。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストでキャッチされないエラーが発生した場合
  // Next.jsは現在表示中のページを無効化せず、
  // 次回のリクエストでgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーがある場合、
    // 次回の成功したリクエストまで
    // キャッシュが更新されないようにエラーを投げたいかもしれません。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.jsはリクエストが来たときに
    // キャッシュを無効化しますが、最大で60秒に1回だけです。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュ位置のカスタマイズ {#customizing-the-cache-location}

インクリメンタルな静的再生成（ISR）を使用してページをキャッシュし、再検証することで、共通のキャッシュを使用します。[Vercelにデプロイ](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)する際、ISRキャッシュは自動的に耐久性のあるストレージに保持されます。

セルフホスティング時には、ISRキャッシュがNext.jsサーバー上のファイルシステム（ディスク）に保存されます。セルフホスティング時にPages RouterとApp Routerの両方を使用する場合は自動的に動作します。

Next.jsアプリケーションの複数のコンテナやインスタンス間でキャッシュを共有する必要がある場合や、キャッシュされたページとデータを耐久性のあるストレージに保持したい場合、Next.jsキャッシュの場所を設定できます。[詳細はこちらを学ぶ](/docs/app/building-your-application/deploying#caching-and-isr)

## トラブルシューティング {#troubleshooting}

### ローカル開発でのキャッシュデータのデバッグ {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、キャッシュされたリクエストとキャッシュされていないリクエストを理解するための追加のロギングを追加できます。[`logging`オプションに関する詳細はこちら](/docs/app/api-reference/config/next-config-js/logging)

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しいプロダクション動作の確認 {#verifying-correct-production-behavior}

本番環境でのページが正しくキャッシュおよび再検証されているかを確認するには、ローカルで`next build`を実行後に`next start`を実行してプロダクション用のNext.jsサーバーを起動してテストできます。

これにより、本番環境でのISR動作をテストできます。さらにデバッグするためには、以下の環境変数を`.env`ファイルに追加してください：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーのコンソールログにISRキャッシュのヒットとミスが記録されます。`next build`中にどのページが生成されたか、オンデマンドのようにパスがアクセスされるとページが更新される方法を確認するために出力を調べることができます。

## 注意点 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用する場合にのみサポートされます。
- ISRは[Static Export](/docs/app/building-your-application/deploying/static-exports)を作成する際にはサポートされていません。
- 静的にレンダリングされたルートに複数の`fetch`リクエストがあり、それぞれ異なる`revalidate`頻度を持つ場合、ISRには最小の時間が使用されますが、これらの再検証頻度は[Data Cache](/docs/app/building-your-application/caching#data-cache)によって依然として尊重されます。
- ルートで使用される`fetch`リクエストのいずれかに`revalidate`時間が`0`、または明示された`no-store`がある場合、そのルートは[動的にレンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)されます。
- オンデマンドISRリクエストのためにミドルウェアは実行されないため、ミドルウェア内のパス書き換えやロジックは適用されません。正確なパスを再検証していることを確認してください。例えば、`/post/1`のように書き換えられた`/post-1`ではない場合。

</AppOnly>
