Incremental Static Regeneration (ISR) では、次のことが可能です：

- サイト全体を再構築せずに静的コンテンツを更新
- ほとんどのリクエストに対してプリレンダリングされた静的ページを提供することでサーバー負荷を軽減
- 適切な `cache-control` ヘッダーがページに自動で追加されるようにする
- 大量のコンテンツページを長い `next build` 時間なしで処理

以下は最小限の例です：

interface Post {
  id: string
  title: string
  content: string
}

// Next.jsはリクエストが入ると、最大60秒ごとにキャッシュを無効にします。
export const revalidate = 60

// `generateStaticParams` からのパラメータのみビルド時間にプリレンダリングされます。
// 生成されていないパスへのリクエストが入った場合、
// Next.jsはそのページをオンデマンドでサーバーレンダリングします。
export const dynamicParams = true // 未知のパスで404にするにはfalseにします

export async function generateStaticParams() {
  const posts: Post[] = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  return posts.map((post) => ({
    id: String(post.id),
  }))
}

export default async function Page({
  params,
}: {
  params: Promise<{ id: string }>
}) {
  const id = (await params).id
  const post: Post = await fetch(`https://api.vercel.app/blog/${id}`).then(
    (res) => res.json()
  )
  return (
    <main>
      <h1>{post.title}</h1>
      <p>{post.content}</p>
    </main>
  )
}
```

import type { GetStaticPaths, GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticPaths: GetStaticPaths = async () => {
  const posts = await fetch('https://api.vercel.app/blog').then((res) =>
    res.json()
  )
  const paths = posts.map((post: Post) => ({
    params: { id: String(post.id) },
  }))

  // これらのパスのみビルド時間にプリレンダリングされます。
  // { fallback: 'blocking' } は、パスが存在しない場合にオンデマンドでページをサーバーレンダリングします。
  return { paths, fallback: false }
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  const post = await fetch(`https://api.vercel.app/blog/${params.id}`).then(
    (res) => res.json()
  )

  return {
    props: { post },
    // Next.jsはリクエストが入ると、最大60秒ごとにキャッシュを無効にします。
    revalidate: 60,
  }
}

export default function Page({ post }: Props) {
  return (

    </main>
  )
}

この例がどのように動作するかは以下の通りです：

1. `next build` 中に、既知のブログ投稿のすべてが生成されます（この例では25件）
2. これらのページに対するすべてのリクエスト（例： `/blog/1`）はキャッシュされ、瞬時に応答します
3. 60秒が経過すると、次のリクエストはキャッシュされた（古い）ページを表示します
4. キャッシュが無効化され、新しいバージョンのページの生成がバックグラウンドで開始されます
5. 正常に生成されると、Next.jsは更新されたページを表示しキャッシュします
6. `/blog/26` がリクエストされた場合、Next.jsはこのページをオンデマンドで生成しキャッシュします

### 時間ベースの再検証 {#time-based-revalidation}

これは `/blog` でブログ投稿のリストをフェッチし表示します。1時間後、このページのキャッシュが次の訪問時に無効化されます。そしてバックグラウンドで、新しいバージョンのページが最新のブログ投稿で生成されます。

      <ul>

          <li key={post.id}>{post.title}</li>

      </ul>

再検証の時間を長く設定することを推奨します。たとえば、1秒ではなく1時間にします。より精密さが必要な場合は、オンデマンド再検証の使用を検討してください。リアルタイムデータが必要な場合は、[動的レンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)に切り替えることを検討してください。

### `revalidatePath`を使用したオンデマンド再検証 {#on-demand-revalidation-with-revalidatepath}

より正確な再検証方法として、`revalidatePath` 関数を使用してオンデマンドでページを無効化します。

たとえば、新しい投稿を追加した後に呼び出されるこのServer Actionがあります。Server Componentでデータを取得するために `fetch` を使用するかデータベースに接続するかに関わらず、これによりルート全体のキャッシュがクリアされ、Server Componentが新しいデータを取得できるようになります。

export async function createPost() {
  // キャッシュ内の/postsルートを無効化する
  revalidatePath('/posts')
}
```

</TabItem>

### `res.revalidate()` を使用したオンデマンド検証 {#on-demand-validation-with-res-revalidate}

より正確な再検証方法として、API Routerから `res.revalidate` を使用してオンデマンドで新しいページを生成します。

たとえば、このAPI Routeは `/api/revalidate?secret=<token>` で呼び出され、指定されたブログ投稿を再検証します。Next.jsアプリだけが知っているシークレットトークンを作成します。このシークレットは、再検証API Routeへの不正アクセスを防ぐために使用されます。

<Tabs>
<TabItem value="ts" label="TypeScript">

```ts title="pages/api/revalidate.ts" switcher
import type { NextApiRequest, NextApiResponse } from 'next'

export default async function handler(
  req: NextApiRequest,
  res: NextApiResponse
) {
  // このリクエストが有効であることを確認するための秘密のチェック
  if (req.query.secret !== process.env.MY_SECRET_TOKEN) {
    return res.status(401).json({ message: 'Invalid token' })
  }

  try {
    // これは書き換えられたパスではなく、実際のパスであるべきです。
    // 例えば、"/posts/[id]" の場合は、"/posts/1" であるべきです。
    await res.revalidate('/posts/1')
    return res.json({ revalidated: true })
  } catch (err) {
    // エラーがあった場合、Next.jsは引き続き
    // 最後に正常に生成されたページを表示します。
    return res.status(500).send('Error revalidating')
  }
}
```

</TabItem>

### 許可されていない例外の処理 {#handling-uncaught-exceptions}

<AppOnly>

データを再検証しようとする際にエラーが発生した場合、最後に正常に生成されたデータがキャッシュから引き続き提供されます。次のリクエスト時にNext.jsはデータの再検証を試みます。[エラー処理について詳しく知る](/docs/app/building-your-application/routing/error-handling)。

</AppOnly>

<PagesOnly>

バックグラウンド再生成の際に `getStaticProps` 内でエラーが発生した場合、または手動でエラーを投げた場合、最後に正常に生成されたページが引き続き表示されます。次のリクエスト時にNext.jsは `getStaticProps` の呼び出しを再試行します。

<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="pages/blog/[id].tsx" switcher
import type { GetStaticProps } from 'next'

interface Post {
  id: string
  title: string
  content: string
}

interface Props {
  post: Post
}

export const getStaticProps: GetStaticProps<Props> = async ({
  params,
}: {
  params: { id: string }
}) => {
  // このリクエストで許可されていないエラーを投げた場合、
  // Next.jsは現在表示されているページを無効にせず、
  // 次のリクエスト時にgetStaticPropsを再試行します。
  const res = await fetch(`https://api.vercel.app/blog/${params.id}`)
  const post: Post = await res.json()

  if (!res.ok) {
    // サーバーエラーがある場合は、エラーを投げて
    // キャッシュが次の成功したリクエストまで更新されないようにすることも考えられます。
    throw new Error(`Failed to fetch posts, received status ${res.status}`)
  }

  return {
    props: { post },
    // Next.jsはリクエストが入ると、最大60秒ごとにキャッシュを無効にします。
    revalidate: 60,
  }
}
```

</TabItem>

### キャッシュの場所のカスタマイズ {#customizing-the-cache-location}

ページのキャッシングおよび再検証（Incremental Static Regenerationを使用）は、同じ共有キャッシュを使用します。[Vercelにデプロイする際](https://vercel.com/docs/incremental-static-regeneration?utm_source=next-site&utm_medium=docs&utm_campaign=next-website)、ISRキャッシュは自動的に耐久ストレージに保存されます。

自社ホスティングの場合、ISRキャッシュはNext.jsサーバー上のファイルシステム（ディスク上）に保存されます。PagesとApp Routerの両方を使用した自社ホスティングで自動的に機能します。

キャッシュされたページとデータを永続的なストレージに保存したり、Next.jsアプリの複数のコンテナやインスタンス間でキャッシュを共有したい場合は、Next.jsのキャッシュ場所を設定できます。[詳細はこちら](/docs/app/building-your-application/deploying#caching-and-isr)。

## トラブルシューティング {#troubleshooting}

### ローカル開発におけるキャッシュされたデータのデバッグ {#debugging-cached-data-in-local-development}

`fetch` APIを使用している場合、リクエストがキャッシュされているかどうかを理解するために追加のログを追加できます。[`logging` オプションの詳細](/docs/app/api-reference/config/next-config-js/logging)について学びましょう。

```jsx title="next.config.js"
module.exports = {
  logging: {
    fetches: {
      fullUrl: true,
    },
  },
}
```

### 正しい本番挙動の確認 {#verifying-correct-production-behavior}

本番環境でページが正しくキャッシュされ再検証されることを確認するには、ローカルで `next build` を実行し、その後 `next start` を実行して本番Next.jsサーバーを稼働させることで、ローカルでテストできます。

これにより、本番環境でのISRの挙動をテストできます。さらにデバッグを行うために、次の環境変数を `.env` ファイルに追加してください：

```bash title=".env"
NEXT_PRIVATE_DEBUG_CACHE=1
```

これにより、Next.jsサーバーのコンソールはISRキャッシュのヒットやミスをログに表示します。この出力を検査して、`next build` 中にどのページが生成されたか、またどのようにしてページがオンデマンドでアクセスされたときに更新されたかを確認できます。

## 注意点 {#caveats}

<AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用しているときにのみサポートされています。
- ISRは[Static Export](/docs/app/building-your-application/deploying/static-exports)を作成するときにサポートされていません。
- 静的にレンダリングされたルートに複数の `fetch` リクエストがあり、それぞれが異なる `revalidate` 頻度を持つ場合、ISRには最小の時間が使用されます。ただし、これらの再検証頻度は[Data Cache](/docs/app/building-your-application/caching#data-cache)によって引き続き尊重されます。
- ルートで使用される `fetch` リクエストのいずれかに `revalidate` の時間が `0` または明示的な `no-store` がある場合、ルートは[動的にレンダリング](/docs/app/building-your-application/rendering/server-components#dynamic-rendering)されます。
- オンデマンドISRリクエストに対してMiddlewareは実行されないため、Middleware内のパスの書き換えやロジックは適用されません。正確なパスで再検証していることを確認します。たとえば、書き換えられた `/post-1` ではなく、`/post/1` のようにします。

</AppOnly>

- ISRはNode.jsランタイム（デフォルト）を使用しているときにのみサポートされています。
- ISRは[Static Export](/docs/app/building-your-application/deploying/static-exports)を作成するときにサポートされていません。
- オンデマンドISRリクエストに対してMiddlewareは実行されないため、Middleware内のパスの書き換えやロジックは適用されません。正確なパスで再検証していることを確認します。たとえば、書き換えられた `/post-1` ではなく、`/post/1` のようにします。
