プログレッシブ ウェブ アプリケーション (PWA) は、ウェブアプリケーションの到達性とアクセシビリティを提供しつつ、ネイティブ モバイル アプリの機能とユーザー エクスペリエンスを組み合わせたものです。Next.js を使用すると、マルチコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスでアプリのようなエクスペリエンスを提供する PWA を作成できます。

PWA は次のことを可能にします：

- アプリストアの承認を待たずにアップデートを即座に展開する
- 単一のコードベースでクロスプラットフォーム アプリケーションを作成する
- ホーム画面へのインストールやプッシュ通知といったネイティブのような機能を提供する

## Next.jsでPWAを作成する {#creating-a-pwa-with-next-js}

### 1. ウェブ アプリ マニフェストを作成する {#1-creating-the-web-app-manifest}

Next.js は App Router を使用して[ウェブ アプリ マニフェスト](/docs/app/api-reference/file-conventions/metadata/manifest) を作成するための組み込みサポートを提供しています。静的または動的なマニフェストファイルを作成できます：

例えば、`app/manifest.ts` または `app/manifest.json` ファイルを作成します：

このファイルには、名前やアイコン、ユーザーのデバイスにアイコンとして表示される方法に関する情報を含める必要があります。これにより、ユーザーはホーム スクリーンに PWA をインストールできます。ネイティブのアプリのようなエクスペリエンスを提供します。

[favicon generators](https://realfavicongenerator.net/) のようなツールを使用して、異なるアイコンセットを作成し、生成されたファイルを `public/` フォルダに配置することができます。

### 2. Webプッシュ通知の実装 {#2-implementing-web-push-notifications}

Webプッシュ通知は、最新のすべてのブラウザでサポートされています。次の環境を含みます：

- ホーム画面にインストールされたアプリケーション向けの iOS 16.4+
- macOS 13 またはそれ以降の Safari 16
- Chromiumベースのブラウザ
- Firefox

このため、PWAはネイティブアプリの有効な代替手段になります。特に、オフラインサポートなしでインストールプロンプトをトリガーすることができます。

Webプッシュ通知は、ユーザーがアプリを積極的に使用していないときでも再度関与させることができます。Next.jsアプリケーションでの実装方法は次のとおりです：

まず、`app/page.tsx` にメインページコンポーネントを作成します。理解を深めるために、小さな部分に分割します。まず、必要なインポートとユーティリティの一部を追加します。参照されているサーバーアクションはまだ存在しなくても問題ありません：

```tsx switcher
'use client'

// 必要なインポート
import { useState, useEffect } from 'react'
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

// Base64文字列をUint8Arrayに変換する関数
function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

次に、プッシュ通知のサブスクライブ、解除、送信を管理するコンポーネントを追加します。

```tsx switcher
// プッシュ通知マネージャコンポーネント
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  // サービスワーカーを登録
  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  // プッシュ通知にサブスクライブ
  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  // プッシュ通知のサブスクライブを解除
  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  // テスト通知を送信
  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>このブラウザではプッシュ通知はサポートされていません。</p>
  }

  return (
    <div>
      <h3>プッシュ通知</h3>
      {subscription ? (
        <>
          <p>プッシュ通知を購読しています。</p>
          <button onClick={unsubscribeFromPush}>登録解除</button>
          <input
            type="text"
            placeholder="通知メッセージを入力"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>テストを送信</button>
        </>
      ) : (
        <>
          <p>プッシュ通知を購読していません。</p>
          <button onClick={subscribeToPush}>登録する</button>
        </>
      )}
    </div>
  )
}
```

### 3. Server Actions の実装 {#3-implementing-server-actions}

アクションを含めるために、新しいファイルを `app/actions.ts` に作成します。このファイルは、サブスクリプションの作成、削除、通知の送信を処理します。

```tsx title="app/actions.ts" switcher
'use server'

import webpush from 'web-push'

// VAPIDの詳細を設定
webpush.setVapidDetails(
  '<mailto:your-email@example.com>',
  process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!,
  process.env.VAPID_PRIVATE_KEY!
)

let subscription: PushSubscription | null = null

// ユーザーをサブスクライブする関数
export async function subscribeUser(sub: PushSubscription) {
  subscription = sub
  // 実運用環境では、データベースにサブスクリプションを保存することをお勧めします
  // 例えば：await db.subscriptions.create({ data: sub })
  return { success: true }
}

// ユーザーサブスクリプションを解除する関数
export async function unsubscribeUser() {
  subscription = null
  // 実運用環境では、データベースからサブスクリプションを削除することをお勧めします
  // 例えば：await db.subscriptions.delete({ where: { ... } })
  return { success: true }
}

// 通知を送信する関数
export async function sendNotification(message: string) {
  if (!subscription) {
    throw new Error('利用可能なサブスクリプションがありません')
  }

  try {
    await webpush.sendNotification(
      subscription,
      JSON.stringify({
        title: 'テスト通知',
        body: message,
        icon: '/icon.png',
      })
    )
    return { success: true }
  } catch (error) {
    console.error('プッシュ通知の送信に失敗しました:', error)
    return { success: false, error: '通知の送信に失敗しました' }
  }
}
```

実運用環境では、サーバーの再起動時に永続性を保ち、複数のユーザーのサブスクリプションを管理するために、サブスクリプションをデータベースに保存することをお勧めします。

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push API を使用するには、[VAPID](https://vapidkeys.com/) キーを生成する必要があります。最も簡単な方法は、web-push CLI を直接使用することです。

まず、web-push をグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

次に、VAPIDキーを生成します：

```bash title="Terminal"
web-push generate-vapid-keys
```

出力をコピーし、キーを `.env` ファイルに貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. サービスワーカーの作成 {#5-creating-a-service-worker}

`public/sw.js` ファイルを作成し、サービスワーカーを記述します：

```js title="public/sw.js"
// プッシュイベントをリッスン
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

// 通知クリックイベントをリッスン
self.addEventListener('notificationclick', function (event) {
  console.log('通知クリックを受信しました。')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このサービスワーカーはカスタム画像や通知をサポートしています。プッシュイベントと通知クリックを扱います。

- `icon`や`badge`プロパティを使用して通知のカスタムアイコンを設定できます。
- `vibrate`パターンを調整して、サポートされているデバイスでカスタムバイブレーションアラートを作成できます。
- `data`プロパティを使用して通知に追加データを添付できます。

サービスワーカーが、異なるデバイスやブラウザで期待通りに動作するかを入念にテストしてください。また、`notificationclick` イベントリスナー内の `'https://your-website.com'` リンクを、あなたのアプリケーションの適切なURLに更新することを忘れないでください。

### 6. ホームスクリーンへの追加 {#6-adding-to-home-screen}

ステップ2で定義された `InstallPrompt` コンポーネントは、iOSデバイスにホームスクリーンにインストールする方法を指示するメッセージを表示します。

モバイルホームスクリーンにアプリをインストールできるようにするには、以下の条件を満たす必要があります：

1. 有効なウェブアプリマニフェスト（ステップ1で作成した）
2. HTTPS 上で提供されるウェブサイト

これらの条件が満たされている場合、モダンブラウザは自動的にユーザーにインストールプロンプトを表示します。[`beforeinstallprompt`](https://developer.mozilla.org/en-US/docs/Web/API/Window/beforeinstallprompt_event) を使用してカスタムインストールボタンを提供することもできますが、これはクロスブラウザおよびプラットフォームに対応していないため（Safari iOS では動作しないため）、お勧めしません。

### 7. ローカルでのテスト {#7-testing-locally}

通知をローカルで確認するためには、以下のことを確認してください：

- [HTTPSでローカルを実行する](/docs/app/api-reference/cli/next#using-https-during-development)
  - テストのために `next dev --experimental-https` を使用
- 使用するブラウザ（Chrome、Safari、Firefox）に通知が有効になっていること
  - ローカルでプロンプトが表示されたときに、通知の使用許可を受け入れ
  - 通知がブラウザ全体で無効になっていないことを確認する
  - 通知が表示されない場合は、別のブラウザを使用してデバッグすることを試してみる

### 8. アプリケーションのセキュリティ {#8-securing-your-application}

セキュリティは、特にPWAにおいてあらゆるウェブアプリケーションにおいて重要な要素です。Next.js は `next.config.js` ファイルを使用してセキュリティヘッダーを設定することが可能です。例：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

各オプションを説明します：

1. グローバルヘッダー（すべてのルートに適用）：
   1. `X-Content-Type-Options: nosniff`: MIMEタイプスニッフィングを防ぎ、不正なファイルアップロードのリスクを低減します
   2. `X-Frame-Options: DENY`: クロスサイトスクリプティング攻撃から保護するため、サイトがiframeに組み込まれるのを防ぎます
   3. `Referrer-Policy: strict-origin-when-cross-origin`: セキュリティと機能のバランスを取りつつ、リファラー情報の量を制御します
2. サービスワーカー特有のヘッダー：
   1. `Content-Type: application/javascript; charset=utf-8`: サービスワーカーがJavaScriptとして正しく解釈されるようにします
   2. `Cache-Control: no-cache, no-store, must-revalidate`: サービスワーカーのキャッシュを防ぎ、常に最新バージョンを取得できるようにします
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`: 同じオリジンからのスクリプトのみを許可する厳格なコンテンツセキュリティポリシーをサービスワーカーに適用します

Next.js を使用して[コンテンツセキュリティポリシー](/docs/app/building-your-application/configuring/content-security-policy) を定義する方法についてさらに知ることができます。

## 次のステップ {#next-steps}

1. **PWAの機能を探る**: PWAはさまざまなウェブAPIを活用して、より高度な機能を提供することができます。バックグラウンド同期、定期的なバックグラウンド同期、またはファイルシステムアクセスAPIなどの機能を調査してアプリケーションを強化することを検討してください。インスピレーションや最新情報を得るために、[What PWA Can Do Today](https://whatpwacando.today/)のようなリソースを参照できます。
2. **静的エクスポート**: アプリケーションがサーバーを動かさずに代わりにファイルの静的エクスポートを使用する必要がある場合、Next.js の設定を更新してこの変更を有効にできます。詳細は[Next.js の静的エクスポートドキュメント](/docs/app/building-your-application/deploying/static-exports)をご覧ください。ただし、Server Actionsから外部APIを呼び出すように移行し、定義したヘッダーをプロキシに移動する必要があるでしょう。
3. **オフラインサポート**: オフライン機能を提供するためのオプションの1つは、Next.jsと[Serwist](https://github.com/serwist/serwist)を使用することです。Next.jsとのSerwistの統合方法の例は、彼らの[ドキュメント](https://github.com/serwist/serwist/tree/main/examples/next-basic)で見つけることができます。**注意**：このプラグインは現在webpack設定が必要です。
4. **セキュリティ対策**: サービスワーカーが適切に保護されていることを確認してください。これには、HTTPSの使用、プッシュメッセージのソースの検証、適切なエラーハンドリングの実装が含まれます。
5. **ユーザーエクスペリエンス**: PWA機能がユーザーのブラウザでサポートされていない場合でも、アプリが問題なく動作するために漸進的な機能向上の技術を実装することを検討してください。
