Progressive Web Applications（PWAs）は、ネイティブモバイルアプリの機能とユーザーエクスペリエンスを組み合わせた、Webアプリケーションの利便性とアクセスしやすさを提供します。Next.jsを使用すると、複数のコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスかつアプリのような体験を提供するPWAを作成できます。

PWAsを利用することで、以下のことが可能になります：

- アプリストアの承認を待たずに、即座にアップデートをデプロイ
- 単一のコードベースでクロスプラットフォームのアプリケーションを作成
- ホーム画面へのインストールやプッシュ通知など、ネイティブのような機能を提供

## Next.jsでPWAを作成する {#creating-a-pwa-with-next-js}

### 1. Webアプリマニフェストの作成 {#1-creating-the-web-app-manifest}

Next.jsは、App Routerを使用した[webアプリマニフェスト](/docs/app/api-reference/file-conventions/metadata/manifest)の作成をサポートしています。静的または動的なマニフェストファイルを作成することができます：

例えば、`app/manifest.ts`または`app/manifest.json`ファイルを作成します：

このファイルには、名前、アイコン、そしてユーザーのデバイスにアイコンとしてどのように表示されるかに関する情報を含める必要があります。これにより、ユーザーはホーム画面にPWAをインストールすることができ、ネイティブアプリのような体験を提供します。

異なるアイコンセットを作成し、生成されたファイルを`public/`フォルダに配置するために、[Faviconジェネレーター](https://realfavicongenerator.net/)のようなツールを使用することができます。

### 2. Webプッシュ通知の実装 {#2-implementing-web-push-notifications}

Webプッシュ通知は、以下のすべての最新ブラウザでサポートされています：

- ホーム画面にインストールされたアプリケーション用のiOS 16.4+
- macOS 13以降のSafari 16
- Chromiumベースのブラウザ
- Firefox

これはPWAsがネイティブアプリの代替として有効な理由です。特に、オフラインサポートを必要とせずにインストールプロンプトを起動できる点が注目されます。

Webプッシュ通知は、ユーザーがアプリを積極的に使用していないときでもユーザーとの関係を維持することができます。Next.jsアプリケーションでそれらを実装する方法を紹介します：

まず、`app/page.tsx`内にメインページコンポーネントを作成します。理解を深めるために、小さく分割して進めます。まず必要なインポートとユーティリティを追加します。まだ存在していないServer Actionsが参照されていても問題ありません：

```tsx switcher
'use client'

import { useState, useEffect } from 'react'
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

// base64文字列をUint8Arrayに変換する関数
function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

次に、プッシュ通知の購読、解約、送信を管理するコンポーネントを追加しましょう。

```tsx switcher
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>このブラウザではプッシュ通知はサポートされていません。</p>
  }

  return (
    <div>
      <h3>Push Notifications</h3>
      {subscription ? (
        <>
          <p>プッシュ通知に購読しています。</p>
          <button onClick={unsubscribeFromPush}>Unsubscribe</button>
          <input
            type="text"
            placeholder="Enter notification message"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>Send Test</button>
        </>
      ) : (
        <>
          <p>プッシュ通知に購読していません。</p>
          <button onClick={subscribeToPush}>Subscribe</button>
        </>
      )}
    </div>
  )
}
```

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push APIを使用するには、VAPIDキーを生成する必要があります。最も簡単な方法は、Web-push CLIを直接使用することです：

まず、Web-pushをグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

VAPIDキーを生成するには、以下を実行します：

```bash title="Terminal"
web-push generate-vapid-keys
```

出力されたキーをコピーし、`.env`ファイルに貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. Service Workerの作成 {#5-creating-a-service-worker}

service worker用に`public/sw.js`ファイルを作成します：

```js title="public/sw.js"
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

self.addEventListener('notificationclick', function (event) {
  console.log('通知のクリックが受信されました。')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このservice workerはカスタム画像と通知をサポートします。受信したプッシュイベントと通知クリックを処理します。

- 通知のカスタムアイコンは、`icon`と`badge`プロパティを使用して設定できます。
- `vibrate`パターンは、サポートされているデバイスでカスタム振動アラートを作成するために調整できます。
- `data`プロパティを使用して、通知に追加データを添付することができます。

service workerが異なるデバイスやブラウザで期待通りに動作することを十分にテストすることを忘れないでください。また、`notificationclick`イベントリスナーの中の`'https://your-website.com'`リンクをアプリケーションに適切なURLに更新してください。

### 6. ホーム画面に追加 {#6-adding-to-home-screen}

ステップ2で定義した`InstallPrompt`コンポーネントは、iOSデバイスに対しホーム画面にインストールし指示するメッセージを表示します。

モバイルのホーム画面にインストールできるようにするためには、以下が必要です：

1. 有効なWebアプリマニフェスト（ステップ1で作成）
2. HTTPSで提供されるWebサイト

これらの条件が満たされると、現代のブラウザは自動的にユーザーにインストールのプロンプトを表示します。[`beforeinstallprompt`](https://developer.mozilla.org/en-US/docs/Web/API/Window/beforeinstallprompt_event)を使用してカスタムインストールボタンを提供することができますが、Safari iOSでは動作しないため、クロスブラウザやプラットフォームでの使用は推奨されません。

### 7. ローカルでのテスト {#7-testing-locally}

ローカルで通知を確認できるようにするために、以下を確認してください：

- [HTTPSでローカル実行](/docs/app/api-reference/cli/next#using-https-during-development)していること
  - `next dev --experimental-https`を使用してテスト
- ブラウザ（Chrome、Safari、Firefox）に通知が有効になっていること
  - ローカルでプロンプトされた際に、通知の使用権限を受け入れること
  - ブラウザ全体で通知が無効になっていないことを確認すること
  - 通知が表示されない場合は、デバッグに別のブラウザを使用してみてください

### 8. アプリケーションのセキュリティ確保 {#8-securing-your-application}

セキュリティは任意のWebアプリケーション、特にPWAsにおいて重要な側面です。Next.jsを使えば、`next.config.js`ファイルを使ってセキュリティヘッダーを設定できます。例えば：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

これらのオプションをそれぞれ確認しましょう：

1. グローバルヘッダー（すべてのルートに適用）:
   1. `X-Content-Type-Options: nosniff`: MIMEタイプのスニッフィングを防止し、悪意のあるファイルアップロードのリスクを削減します。
   2. `X-Frame-Options: DENY`: クリックジャッキング攻撃から保護し、サイトがiframeに埋め込まれることを防止します。
   3. `Referrer-Policy: strict-origin-when-cross-origin`: 要求に含まれるリファラー情報の量を制御し、セキュリティと機能のバランスをとります。
2. Service Worker専用ヘッダー：
   1. `Content-Type: application/javascript; charset=utf-8`: service workerがJavaScriptとして正しく解釈されるようにします。
   2. `Cache-Control: no-cache, no-store, must-revalidate`: service workerのキャッシュを防止し、ユーザーが常に最新バージョンを取得できるようにします。
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`: 厳格なコンテンツセキュリティポリシーをservice workerに実装し、同じオリジンからのスクリプトのみを許可します。

Next.jsでの[コンテンツセキュリティポリシーの定義](/docs/app/building-your-application/configuring/content-security-policy)についてさらに学びましょう。

## 次のステップ {#next-steps}

1. **PWAの機能を探る**: PWAsはさまざまなWeb APIを活用して高度な機能を提供できます。アプリケーションを強化するためにバックグラウンド同期、定期的なバックグラウンド同期、またはファイルシステムアクセスAPIなどの機能を検討してください。PWAの機能に関する最新情報やインスピレーションについては、[What PWA Can Do Today](https://whatpwacando.today/)などのリソースをご参照ください。
2. **静的エクスポート**: アプリケーションがサーバーを実行せず、代わりにファイルの静的エクスポートを使用する必要がある場合、Next.jsの設定を更新してこの変更を有効にすることができます。詳細は[Next.js静的エクスポートドキュメント](/docs/app/building-your-application/deploying/static-exports)を参照してください。ただし、Server Actionsから外部APIを呼び出す方法に変更し、定義したヘッダーをプロキシに移行する必要があります。
3. **オフラインサポート**: オフライン機能を提供するためのオプションとして、Next.jsを使用した[Serwist](https://github.com/serwist/serwist)があります。Next.jsとSerwistを統合する方法の例は、[ドキュメント](https://github.com/serwist/serwist/tree/main/examples/next-basic)で確認できます。 **注:**このプラグインには現在webpackの設定が必要です。
4. **セキュリティの考慮事項**: あなたのservice workerが適切にセキュリティ対策されていることを確認してください。これにはHTTPSの使用、プッシュメッセージのソースの検証、適切なエラーハンドリングの実装が含まれます。
5. **ユーザーエクスペリエンス**: プログレッシブエンハンスメント技術を実装することを検討し、ユーザーのブラウザが一部のPWA機能をサポートしていない場合でもアプリが正常に機能するようにしましょう。
