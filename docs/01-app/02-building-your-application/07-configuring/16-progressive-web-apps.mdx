Progressive Web Applications (PWAs) は、ネイティブモバイルアプリの機能とユーザー体験を組み合わせた、Webアプリケーションの到達性とアクセシビリティを提供します。Next.jsを使用することで、複数のコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスでアプリのような体験を提供するPWAを作成することができます。

PWAsを使用すると、次のことが可能になります：

- アプリストアの承認を待たずに更新を即座にデプロイすること
- 単一のコードベースでクロスプラットフォームアプリケーションを作成すること
- ホーム画面へのインストールやプッシュ通知など、ネイティブのような機能を提供すること

## Next.jsを使用したPWAの作成 {#creating-a-pwa-with-next-js}

### 1. Webアプリマニフェストの作成 {#1-creating-the-web-app-manifest}

Next.jsはApp Routerを使って[webアプリマニフェスト](/docs/app/api-reference/file-conventions/metadata/manifest)を作成するための組み込みサポートを提供しています。静的または動的なマニフェストファイルを作成できます：

例として、`app/manifest.ts`または`app/manifest.json`ファイルを作成します：

このファイルには、ユーザーのデバイスでどのようにアイコンとして表示されるかについての名前やアイコン情報が含まれている必要があります。これにより、ユーザーがホーム画面にPWAをインストールでき、ネイティブアプリのような体験が提供されます。

[favicon generators](https://realfavicongenerator.net/)のようなツールを使用して、さまざまなアイコンセットを作成し、生成されたファイルを`public/`フォルダに配置することができます。

### 2. Webプッシュ通知の実装 {#2-implementing-web-push-notifications}

Webプッシュ通知は、最新のブラウザでサポートされています。以下の環境が含まれます：

- ホーム画面にインストールされたアプリケーションのためのiOS 16.4+
- macOS 13以降のSafari 16
- Chromiumベースのブラウザ
- Firefox

これにより、PWAsはネイティブアプリの現実的な代替となります。特に、オフラインサポートなしでインストールプロンプトをトリガーすることが可能です。

Webプッシュ通知を使用することで、ユーザーがアプリを積極的に使用していないときでも、再びユーザーに関心を持たせることができます。Next.jsアプリケーションでそれを実装する方法について以下に説明します：

まず、`app/page.tsx`にメインのページコンポーネントを作成しましょう。より理解しやすくするために、小さな部分に分けて説明します。まず、必要なインポートとユーティリティを追加します。参照されているServer Actionsがまだ存在していなくても問題ありません：

```tsx switcher
'use client'

import { useState, useEffect } from 'react'
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

それでは、プッシュ通知の登録、解除、および送信を管理するコンポーネントを追加します。

```tsx switcher
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>プッシュ通知はこのブラウザでサポートされていません。</p>
  }

  return (
    <div>
      <h3>プッシュ通知</h3>
      {subscription ? (
        <>
          <p>プッシュ通知に登録されています。</p>
          <button onClick={unsubscribeFromPush}>登録解除</button>
          <input
            type="text"
            placeholder="通知メッセージを入力"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>テストを送信</button>
        </>
      ) : (
        <>
          <p>プッシュ通知に登録されていません。</p>
          <button onClick={subscribeToPush}>登録</button>
        </>
      )}
    </div>
  )
}
```

### 3. Server Actionsの実装 {#3-implementing-server-actions}

`app/actions.ts`でアクションを含む新しいファイルを作成します。このファイルは、サブスクリプションの作成、削除、および通知の送信を処理します。

通知の送信は、ステップ5で作成されたサービスワーカーによって処理されます。

本番環境では、サーバーの再起動時にサブスクリプションを保持するため、データベースにサブスクリプションを保存し、複数のユーザーのサブスクリプションを管理することが推奨されます。

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push APIを使用するには、[VAPID](https://vapidkeys.com/)キーを生成する必要があります。一番簡単な方法はweb-push CLIを直接使用することです：

まず、web-pushをグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

CLIを使用してVAPIDキーを生成します：

```bash title="Terminal"
web-push generate-vapid-keys
```

出力をコピーし、キーを`.env`ファイルに貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. サービスワーカーの作成 {#5-creating-a-service-worker}

サービスワーカーのために`public/sw.js`ファイルを作成します：

```js title="public/sw.js"
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

self.addEventListener('notificationclick', function (event) {
  console.log('Notification click received.')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このサービスワーカーはカスタム画像と通知をサポートしています。受信プッシュイベントと通知クリックを処理します。

- 通知のカスタムアイコンを`icon`および`badge`プロパティを使用して設定できます。
- `vibrate`パターンは、サポートされているデバイスでカスタム振動アラートを作成するように調整できます。
- `data`プロパティを使用して通知に追加データを添付できます。

サービスワーカーが異なるデバイスやブラウザで期待通りに動作することを確認するために、十分にテストすることを忘れないでください。また、`notificationclick`イベントリスナーの`'https://your-website.com'`リンクをアプリケーションの適切なURLに更新してください。

### 6. ホーム画面への追加 {#6-adding-to-home-screen}

ステップ2で定義した`InstallPrompt`コンポーネントは、iOSデバイスにインストールするよう指示するメッセージを表示します。

モバイルホーム画面にアプリケーションをインストールできるようにするには、次の条件を満たしている必要があります：

1. 有効なwebアプリマニフェスト (ステップ1で作成)
2. HTTPSで配信されたウェブサイト

モダンブラウザは、これらの条件が満たされると、自動的にインストールプロンプトをユーザーに表示します。`beforeinstallprompt`を使ってカスタムインストールボタンを提供することができますが、これはすべてのブラウザやプラットフォームで動作するわけではない（Safari iOSでは動作しない）ため、おすすめしません。

### 7. ローカルでのテスト {#7-testing-locally}

ローカルで通知を確認できるようにするためには、次を確認してください：

- [ローカルでHTTPSを使用して実行している](/docs/app/api-reference/cli/next#using-https-during-development)
  - `next dev --experimental-https`を使用してテストする
- ブラウザ (Chrome, Safari, Firefox) が通知を有効にしている
  - ローカルでプロンプトされた際に、通知使用の許可を受け入れる
  - ブラウザ全体で通知が無効になっていないことを確認する
  - 通知が表示されない場合は、別のブラウザを使ってデバッグする

### 8. アプリケーションのセキュリティを確保する {#8-securing-your-application}

セキュリティは、特にPWAの場合、Webアプリケーションの重要な側面です。Next.jsでは、`next.config.js`ファイルを使用してセキュリティヘッダーを設定することができます。例えば：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

これらのオプションについて説明しましょう：

1. グローバルヘッダー（すべてのルートに適用）：
   1. `X-Content-Type-Options: nosniff`: MIME型スニッフィングを防ぎ、悪意のあるファイルアップロードのリスクを減少させます。
   2. `X-Frame-Options: DENY`: クリックジャッキング攻撃を防ぐために、サイトがiframeに埋め込まれるのを防ぎます。
   3. `Referrer-Policy: strict-origin-when-cross-origin`: セキュリティと機能性のバランスを取り、リクエストと一緒に含まれるリファラー情報を制御します。
2. サービスワーカー専用のヘッダー：
   1. `Content-Type: application/javascript; charset=utf-8`: サービスワーカーがJavaScriptとして正しく解釈されるように保証します。
   2. `Cache-Control: no-cache, no-store, must-revalidate`: サービスワーカーのキャッシュを防ぎ、ユーザーが常に最新バージョンを取得できるようにします。
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`: サービスワーカーのために厳しいコンテンツセキュリティポリシーを実施し、同じオリジンからのスクリプトのみを許可します。

Next.jsで[コンテンツセキュリティポリシーを定義する方法](/docs/app/building-your-application/configuring/content-security-policy)について詳しく学びましょう。

## 次のステップ {#next-steps}

1. **PWAの機能探索**: PWAsは、高度な機能を提供するためにさまざまなWeb APIを活用できます。バックグラウンド同期、定期的なバックグラウンド同期、またはFile System Access APIなどの機能を探索し、アプリケーションを強化することを検討してください。PWA機能についてのインスピレーションと最新情報を得るために、[What PWA Can Do Today](https://whatpwacando.today/)のようなリソースを参照できます。
2. **静的エクスポート**：アプリケーションがサーバーを実行する必要がなく、代わりにファイルの静的エクスポートを使用する必要がある場合は、Next.jsの設定を更新してこの変更を可能にすることができます。[Next.js Static Exportドキュメント](/docs/app/building-your-application/deploying/static-exports)で詳細を学びましょう。ただし、Server Actionsから外部API呼び出しに移行し、定義されたヘッダーをプロキシに移動する必要があります。
3. **オフラインサポート**：オフライン機能を提供するための一つの選択肢は、Next.jsでの[Serwist](https://github.com/serwist/serwist)です。Next.jsとSerwistを統合する方法については、彼らの[ドキュメント](https://github.com/serwist/serwist/tree/main/examples/next-basic)に例が掲載されています。**注**：このプラグインは現在webpack設定を必要としています。
4. **セキュリティの考慮事項**：サービスワーカーが適切に保護されていることを確認してください。これにはHTTPSの使用、プッシュメッセージの送信元の検証、適切なエラーハンドリングの実装が含まれます。
5. **ユーザー体験**：ユーザーのブラウザが特定のPWA機能をサポートしていない場合でも、アプリが正常に動作するように、漸進的な強化技術を導入することを検討してください。
