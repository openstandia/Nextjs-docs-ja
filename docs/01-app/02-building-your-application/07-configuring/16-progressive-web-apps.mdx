プログレッシブウェブアプリケーション (PWA) は、Webアプリケーションの到達範囲とアクセシビリティを活用しつつ、ネイティブモバイルアプリの機能とユーザーエクスペリエンスを提供します。Next.jsを使用すると、複数のコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスなアプリのような体験を提供するPWAを作成できます。

PWAを使用すると：

- アプリストアの承認を待つことなく、即時に更新をデプロイできる
- 単一のコードベースでクロスプラットフォームアプリケーションを作成できる
- ホーム画面へのインストールやプッシュ通知などのネイティブのような機能を提供できる

## Next.jsでPWAを作成する {#creating-a-pwa-with-next-js}

### 1. Web App Manifestの作成 {#1-creating-the-web-app-manifest}

Next.jsは、App Routerを使用して[Web App Manifest](/docs/app/api-reference/file-conventions/metadata/manifest)を作成するための組み込みサポートを提供します。静的または動的なマニフェストファイルを作成できます。

例として、`app/manifest.ts`や`app/manifest.json`ファイルを作成します：

このファイルには、名前やアイコン、ユーザーのデバイスにアイコンとして表示する方法が必要です。これにより、ユーザーはあなたのPWAをホームスクリーンにインストールでき、ネイティブアプリのような体験を提供します。

異なるアイコンセットを作成するために、[ファビコンジェネレータ](https://realfavicongenerator.net/)のようなツールを使用し、生成されたファイルを`public/`フォルダに配置できます。

### 2. Webプッシュ通知の実装 {#2-implementing-web-push-notifications}

Webプッシュ通知は、すべての最新ブラウザでサポートされており、以下を含みます：

- ホームスクリーンにインストールされたアプリケーション用のiOS 16.4+
- macOS 13以降のSafari 16
- Chromiumベースのブラウザ
- Firefox

これにより、PWAはネイティブアプリの有力な代替手段となります。特に、オフラインサポートなしでインストールプロンプトをトリガーできる点が特徴です。

Webプッシュ通知を使用すると、ユーザーがアプリを積極的に使用していないときでも再エンゲージできます。Next.jsアプリケーションでの実装方法は以下のとおりです：

まず、`app/page.tsx`にメインページコンポーネントを作成しましょう。より良く理解するために、小さなパーツに分けていきます。まず、いくつかのインポートと必要なユーティリティを追加します。参照するServer Actionsがまだ存在しないのは問題ありません：

```tsx switcher
'use client'

import { useState, useEffect } from 'react'
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

次に、プッシュ通知の購読、購読解除、および送信を管理するコンポーネントを追加します。

```tsx switcher
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>このブラウザではプッシュ通知がサポートされていません。</p>
  }

  return (
    <div>
      <h3>プッシュ通知</h3>
      {subscription ? (
        <>
          <p>プッシュ通知の購読をしています。</p>
          <button onClick={unsubscribeFromPush}>購読解除</button>
          <input
            type="text"
            placeholder="通知メッセージを入力"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>テスト通知を送信</button>
        </>
      ) : (
        <>
          <p>プッシュ通知の購読をしていません。</p>
          <button onClick={subscribeToPush}>購読</button>
        </>
      )}
    </div>
  )
}
```

`app/actions.ts`という新しいファイルを作成し、アクションを含めます。このファイルは、サブスクリプションの作成、削除、および通知の送信を行います。

本番環境では、サーバーの再起動時に永続性を確保するためのサブスクリプションのデータベース保存が必要で、複数のユーザーのサブスクリプションを管理するためにも必要です。

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push APIを使用するには、[VAPID](https://vapidkeys.com/)キーを生成する必要があります。最も簡単な方法は、web-push CLIを直接使用することです：

まず、web-pushをグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

VAPIDキーを生成するには次を実行します：

```bash title="Terminal"
web-push generate-vapid-keys
```

生成されたキーをコピーし、`.env`ファイルに貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. Service Workerの作成 {#5-creating-a-service-worker}

`public/sw.js`にservice workerファイルを作成します：

```js title="public/sw.js"
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

self.addEventListener('notificationclick', function (event) {
  console.log('通知クリックを受け取りました。')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このservice workerは、カスタム画像と通知をサポートします。受信したプッシュイベントと通知クリックを処理します。

- 通知のカスタムアイコンは、`icon`および`badge`プロパティを使用して設定できます。
- サポートされているデバイスでカスタムバイブレーションアラートを作成するために`vibrate`パターンを調整できます。
- 通知に追加のデータを追加できます。

service workerが異なるデバイスやブラウザで期待通りに動作することを確認するために、十分にテストしてください。また、`notificationclick`イベントリスナー内の'`https://your-website.com'`リンクを、アプリケーションに適したURLに更新してください。

### 6. ホームスクリーンに追加 {#6-adding-to-home-screen}

ステップ2で定義した`InstallPrompt`コンポーネントは、iOSデバイス向けのホームスクリーンに追加するためのメッセージを表示します。

モバイルホームスクリーンにインストールできるようにするために、以下を確保する必要があります：

1. 有効なWeb App Manifest（ステップ1で作成）
2. HTTPS経由で提供されるウェブサイト

モダンブラウザは、これらの条件が満たされたときに自動的にユーザーにインストールプロンプトを表示します。[`beforeinstallprompt`](https://developer.mozilla.org/en-US/docs/Web/API/Window/beforeinstallprompt_event)を使用してカスタムインストールボタンを提供することもできますが、これはクロスブラウザおよびプラットフォームではない（Safari iOSで機能しない）ため、推奨しません。

### 7. ローカルでのテスト {#7-testing-locally}

ローカルで通知を表示できるようにするために、以下を確認してください：

- [ローカルでHTTPSを使用して実行](/docs/app/api-reference/cli/next#using-https-during-development)
  - テストには`next dev --experimental-https`を使用
- ブラウザ（Chrome, Safari, Firefox）で通知が有効にされている
  - ローカルでプロンプトが表示された場合、通知の使用を許可する
  - ブラウザ全体で通知が全般的に無効になっていないことを確認
  - まだ通知が表示されない場合は、別のブラウザでデバッグを試みる

### 8. アプリケーションのセキュリティ {#8-securing-your-application}

セキュリティは、特にPWAのようなWebアプリケーションにとって重要な側面です。Next.jsでは`next.config.js`ファイルを使用してセキュリティヘッダーを構成できます。例えば：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

これらのオプションを以下に説明します：

1. グローバルヘッダー（すべてのルートに適用）：
   1. `X-Content-Type-Options: nosniff`: MIMEタイプのスニッフィングを防ぎ、悪意のあるファイルアップロードのリスクを減少させます。
   2. `X-Frame-Options: DENY`: あなたのサイトがiframeに埋め込まれることを防ぎ、クリックジャック攻撃から保護します。
   3. `Referrer-Policy: strict-origin-when-cross-origin`: リファラー情報がリクエストと共にどのように含まれるかを制御し、セキュリティと機能性のバランスを保ちます。
2. Service Worker専用ヘッダー：
   1. `Content-Type: application/javascript; charset=utf-8`: Service Workerが正しくJavaScriptとして解釈されることを保証します。
   2. `Cache-Control: no-cache, no-store, must-revalidate`: Service Workerのキャッシュを防ぎ、ユーザーが常に最新のバージョンを取得することを保証します。
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`: Service Workerの厳格なコンテンツセキュリティポリシーを実装し、同じオリジンからのスクリプトのみを許可します。

Next.jsでの[コンテンツセキュリティポリシーの定義](/docs/app/building-your-application/configuring/content-security-policy)についてさらに学びましょう。

## 次のステップ {#next-steps}

1. **PWAの機能の探求**: PWAは高度な機能を提供するためにさまざまなWeb APIを活用できます。バックグラウンド同期、定期バックグラウンド同期、ファイルシステムアクセスAPIなどの機能を探求し、アプリケーションを強化することを検討してください。PWAの機能に関するインスピレーションや最新情報を得るために、[What PWA Can Do Today](https://whatpwacando.today/)のようなリソースを参照できます。
2. **静的エクスポート**： アプリケーションがサーバーを実行せず、ファイルの静的エクスポートを使用する場合、Next.jsの設定を更新してこの変更を有効にすることができます。詳細は[Next.js Static Exportのドキュメント](/docs/app/building-your-application/deploying/static-exports)で学びましょう。ただし、Server Actionsから外部APIの呼び出しに移行し、定義されたヘッダーをプロキシに移動する必要があります。
3. **オフラインサポート**: オフライン機能を提供するには、Next.jsとともに[Serwist](https://github.com/serwist/serwist)を使用するオプションがあります。SerwistとNext.jsを統合する方法の例は、[ドキュメント](https://github.com/serwist/serwist/tree/main/examples/next-basic)で見つけることができます。 **注意**：このプラグインは現在webpackの設定が必要です。
4. **セキュリティの考慮**: service workerが適切に保護されていることを確認してください。これには、HTTPSの使用、プッシュメッセージのソースの検証、および適切なエラーハンドリングの実装が含まれます。
5. **ユーザーエクスペリエンス**: 一部のPWA機能がユーザーのブラウザでサポートされていない場合でもアプリが正常に動作するよう、プログレッシブエンハンスメント技術を実装することを検討してください。
