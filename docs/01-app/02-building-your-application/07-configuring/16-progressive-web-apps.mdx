プログレッシブ Web アプリケーション (PWA) は、Webアプリケーションの広範なアクセス性と、ネイティブモバイルアプリの特徴やユーザー体験を組み合わせたものです。Next.jsを使えば、複数のコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスでアプリのような体験を提供するPWAを作成できます。

PWAでは以下が可能です：

- アプリストア承認を待たずに即時にアップデートをデプロイ
- 単一のコードベースでクロスプラットフォームのアプリケーションを作成
- ホームスクリーンへのインストールやプッシュ通知などのネイティブに似た機能を提供

## Next.jsでのPWAの作成 {#creating-a-pwa-with-next-js}

### 1. Web App マニフェストの作成 {#1-creating-the-web-app-manifest}

Next.js は App Router を使用した [Web App マニフェスト](/docs/app/api-reference/file-conventions/metadata/manifest) の作成をサポートしています。静的または動的なマニフェストファイルを作成できます：

例えば、`app/manifest.ts` または `app/manifest.json` ファイルを作成します：

このファイルには、名前やアイコン、ユーザーのデバイス上でどのように表示されるべきかについての情報が含まれている必要があります。これにより、ユーザーはPWAをホームスクリーンにインストールすることができ、ネイティブアプリのような体験を提供できます。

[favicon生成ツール](https://realfavicongenerator.net/) を利用して、異なるアイコンセットを作成し、生成されたファイルを `public/` フォルダに配置することができます。

### 2. Web プッシュ通知の実装 {#2-implementing-web-push-notifications}

Web Push Notificationは、すべてのモダンブラウザでサポートされています。以下を含みます：

- ホームスクリーンにインストールされたアプリケーション用のiOS 16.4以上
- macOS 13以降用のSafari 16
- Chromiumベースのブラウザ
- Firefox

これにより、PWAはネイティブアプリの代替として有効です。特に、オフラインサポートを必要とせずに、インストールプロンプトをトリガーすることができます。

Webプッシュ通知を使用することで、ユーザーがアプリをアクティブに使用していなくても再度エンゲージできます。次に、Next.jsアプリケーションでこれを実装する方法を示します：

まず、`app/page.tsx`にメインページコンポーネントを作成します。ここでは理解を深めるために、部分に分けて説明します。まず、必要なインポートとユーティリティを追加します。参照されているServer Actionsがまだ存在しないことは問題ありません：

```tsx switcher
'use client'

import { useState, useEffect } from 'react'
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

// Base64文字列をUint8Arrayに変換する関数
function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

では、プッシュ通知の購読、購読解除、送信を管理するコンポーネントを追加しましょう。

```tsx switcher
// プッシュ通知を管理するコンポーネント
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  // サービスワーカーを登録
  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  // プッシュ通知を購読
  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  // プッシュ通知の購読を解除
  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  // テスト通知を送信
  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>このブラウザではプッシュ通知はサポートされていません。</p>
  }

  return (
    <div>
      <h3>プッシュ通知</h3>
      {subscription ? (
        <>
          <p>プッシュ通知を購読しています。</p>
          <button onClick={unsubscribeFromPush}>購読解除</button>
          <input
            type="text"
            placeholder="通知メッセージを入力"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>テスト送信</button>
        </>
      ) : (
        <>
          <p>プッシュ通知を購読していません。</p>
          <button onClick={subscribeToPush}>購読</button>
        </>
      )}
    </div>
  )
}
```

`app/actions.ts`にアクションを含む新しいファイルを作成します。このファイルはサブスクリプションの作成、サブスクリプションの削除、および通知の送信を処理します。

'use server'

import webpush from 'web-push'

// VAPID の詳細を設定
webpush.setVapidDetails(
  '<mailto:your-email@example.com>',
  process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!,
  process.env.VAPID_PRIVATE_KEY!

  subscription = sub
  // 本番環境では、サブスクリプションをデータベースに保存する必要があります

  return { success: true }
}

// ユーザーの購読を解除
export async function unsubscribeUser() {
  subscription = null
  // 本番環境では、データベースからサブスクリプションを削除する必要があります
  // 例： await db.subscriptions.delete({ where: { ... } })
  return { success: true }
}

// 通知を送信
export async function sendNotification(message: string) {
  if (!subscription) {
    throw new Error('No subscription available')

  try {
    await webpush.sendNotification(
      subscription,
      JSON.stringify({
        title: 'テスト通知',
        body: message,
        icon: '/icon.png',
      })
    )
    return { success: true }
  } catch (error) {
    console.error('プッシュ通知の送信エラー：', error)
    return { success: false, error: '通知の送信に失敗しました' }
  }
}

通知の送信は、ステップ5で作成されたサービスワーカーによって処理されます。

本番環境では、サーバーの再起動や複数のユーザーのサブスクリプション管理のために、サブスクリプションをデータベースに保存することをお勧めします。

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push APIを使用するには、[VAPID](https://vapidkeys.com/) キーを生成する必要があります。最も簡単な方法は、web-push CLI を直接使用することです：

まず、web-pushをグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

次に、以下のコマンドを実行してVAPID キーを生成します：

```bash title="Terminal"
web-push generate-vapid-keys
```

出力をコピーして、`.env` ファイルにキーを貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. サービスワーカーの作成 {#5-creating-a-service-worker}

`public/sw.js` ファイルを作成し、サービスワーカーの記述を行います：

```js title="public/sw.js"
// プッシュイベントのリスナーを追加
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

// 通知クリックのリスナーを追加
self.addEventListener('notificationclick', function (event) {
  console.log('通知クリックを受信しました。')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このサービスワーカーは、カスタム画像と通知をサポートしています。受信したプッシュイベントと通知クリックを処理します。

- `icon` と `badge` プロパティを使用して、通知用のカスタムアイコンを設定できます。
- `vibrate` パターンを調整して、対応するデバイスでカスタムバイブレーションアラートを作成できます。
- `data` プロパティを使用して、通知に追加のデータを添付できます。

サービスワーカーが異なるデバイスやブラウザでも予想どおりに動作することを確認するために、十分にテストすることを忘れないでください。また、`'https://your-website.com'` リンクを、アプリケーションの適切なURLに更新してください。

### 6. ホームスクリーンに追加 {#6-adding-to-home-screen}

ステップ2で定義した`InstallPrompt` コンポーネントが、iOS デバイスにホームスクリーンにインストールするよう指示するメッセージを表示します。

モバイルホームスクリーンにアプリケーションをインストールできるようにするには、以下が必要です：

1. 有効なWeb App マニフェスト（ステップ1で作成）
2. HTTPS で配信されるウェブサイト

最新のブラウザは、これらの基準が満たされると、自動的にインストールプロンプトをユーザーに表示します。[`beforeinstallprompt`](https://developer.mozilla.org/en-US/docs/Web/API/Window/beforeinstallprompt_event) を使用してカスタムインストールボタンを提供できますが、Safari iOS では機能しないため、クロスブラウザおよびプラットフォーム間での互換性がないため推奨しません。

### 7. ローカルでのテスト {#7-testing-locally}

通知をローカルで確認するためには、以下を確認してください：

- [HTTPSでローカル実行](/docs/app/api-reference/cli/next#using-https-during-development)
  - テストには `next dev --experimental-https` を使用
- ブラウザ（Chrome、Safari、Firefox）で通知が有効になっていること
  - ローカルでプロンプトが表示された場合、通知を使用する権限を承認
  - ブラウザ全体で通知が無効化されていないことを確認
  - 通知が表示されない場合は、別のブラウザでデバッグを試みる

### 8. アプリケーションのセキュリティ {#8-securing-your-application}

セキュリティは、特にPWAにとって重要な側面です。Next.jsは、`next.config.js` ファイルを使用してセキュリティヘッダを設定できます。例：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

これらのオプションについて説明します：

1. グローバルヘッダ（すべてのルートに適用）：
   1. `X-Content-Type-Options: nosniff`：MIME タイプのスニッフィングを防止し、悪意のあるファイルのアップロードのリスクを減少させます。
   2. `X-Frame-Options: DENY`：クリックジャッキング攻撃から保護するため、サイトが iframe に埋め込まれることを防ぎます。
   3. `Referrer-Policy: strict-origin-when-cross-origin`：要求に含まれるリファラー情報の量を制御し、セキュリティと機能性のバランスを保ちます。
2. サービスワーカー専用ヘッダ：
   1. `Content-Type: application/javascript; charset=utf-8`：サービスワーカーが正しいスクリプトとして解釈されることを保証します。
   2. `Cache-Control: no-cache, no-store, must-revalidate`：サービスワーカーのキャッシュを防ぎ、常に最新バージョンが取得されるようにします。
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`：サービスワーカーのために、同一オリジンのスクリプトのみを許可する厳しいコンテンツセキュリティポリシーを実装します。

Next.jsを使用した [Content Security Policies](/docs/app/building-your-application/configuring/content-security-policy) の定義についての詳細を学びましょう。

## 次のステップ {#next-steps}

1. **PWAの機能の探索**：PWAはさまざまなWeb APIを活用して高度な機能を提供できます。バックグラウンド同期、定期的なバックグラウンド同期、またはFile System Access API などの機能を検討してアプリケーションを強化してください。最新のPWA の機能についてのインスピレーションや情報は、[What PWA Can Do Today](https://whatpwacando.today/) などのリソースを参照してください。
2. **静的エクスポート**：サーバーを実行する必要がなく、代わりにファイルの静的エクスポートを使用する場合、Next.js の設定を更新してこの変更を有効にすることができます。[Next.js Static Export documentation](/docs/app/building-your-application/deploying/static-exports)でさらに詳しく学んでください。ただし、Server Actionsから外部API 呼び出しに移行し、定義済みのヘッダをプロキシに移動する必要があります。
3. **オフラインサポート**：オフライン機能を提供するには、Next.jsと一緒に[Serwist](https://github.com/serwist/serwist)を使用するオプションがあります。Next.jsとの統合方法についての例は、[ドキュメンテーション](https://github.com/serwist/serwist/tree/main/examples/next-basic)で確認できます。**注意**：このプラグインは現在webpackの設定が必要です。
4. **セキュリティの考慮事項**：サービスワーカーが適切に保護されていることを確認してください。これには、HTTPSの使用、プッシュメッセージのソースの検証、および適切なエラーハンドリングの実装が含まれます。
5. **ユーザー体験**：特定のPWA機能がユーザーのブラウザでサポートされていない場合でも、アプリが正常に作動するようにするためにプログレッシブエンハンスメント技術の実装を検討してください。
