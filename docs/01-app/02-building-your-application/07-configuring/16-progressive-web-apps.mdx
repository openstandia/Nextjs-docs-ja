Progressive Web Applications (PWAs) は、ウェブアプリケーションのリーチとアクセシビリティをネイティブモバイルアプリの機能とユーザーエクスペリエンスと組み合わせたものを提供します。Next.js を使用すると、複数のコードベースやアプリストアの承認を必要とせずに、すべてのプラットフォームでシームレスでアプリのような体験を提供する PWA を作成できます。

PWAs を利用することで以下が可能です：

- アプリストアの承認を待たずに瞬時に更新をデプロイできる
- 単一のコードベースでクロスプラットフォームアプリケーションを作成できる
- ホーム画面へのインストールやプッシュ通知などのネイティブライクな機能を提供する

## Next.jsでPWAを作成する {#creating-a-pwa-with-next-js}

### 1. Webアプリマニフェストの作成 {#1-creating-the-web-app-manifest}

Next.jsは、App Routerを使用した[Webアプリマニフェスト](/docs/app/api-reference/file-conventions/metadata/manifest)の作成をサポートしています。静的または動的なマニフェストファイルのいずれかを作成できます：

例えば、`app/manifest.ts` または `app/manifest.json` ファイルを作成します：

このファイルには、名称、アイコン、およびユーザーのデバイス上でアイコンとしてどのように表示されるかに関する情報を含めるべきです。これにより、ユーザーはホーム画面にPWAをインストールし、ネイティブアプリのような体験を提供できます。

[favicon generators](https://realfavicongenerator.net/)のようなツールを使用して、異なるアイコンセットを作成し、生成されたファイルを`public/` フォルダに配置できます。

### 2. Webプッシュ通知の実装 {#2-implementing-web-push-notifications}

Webプッシュ通知は、すべての最新ブラウザでサポートされています。次のブラウザを含みます：

- ホーム画面にインストールされたiOS 16.4以降のアプリケーション
- macOS 13以降のSafari 16
- Chromiumベースのブラウザ
- Firefox

これにより、PWAはネイティブアプリの代替として実行可能になります。特に、オフラインサポートを必要とせずにインストールプロンプトをトリガーできます。

Webプッシュ通知は、ユーザーがアプリをアクティブに使用していないときでもユーザーを再エンゲージすることを可能にします。Next.jsアプリケーションでそれらを実装する方法は次のとおりです：

まず、`app/page.tsx` にメインページコンポーネントを作成しましょう。より理解しやすいように、小さい部分に分解します。最初に、必要なインポートとユーティリティを追加します。参照されているServer Actionsがまだ存在しないことは問題ありません：

```tsx switcher
'use client'

// ステートとエフェクトのためのインポート
import { useState, useEffect } from 'react'
// アクションをインポートします
import { subscribeUser, unsubscribeUser, sendNotification } from './actions'

// URLをBase64からUint8Arrayに変換するユーティリティ関数
function urlBase64ToUint8Array(base64String: string) {
  const padding = '='.repeat((4 - (base64String.length % 4)) % 4)
  const base64 = (base64String + padding)
    .replace(/\\-/g, '+')
    .replace(/_/g, '/')

  const rawData = window.atob(base64)
  const outputArray = new Uint8Array(rawData.length)

  for (let i = 0; i < rawData.length; ++i) {
    outputArray[i] = rawData.charCodeAt(i)
  }
  return outputArray
}
```

次に、サブスクリプションを管理し、プッシュ通知を送信するためのコンポーネントを追加します。

```tsx switcher
function PushNotificationManager() {
  const [isSupported, setIsSupported] = useState(false)
  const [subscription, setSubscription] = useState<PushSubscription | null>(
    null
  )
  const [message, setMessage] = useState('')

  // サービスワーカーの登録
  useEffect(() => {
    if ('serviceWorker' in navigator && 'PushManager' in window) {
      setIsSupported(true)
      registerServiceWorker()
    }
  }, [])

  async function registerServiceWorker() {
    const registration = await navigator.serviceWorker.register('/sw.js', {
      scope: '/',
      updateViaCache: 'none',
    })
    const sub = await registration.pushManager.getSubscription()
    setSubscription(sub)
  }

  // プッシュ通知に登録
  async function subscribeToPush() {
    const registration = await navigator.serviceWorker.ready
    const sub = await registration.pushManager.subscribe({
      userVisibleOnly: true,
      applicationServerKey: urlBase64ToUint8Array(
        process.env.NEXT_PUBLIC_VAPID_PUBLIC_KEY!
      ),
    })
    setSubscription(sub)
    await subscribeUser(sub)
  }

  // プッシュ通知の登録を解除
  async function unsubscribeFromPush() {
    await subscription?.unsubscribe()
    setSubscription(null)
    await unsubscribeUser()
  }

  // テスト通知の送信
  async function sendTestNotification() {
    if (subscription) {
      await sendNotification(message)
      setMessage('')
    }
  }

  if (!isSupported) {
    return <p>このブラウザではプッシュ通知がサポートされていません。</p>
  }

  return (
    <div>
      <h3>プッシュ通知</h3>
      {subscription ? (
        <>
          <p>プッシュ通知に登録されています。</p>
          <button onClick={unsubscribeFromPush}>登録解除</button>
          <input
            type="text"
            placeholder="通知メッセージを入力してください"
            value={message}
            onChange={(e) => setMessage(e.target.value)}
          />
          <button onClick={sendTestNotification}>テスト送信</button>
        </>
      ) : (
        <>
          <p>プッシュ通知に登録されていません。</p>
          <button onClick={subscribeToPush}>登録</button>
        </>
      )}
    </div>
  )
}
```

### 4. VAPIDキーの生成 {#4-generating-vapid-keys}

Web Push APIを使用するためには、[VAPID](https://vapidkeys.com/)キーを生成する必要があります。最も簡単な方法は、web-push CLIを直接使用することです：

まず、web-pushをグローバルにインストールします：

```bash title="Terminal"
npm install -g web-push
```

次に、VAPIDキーを生成します：

```bash title="Terminal"
web-push generate-vapid-keys
```

出力をコピーし、キーを`.env`ファイルに貼り付けます：

```.env
NEXT_PUBLIC_VAPID_PUBLIC_KEY=your_public_key_here
VAPID_PRIVATE_KEY=your_private_key_here
```

### 5. Service Workerの作成 {#5-creating-a-service-worker}

サービスワーカー用の`public/sw.js`ファイルを作成します：

```js title="public/sw.js"
self.addEventListener('push', function (event) {
  if (event.data) {
    const data = event.data.json()
    const options = {
      body: data.body,
      icon: data.icon || '/icon.png',
      badge: '/badge.png',
      vibrate: [100, 50, 100],
      data: {
        dateOfArrival: Date.now(),
        primaryKey: '2',
      },
    }
    event.waitUntil(self.registration.showNotification(data.title, options))
  }
})

self.addEventListener('notificationclick', function (event) {
  console.log('通知のクリックを受け取りました。')
  event.notification.close()
  event.waitUntil(clients.openWindow('<https://your-website.com>'))
})
```

このサービスワーカーはカスタム画像と通知をサポートします。受信したプッシュイベントと通知のクリックを処理します。

- 通知には`icon`と`badge`プロパティを使用してカスタムアイコンを設定できます。
- `vibrate`パターンを調整して、サポートされているデバイスでカスタム振動アラートを作成できます。
- 追加のデータを`data`プロパティを使用して通知に添付できます。

サービスワーカーがさまざまなデバイスやブラウザで期待どおりに動作することを確認するために、十分にテストすることを忘れないでください。また、`notificationclick`イベントリスナー内の`'https://your-website.com'`のリンクを、アプリケーションに適したURLに更新してください。

### 6. ホーム画面への追加 {#6-adding-to-home-screen}

ステップ2で定義した`InstallPrompt`コンポーネントは、iOSデバイスにホーム画面へのインストールを指示するメッセージを表示します。

アプリケーションをモバイルのホーム画面にインストールできるようにするには、以下が必要です：

1. 有効なウェブアプリマニフェスト（ステップ1で作成）
2. HTTPSを介して提供されるウェブサイト

これらの条件を満たすと、最新のブラウザは自動的にインストールプロンプトをユーザーに表示します。[`beforeinstallprompt`](https://developer.mozilla.org/en-US/docs/Web/API/Window/beforeinstallprompt_event)を使用してカスタムインストールボタンを提供できますが、これはクロスブラウザおよびプラットフォームではないため（Safari iOSでは機能しません）、推奨しません。

### 7. ローカルでのテスト {#7-testing-locally}

ローカルで通知を確認するためには、以下を確認してください：

- [HTTPSでローカルを実行している](/docs/app/api-reference/cli/next#using-https-during-development)
  - テストには`next dev --experimental-https`を使用
- あなたのブラウザ（Chrome、Safari、Firefox）は通知が有効にされています
  - ローカルでプロンプトが表示されたら、通知の使用を許可してください
  - ブラウザ全体で通知が無効になっていないことを確認してください
  - それでも通知が表示されない場合は、デバッグのために別のブラウザを使用してみてください

### 8. アプリケーションのセキュリティ強化 {#8-securing-your-application}

セキュリティは、特にPWAにとって、ウェブアプリケーションの重要な要素です。Next.jsでは、`next.config.js`ファイルを使用してセキュリティヘッダーを設定できます。例えば：

```js title="next.config.js"
module.exports = {
  async headers() {
    return [
      {
        source: '/(.*)',
        headers: [
          {
            key: 'X-Content-Type-Options',
            value: 'nosniff',
          },
          {
            key: 'X-Frame-Options',
            value: 'DENY',
          },
          {
            key: 'Referrer-Policy',
            value: 'strict-origin-when-cross-origin',
          },
        ],
      },
      {
        source: '/sw.js',
        headers: [
          {
            key: 'Content-Type',
            value: 'application/javascript; charset=utf-8',
          },
          {
            key: 'Cache-Control',
            value: 'no-cache, no-store, must-revalidate',
          },
          {
            key: 'Content-Security-Policy',
            value: "default-src 'self'; script-src 'self'",
          },
        ],
      },
    ]
  },
}
```

これらのオプションについて説明します：

1. グローバルヘッダー（すべてのルートに適用）：
   1. `X-Content-Type-Options: nosniff`: MIMEタイプのスニッフィングを防止し、悪意のあるファイルアップロードのリスクを低減します。
   2. `X-Frame-Options: DENY`: あなたのサイトがiframeに埋め込まれるのを防止し、クリックジャッキング攻撃を防ぎます。
   3. `Referrer-Policy: strict-origin-when-cross-origin`: リファラー情報の含有量を制御し、セキュリティと機能性をバランスさせます。
2. サービスワーカー固有のヘッダー：
   1. `Content-Type: application/javascript; charset=utf-8`: サービスワーカーがJavaScriptとして正しく解釈されることを保証します。
   2. `Cache-Control: no-cache, no-store, must-revalidate`: サービスワーカーのキャッシュを防止し、常に最新バージョンをユーザーに提供します。
   3. `Content-Security-Policy: default-src 'self'; script-src 'self'`: サービスワーカーに対する厳格なコンテンツセキュリティポリシーを実装し、同一オリジンからのみスクリプトを許可します。

Next.jsを使用した[Content Security Policiesの定義](/docs/app/building-your-application/configuring/content-security-policy)について詳しく学びましょう。

## 次のステップ {#next-steps}

1. **PWAの機能を探る**: PWAsは、さまざまなウェブAPIを活用して高度な機能を提供できます。背景同期、定期的な背景同期、またはファイルシステムアクセルAPIなどの機能を探求して、アプリケーションを強化することを考慮してください。インスピレーションやPWAの機能に関する最新情報については、[What PWA Can Do Today](https://whatpwacando.today/)のようなリソースを参照できます。
2. **静的エクスポート**: アプリケーションがサーバーを実行せず、代わりにファイルの静的エクスポートを使用する必要がある場合、Next.jsの設定を更新してこの変更を有効にすることができます。[Next.js Static Export documentation](/docs/app/building-your-application/deploying/static-exports)で詳しく学びましょう。ただし、Server Actionsから外部APIの呼び出しに切り替え、定義されたヘッダーをプロキシに移動する必要があります。
3. **オフラインサポート**: オフライン機能を提供する1つのオプションは、Next.jsと一緒に[Serwist](https://github.com/serwist/serwist)を使用することです。[documentation](https://github.com/serwist/serwist/tree/main/examples/next-basic)には、Next.jsとSerwistを統合する方法の例があります。 **注意:** このプラグインは現在、webpackの設定が必要です。
4. **セキュリティの考慮事項**: サービスワーカーが適切にセキュリティされていることを確認してください。これには、HTTPSの使用、プッシュメッセージの送信元の検証、および適切なエラーハンドリングの実装が含まれます。
5. **ユーザーエクスペリエンス**: PWAの機能がユーザーのブラウザでサポートされていない場合でも、アプリがうまく動作するように、漸進的な改善技術を実装することを考慮してください。
