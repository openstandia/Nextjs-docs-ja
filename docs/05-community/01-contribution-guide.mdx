Next.js ドキュメント貢献ガイドへようこそ！ここに来ていただきうれしく思います。

このページでは、Next.js ドキュメントの編集方法についての手順を提供します。私たちの目標は、コミュニティの誰もが自信を持って貢献し、ドキュメントを改善できるようにすることです。

## なぜ貢献するのか？ {#why-contribute}

オープンソースの作業は終わることがなく、ドキュメントも同様です。ドキュメントへの貢献は、初心者がオープンソースに参加する良い方法であり、経験豊富な開発者がより複雑なトピックを明確にしながら、その知識をコミュニティと共有する方法でもあります。

Next.js のドキュメントに貢献することにより、すべての開発者にとってより強力な学習リソースを構築することに貢献しています。誤字を見つけた、紛らわしいセクションを発見した、または特定のトピックが欠けていることに気がついた場合でも、あなたの貢献は歓迎され、感謝されます。

## どのように貢献するか {#how-to-contribute}

ドキュメントのコンテンツは [Next.js リポジトリ](https://github.com/vercel/next.js/tree/canary/docs) にあります。貢献するには、GitHub 上でファイルを直接編集するか、リポジトリをクローンしてローカルでファイルを編集できます。

### GitHub ワークフロー {#github-workflow}

GitHub に慣れていない場合は、リポジトリをフォークし、ブランチを作成し、プルリクエストを送信する方法を学ぶために [GitHub オープンソースガイド](https://opensource.guide/how-to-contribute/#opening-a-pull-request) を読むことをおすすめします。

> **Good to know**: ドキュメントのコードはプライベートのコードベースにあり、Next.js 公開リポジトリに同期されています。これは、ローカルでドキュメントをプレビューできないことを意味します。ただし、プルリクエストをマージした後、変更が [nextjs.org](https://nextjs.org/docs) に反映されます。

### MDX の書き方 {#writing-mdx}

ドキュメントは [MDX](https://mdxjs.com/) という JSX 構文をサポートするマークダウン形式で書かれています。これにより、ドキュメント内に React コンポーネントを埋め込むことができます。マークダウン構文の概要については、[GitHub マークダウンガイド](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax) をご覧ください。

### VSCode {#vscode}

#### ローカルでの変更プレビュー {#previewing-changes-locally}

VSCode には、ローカルで編集内容を確認できる組み込みのマークダウンプレビュー機能があります。MDX ファイルでプレビュー機能を有効にするには、ユーザー設定に設定オプションを追加する必要があります。

コマンドパレットを開き（Mac では `⌘ + ⇧ + P`、Windows では `Ctrl + Shift + P`）、`Preferences: Open User Settings (JSON)` を検索します。

次に、次の行を `settings.json` ファイルに追加します：

```json title="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

次に、再度コマンドパレットを開き、`Markdown: Preview File` または `Markdown: Open Preview to the Side` を検索します。これにより、フォーマットされた変更を表示するプレビューウィンドウが開きます。

#### 拡張機能 {#extensions}

VSCode ユーザーに次の拡張機能もおすすめします：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx): MDX 用のインテリセンスと構文ハイライト
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode): 保存時に MDX ファイルをフォーマット

### レビュープロセス {#review-process}

貢献を提出すると、Next.js または Developer Experience チームが変更をレビューし、フィードバックを提供し、準備が整い次第プルリクエストをマージします。

プルリクエストのコメントで質問がある場合や追加のサポートが必要な場合はお知らせください。Next.js ドキュメントに貢献し、コミュニティの一員であることに感謝します！

> **Tip:** プルリクエストを送信する前に、Prettier を実行するには `pnpm prettier-fix` を実行してください。

## ファイル構造 {#file-structure}

ドキュメントは**ファイルシステムルーティング**を使用しています。各フォルダと [`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 内のファイルはルートセグメントを表し、これらのセグメントは URL パス、ナビゲーション、パンくずリストを生成するために使用されます。

ファイル構造はサイト上で見ることができるナビゲーションを反映しており、デフォルトではナビゲーション項目はアルファベット順にソートされています。ただし、フォルダまたはファイル名の先頭に2桁の番号 (`00-`) を追加することで項目の順序を変更できます。

例えば、[functions API リファレンス](/docs/app/api-reference/functions) では、特定の関数を見つけやすくするためにページがアルファベット順にソートされています:

```txt
03-functions
├── cookies.mdx
├── draft-mode.mdx
├── fetch.mdx
└── ...
```

しかし、[ルーティングセクション](/docs/app/building-your-application/routing) では、開発者がこれらの概念を学ぶ順序でソートされるように、ファイルに2桁の番号が付けられています：

```txt
02-routing
├── 01-defining-routes.mdx
├── 02-pages-and-layouts.mdx
├── 03-linking-and-navigating.mdx
└── ...
```

ページをすばやく見つけるには、VSCode で `⌘ + P` (Mac) または `Ctrl + P` (Windows) を使用して検索バーを開きます。そして、探しているページのスラッグを入力します。例：`defining-routes`

> **なぜマニフェストを使用しないのか？**
>
> ドキュメントナビゲーションを生成するために（別の一般的な方法として）マニフェストファイルを使用することも検討しましたが、マニフェストはファイルの同期がすぐに取れなくなることが分かりました。ファイルシステムルーティングはドキュメントの構造について考えることを強制し、Next.js にとってよりネイティブな感じがします。

## メタデータ {#metadata}

各ページには、三つのダッシュで区切られたファイルのトップにメタデータブロックがあります。

### 必須フィールド {#required-fields}

次のフィールドは**必須**です：

| フィールド    | 説明                                                                  |
| ------------- | --------------------------------------------------------------------- |
| `title`       | ページの `<h1>` タイトル。SEO と OG 画像に使用します。                |
| `description` | ページの説明。SEO 用の `<meta name="description">` タグに使用します。 |

```yaml title="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

ページタイトルは 2~3 語に制限し (例：画像の最適化)、説明は 1~2 文 (例：Next.js で画像を最適化する方法を学びます) に制限するのが良い習慣です。

### オプションフィールド {#optional-fields}

次のフィールドは**オプション**です：

| フィールド  | 説明                                                                                                                                                             |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | ナビゲーション内でページのタイトルを上書きします。ページタイトルが長すぎて収まりきらない場合に有用です。指定されていない場合、`title` フィールドが使用されます。 |
| `source`    | 共通ページにコンテンツをプルします。[共通ページ](#shared-pages) を参照してください。                                                                             |
| `related`   | ドキュメントの下部にある関連ページのリストです。これらは自動的にカードに変換されます。[関連リンク](#related-links) を参照してください。                          |

```yaml title="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
---
```

## `App` と `Pages` ドキュメント {#app-and-pages-docs}

**App Router** と **Pages Router** の機能のほとんどが完全に異なるため、それぞれのドキュメントは別々のセクション (`02-app` と `03-pages`) に保持されています。しかし、これらの間で共有されるいくつかの機能もあります。

### 共通ページ {#shared-pages}

コンテンツの重複を避け、コンテンツが同期から外れるリスクを避けるために、`source` フィールドを使用して、あるページから別のページにコンテンツをプルします。例えば、`<Link>` コンポーネントは**App** と **Pages** のどちらでも主に同じ動作をします。コンテンツを重複させるのではなく、`app/.../link.mdx` から `pages/.../link.mdx` へコンテンツをプルすることができます：

```mdx title="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx title="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* DO NOT EDIT THIS PAGE. */}
{/* The content of this page is pulled from the source above. */}
```

これにより、一か所でコンテンツを編集し、両方のセクションに反映させることができます。

### 共有コンテンツ {#shared-content}

共通ページでは、**App Router** または **Pages Router** 特有のコンテンツが含まれることがあります。例えば、`<Link>` コンポーネントには **Pages** では利用可能だが **App** では利用できない `shallow` prop があります。

正しい Router でのみコンテンツを表示するために、`<AppOnly>` または `<PagesOnly>` コンポーネントでコンテンツブロックをラップできます：

```mdx title="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

This content will only be shown on the Pages docs.

</PagesOnly>

This content is shared between App and Pages.
```

例やコードブロックでは、これらのコンポーネントを使用することが多いでしょう。

## コードブロック {#code-blocks}

コードブロックには、コピーとペーストが可能な最小限の動作する例が含まれているべきです。これは、追加の設定なしでコードが実行できる必要があることを意味します。

例えば、`<Link>` コンポーネントの使用方法を示す場合、`import` 文と `<Link>` コンポーネント自体を含めるべきです。

```tsx title="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

コミットする前に、常にローカルで例を実行してください。これにより、コードが最新で動作していることを確認できます。

### 言語とファイル名 {#language-and-filename}

コードブロックには、言語と `filename` を含むヘッダーが必要です。`filename` プロップを追加して、ユーザーがどこでコマンドを入力するかを示す特別なターミナルアイコンを表示します。例えば：

````mdx title="code-example.mdx"
```bash title="Terminal"
npx create-next-app
```
````

ドキュメントのほとんどの例は `tsx` と `jsx` で書かれており、一部は `bash` で書かれています。ただし、サポートされている任意の言語を使用できます。こちらが[完全なリスト](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)です。

JavaScript のコードブロックを書く場合、以下の言語と拡張子の組み合わせを使用します。

|                                          | 言語   | 拡張子 |
| ---------------------------------------- | ------ | ------ |
| JSX コードを含む JavaScript ファイル     | ```jsx | .js    |
| JSX コードを含まない JavaScript ファイル | ```js  | .js    |
| JSX コードを含む TypeScript ファイル     | ```tsx | .tsx   |
| JSX コードを含まない TypeScript ファイル | ```ts  | .ts    |

### TS と JS の切り替え {#ts-and-js-switcher}

言語スイッチャーを追加して、TypeScript と JavaScript を切り替えます。コードブロックは TypeScript を優先し、JavaScript バージョンを用意してユーザーに対応します。

現在、TS と JS の例は順番に書かれており、`switcher` プロップでリンクされています：

````mdx title="code-example.mdx"
<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/page.tsx" switcher

```

</TabItem>
<TabItem value="jsx" label="JavaScript">

```jsx title="app/page.js" switcher

```

</TabItem>
</Tabs>
````

> **Good to know**: 今後、TypeScript のスニペットを自動的に JavaScript にコンパイルする予定です。それまでの間は、[transform.tools](https://transform.tools/typescript-to-javascript) を利用できます。

### 行のハイライト {#line-highlighting}

コード行をハイライトすることができます。これは、コードの特定の部分に注意を引きたい場合に便利です。`highlight` プロップに数値を渡すことで行をハイライトできます。

**単一の行:** `highlight={1}`

```tsx title="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**複数行:** `highlight={1,3}`

```tsx title="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**行範囲:** `highlight={1-5}`

```tsx title="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## ノート {#notes}

重要だが緊急ではない情報にはノートを使用します。ノートはユーザーをメインコンテンツからそらさずに情報を追加する良い方法です。

```mdx title="notes.mdx"
> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.
```

**出力:**

> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.

## 関連リンク {#related-links}

関連リンクは、ユーザーの学習の旅をガイドするために、論理的な次のステップへのリンクを追加します。

- リンクは、ページのメインコンテンツの下のカードに表示されます
- 子ページがあるページには、リンクが自動的に生成されます。例えば [Optimizing](/docs/app/building-your-application/optimizing) セクションには、すべての子ページへのリンクがあります

ページのメタデータ内の `related` フィールドを使用して関連リンクを作成します。

```yaml title="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### ネストされたフィールド {#nested-fields}

| フィールド    | 必須か？   | 説明                                                                                                                                                               |
| ------------- | ---------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| `title`       | オプション | カードリストのタイトル。デフォルトは **Next Steps** です。                                                                                                         |
| `description` | オプション | カードリストの説明です。                                                                                                                                           |
| `links`       | 必須       | 他のドキュメントページへのリンクリスト。各リストアイテムは相対 URL パス（先頭のスラッシュなし）である必要があります。例：`app/api-reference/file-conventions/page` |

## スタイルガイド {#style-guide}

このセクションには、技術文書を書くことに不慣れな人のためのドキュメント執筆ガイドラインが含まれています。

### ページテンプレート {#page-templates}

ページに厳密なテンプレートはありませんが、ドキュメント全体で繰り返し表示されるページセクションには次のものがあります：

- **概要:** ページの最初の段落で、ユーザーにその機能が何であるか、何に使用されるかを伝えるべきです。続いて、最小限の動作する例またはその API リファレンスを提供します。
- **規約:** 機能に規約がある場合は、ここで説明されるべきです。
- **例:** 機能を異なるユースケースでどのように使用できるかを示します。
- **API テーブル:** API ページには、可能であればジャンプ先リンク付きの概要テーブルをページの先頭に配置するべきです。
- **次のステップ (関連リンク):** 関連ページへのリンクを追加して、ユーザーの学習の旅をガイドします。

必要に応じてこれらのセクションを追加してください。

### ページタイプ {#page-types}

ドキュメントページは、概念とリファレンスの2つのカテゴリーに分かれています。

- **概念** ページは、概念や機能を説明するために使用されます。それらは通常、リファレンスページよりも長く、情報が多く含まれることが多いです。Next.js ドキュメントでは、概念ページは「アプリケーションの構築」セクションにあります。
- **リファレンス** ページは、特定の API を説明するために使用されます。それらは通常、短くて焦点を絞っています。Next.js ドキュメントでは、リファレンスページは「API リファレンス」セクションにあります。

> **Good to know**: 貢献するページによっては、異なる声とスタイルに従う必要がある場合があります。例えば、概念ページはより指導的であり、ユーザーを指すために _あなた_ という語を使用します。リファレンスページはより技術的であり、「作成、更新、受け入れ」などの命令形の言葉を使用し、_あなた_ という語を省略する傾向があります。

### 声のトーン {#voice}

ドキュメント全体で一貫したスタイルと声のトーンを維持するためのガイドラインをいくつか以下に示します：

- 明確で簡潔な文章を書く。逸脱を避ける
  - カンマを多用している場合、文章を複数の文に分けるかリストを使うことを検討する
  - 複雑な言葉をより簡単な言葉に置き換える。例えば、_use_ の代わりに _utilize_ など
- _this_ という言葉に気を付ける。それはあいまいで混乱を招く可能性があるため、文の主語が不明な場合はそれを繰り返すことを恐れない
  - 例えば、_Next.js uses React_ の代わりに _Next.js uses this_
- パッシブボイスではなくアクティブボイスを使う。アクティブな文は読みやすい
  - 例えば、_Next.js uses React_ の代わりに _React is used by Next.js_。_was_ や _by_ などの言葉を使用している場合、パッシブボイスを使用している可能性がある。
- _easy_、_quick_、_simple_、_just_ などの言葉を避ける。これは主観的であり、ユーザーを落胆させる可能性がある
- _don't_、_can't_、_won't_ などの否定的な言葉を避ける。これは読む側を落胆させる可能性がある
  - 例えば、_「ページ間のリンクを作成するには `Link` コンポーネントを使用できます」_ の代わりに _「ページ間のリンクを作成するには `<a>` タグを使用しないでください」_
- 二人称（you/your）で書く。これはより個人的で魅力的である
- ジェンダーニュートラルな言葉を使用する。オーディエンスを指すときは _developers_、_users_、_readers_ を使う
- コード例を追加する場合は、適切にフォーマットされていることと動作することを確認する
