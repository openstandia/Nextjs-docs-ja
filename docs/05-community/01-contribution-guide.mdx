Next.js ドキュメント貢献ガイドへようこそ！ここに来ていただいてとても嬉しいです。

このページでは、Next.js ドキュメントを編集する方法を説明します。私たちの目標は、コミュニティの皆さんがドキュメントに貢献し、改善する力を感じられるようにすることです。

## なぜ貢献するのか？

オープンソースの作業は終わることはなく、ドキュメントも同様です。ドキュメントに貢献することは、初心者がオープンソースに関わる良い方法であり、経験豊富な開発者がコミュニティと知識を共有しながらより複雑なトピックを明確にする手段となります。

Next.js ドキュメントに貢献することで、すべての開発者にとってより充実した学習リソースを構築する手助けをしていただくことになります。タイプミスや紛らわしいセクションを見つけたり、特定のトピックが欠けていることに気づいたりした場合でも、あなたの貢献は歓迎され、感謝されます。

## どのように貢献するか

ドキュメントのコンテンツは [Next.js リポジトリ](https://github.com/vercel/next.js/tree/canary/docs)にあります。貢献するには、GitHub で直接ファイルを編集するか、リポジトリをクローンしてローカルでファイルを編集することができます。

### GitHub ワークフロー

GitHub に不慣れな場合は、[GitHub オープンソースガイド](https://opensource.guide/how-to-contribute/#opening-a-pull-request)を読むことをお勧めします。これにより、リポジトリをフォークし、ブランチを作成し、プルリクエストを送信する方法を学ぶことができます。

> **Good to know**: 元のドキュメントコードはプライベートなコードベースにあり、Next.js のパブリックリポジトリに同期されています。このため、ドキュメントをローカルではプレビューできません。ただし、プルリクエストがマージされると [nextjs.org](https://nextjs.org/docs) に変更が反映されます。

### MDX の記述

ドキュメントは [MDX](https://mdxjs.com/) で記述されており、JSX 構文をサポートするマークダウン形式です。これにより、ドキュメント内に React コンポーネントを埋め込むことができます。マークダウン構文の概要については [GitHub Markdown ガイド](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)をご覧ください。

### VSCode

#### ローカルでの変更プレビュー

VSCode には組み込みのマークダウンプレビュー機能があり、ローカルで編集した内容を見ることができます。MDX ファイル用のプレビューを有効にするには、ユーザー設定に設定オプションを追加する必要があります。

コマンドパレット（Mac では `⌘ + ⇧ + P` 、Windows では `Ctrl + Shift + P`）を開き、`Preferences: Open User Settings (JSON)` を検索してください。

その後、次の行を `settings.json` ファイルに追加します：

```json filename="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

次に、再度コマンドパレットを開き、`Markdown: Preview File` または `Markdown: Open Preview to the Side` を検索します。これにより、フォーマットされた変更を確認できるプレビューウィンドウが開きます。

#### 拡張機能

VSCode 利用者には次の拡張機能をお勧めします：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx)：MDX のインテリセンスとシンタックスハイライト
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode)：保存時に MDX ファイルをフォーマット

### レビュー プロセス

一度貢献を提出すると、Next.js または Developer Experience チームが変更内容をレビューし、フィードバックを提供し、準備ができたらプルリクエストをマージします。

PR のコメントで質問がある場合やさらにサポートが必要な場合はお知らせください。Next.js ドキュメントに貢献し、コミュニティの一員になってくださりありがとうございます！

> **Tip:** `pnpm prettier-fix` を実行して、PR を提出する前に Prettier を実行してください。

## ファイル構造

ドキュメントは **file-system routing** を使用します。[`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 内の各フォルダとファイルは route segment を表します。これらのセグメントは URL パス、ナビゲーション、およびパンくずリストを生成するために使用されます。

ファイル構造はサイト上で表示されるナビゲーションを反映し、デフォルトではナビゲーション項目はアルファベット順にソートされます。ただし、フォルダまたはファイル名の先頭に2桁の数字（`00-`）を追加することで項目の順序を変更できます。

例えば、[functions API Reference](/docs/app/api-reference/functions) では、特定の関数をより簡単に見つけられるようにページがアルファベット順にソートされています：

```txt
03-functions
├── cookies.mdx
├── draft-mode.mdx
├── fetch.mdx
└── ...
```

しかし、[routing section](/docs/app/building-your-application/routing) では、ファイルが2桁の数字のプレフィックスで指定されており、これらの概念を学ぶ順序でソートされています：

```txt
02-routing
├── 01-defining-routes.mdx
├── 02-pages-and-layouts.mdx
├── 03-linking-and-navigating.mdx
└── ...
```

ページをすばやく見つけるには、`⌘ + P`（Mac）または `Ctrl + P`（Windows）を使用して VSCode で検索バーを開きます。そして、探しているページのスラッグを入力します。例：`defining-routes`

> **Why not use a manifest?**
>
> マニフェストファイル（ドキュメンテーションナビゲーションを生成するための他の一般的な方法）を使用することも考慮しましたが、ファイルとすぐに同期が取れなくなることがわかりました。file-system routing を使用すると、ドキュメントの構造を考える必要があり、Next.js にとってより自然に感じられます。

## メタデータ

各ページのファイルの最上部には3つのダッシュで区切られたメタデータブロックがあります。

### 必須フィールド

以下のフィールドは**必須**です：

| フィールド      | 説明                                                 |
| ------------- | --------------------------------------------------- |
| `title`       | ページの `<h1>` タイトル、SEO と OG イメージに使用されます                  |
| `description` | ページの説明、SEO 用の `<meta name="description">` タグで使用されます |

```yaml filename="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

ページのタイトルは2〜3単語（例：Optimizing Images）にし、説明は1〜2文（例：Learn how to optimize images in Next.js）に制限することをお勧めします。

### オプションフィールド

以下のフィールドは**オプション**です：

| フィールド       | 説明                                                                                                                                      |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | ナビゲーションでページのタイトルを上書きします。ページタイトルが長すぎて収まらない場合に便利です。指定されていない場合、`title` フィールドが使用されます。                    |
| `source`    | コンテンツを共有ページに取り込みます。詳細については、[Shared Pages](#shared-pages) を参照してください。                                                 |
| `related`   | ドキュメントの下部にある関連ページのリストです。これらは自動的にカードに変換されます。[Related Links](#related-links) を参照してください。 |

```yaml filename="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
---
```

## `App` と `Pages` のドキュメント

**App Router** と **Pages Router** の多くの機能は完全に異なるため、それぞれのドキュメントは別々のセクション（`02-app` と `03-pages`）に保持されています。ただし、両者で共有される機能もいくつかあります。

### 共有ページ

コンテンツの重複を避け、内容が同期されなくなるリスクを防ぐために、`source` フィールドを使用して1つのページから別のページにコンテンツをプルします。例えば、`<Link>` コンポーネントは **App** と **Pages** で _ほとんど_ 同じ動作をします。コンテンツを重複させる代わりに、`app/.../link.mdx` から `pages/.../link.mdx` にコンテンツを引き込むことができます：

```mdx filename="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx filename="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* DO NOT EDIT THIS PAGE. */}
{/* The content of this page is pulled from the source above. */}
```

このようにして、1か所で内容を編集し、両方のセクションに反映させることができます。

### 共有コンテンツ

共有ページでは、**App Router** または **Pages Router** 固有のコンテンツが含まれている場合があります。例えば `<Link>` コンポーネントには **Pages** でのみ利用可能な `shallow` prop があり、**App** にはありません。

コンテンツが正しいリouter でのみ表示されるようにするためには、`<AppOnly>` や `<PagesOnly>` コンポーネントでコンテンツブロックをラップします：

```mdx filename="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

This content will only be shown on the Pages docs.

</PagesOnly>

This content is shared between App and Pages.
```

これらのコンポーネントは、例やコードブロックに使用することが多いでしょう。

## コードブロック

コードブロックは、コピーペーストして使える最小限の動作例を含むべきです。つまり、コードは追加の設定なしで実行できるものでなければなりません。

例えば、`<Link>` コンポーネントの使い方を説明する場合は、`import` 文と`<Link>` コンポーネント自体を含めるべきです。

```tsx filename="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

コミットする前に例をローカルで必ず実行してください。これにより、コードが最新かつ動作していることを確認できます。

### 言語およびファイル名

コードブロックには、言語と `filename` を含むヘッダが必要です。`filename` プロパティを追加して、コマンドの入力場所をユーザーに示すための特別なターミナルアイコンをレンダリングします。例えば：

````mdx filename="code-example.mdx"
```bash filename="Terminal"
npx create-next-app
```
````

ドキュメントのほとんどの例は `tsx` と `jsx` で書かれていますが、一部 `bash` もあります。しかし、サポートされている言語は何でも使用できます。ここに[完全なリスト](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)があります。

JavaScript コードブロックを書く際には、次の言語と拡張子の組み合わせを使用します。

|                                | 言語   | 拡張子     |
| ------------------------------ | ------ | --------- |
| JSX コードを含む JavaScript ファイル | ```jsx   | .js      |
| JSX コードを含まない JavaScript ファイル | ```js    | .js      |
| JSX コードを含む TypeScript ファイル  | ```tsx   | .tsx     |
| JSX コードを含まない TypeScript ファイル | ```ts    | .ts      |

### TS と JS の切り替え

TypeScriptとJavaScriptを切り替える言語スイッチャーを追加します。コードブロックはまずTypeScriptであり、JavaScriptバージョンもユーザーに提供します。

現在、TSとJSの例を連続して記述し、`switcher` プロップでリンクしています：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx" switcher

```

```jsx filename="app/page.js" switcher

```
````

> **Good to know**: 将来的には、TypeScript のスニペットを自動的に JavaScript にコンパイルする予定です。それまでの間は、[transform.tools](https://transform.tools/typescript-to-javascript) を使用できます。

### ラインハイライト

コードラインをハイライトすることができます。これはコードの特定の部分に注意を引くのに役立ちます。`highlight` プロップに番号を渡すことでラインをハイライトできます。

**単一ライン:** `highlight={1}`

```tsx filename="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**複数ライン:** `highlight={1,3}`

```tsx filename="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**ラインレンジ:** `highlight={1-5}`

```tsx filename="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## Notes

重要ではあるがクリティカルではない情報には、Notes を使用します。Notes を使用することで、メインコンテンツの邪魔をせずに情報を追加する良い方法です。

```mdx filename="notes.mdx"
> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.
```

**出力:**

> **Good to know**: This is a single line note.

> **Good to know**:
>
> - We also use this format for multi-line notes.
> - There are sometimes multiple items worth knowing or keeping in mind.

## Related Links

関連リンクは、ユーザーの学習プロセスをガイドするためのリンクを追加することで、学習の旅を支援します。

- リンクは、ページのメインコンテンツの下にカードとして表示されます。
- 子ページを持つページには自動的にリンクが生成されます。例えば、[Optimizing](/docs/app/building-your-application/optimizing) セクションには、すべての子ページへのリンクがあります。

ページのメタデータで `related` フィールドを使って関連リンクを作成します。

```yaml filename="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### ネストされたフィールド

| フィールド       | 必須か   | 説明                                                                                                                                         |
| ------------- | ----- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | 任意    | カードリストのタイトル。デフォルトは **次のステップ** です。                                                                                              |
| `description` | 任意    | カードリストの説明です。                                                                                                                                |
| `links`       | 必須    | 他のドキュメントページへのリンクのリストです。それぞれのリスト項目は相対URLパスであるべきです（先頭にスラッシュは含めない）、例 `app/api-reference/file-conventions/page` |

## ダイアグラム

ダイアグラムは、複雑な概念を説明するための素晴らしい方法です。私たちは [Figma](https://www.figma.com/) を使用してダイアグラムを作成し、Vercel のデザインガイドを遵守しています。

現在、ダイアグラムは Next.js サイトのプライベート `/public` フォルダに存在します。ダイアグラムの更新や追加を希望する場合は、[GitHub issue](https://github.com/vercel/next.js/issues/new?assignees=&labels=template%3A+documentation&projects=&template=4.docs_request.yml&title=Docs%3A+) を開いてアイデアを送信してください。

## カスタムコンポーネントと HTML

ドキュメントで使用可能な React コンポーネントは次のとおりです：`<Image />` (next/image), `<PagesOnly />`, `<AppOnly />`, `<Cross />`, and `<Check />`. ドキュメントでは `<details>` タグ以外の生の HTML は許可されていません。

新しいコンポーネントのアイデアがある場合は、[GitHubのissue](https://github.com/vercel/next.js/issues/new/choose)を開いてください。

## スタイルガイド

このセクションは、技術文書作成に不慣れな人々のためのガイドラインを含んでいます。

### ページテンプレート

私たちはページのための厳格なテンプレートを持っていませんが、ドキュメント全体で繰り返し表示されるページセクションがあります：

- **概要：** ページの最初の段落では、ユーザーにその機能が何であり、それが何のために使用されるかを伝えるべきです。続いて、最低限の動作例またはその API 参照が示されます。
- **規約：** 機能に関する規約がある場合、ここで説明します。
- **例：** 機能が異なるユースケースでどのように使用できるかを示します。
- **API テーブル：** API ページは、ページの概要テーブルを表示するべきで、（可能な場合）セクションリンクに飛ぶことができます。
- **次のステップ（関連リンク）：** ユーザーの学習プロセスをガイドするために関連ページへのリンクを追加します。

これらのセクションを必要に応じて追加してください。

### ページタイプ

ドキュメントページは、概念と参照の2つのカテゴリに分かれています。

- **概念** ページは、概念や機能を説明するために使用されます。通常、参照ページよりも内容が豊富で長いです。Next.js のドキュメントでは、概念ページは **Building Your Application** セクションにあります。
- **参照** ページは、特定の API を説明するために使用されます。通常、よりフォーカスされ、短いです。Next.js のドキュメントでは、参照ページは **API Reference** セクションにあります。

> **Good to know**: あなたが貢献しているページによって、異なる声やスタイルに従う必要があるかもしれません。例えば、概念ページはより指示的で、ユーザーを「あなた」という言葉で指します。一方、参照ページはより技術的で、「作成する、更新する、受け入れる」などの命令形の言葉を使用し、「あなた」という言葉を省く傾向があります。

### 声

ドキュメント全体で一貫したスタイルと声を維持するためのガイドラインをいくつか紹介します：

- 明確で簡潔な文を書きます。脱線を避けましょう。
  - 多くのコンマを使用していることに気付いた場合、文章を複数の文に分けるかリストに変換することを考慮してください。
  - 複雑な単語は簡単な単語に置き換えましょう。例えば、「利用する」ではなく「使用する」といった具合です。
- 単語「これ」に気をつけましょう。曖昧で混乱を招くことがありますので、必要であれば文章の主題を繰り返すことを恐れないでください。
  - 例えば、「Next.js uses React」ではなく「Next.js uses this」です。
- 受動態ではなく能動態を使用しましょう。能動的な文は読みやすいです。
  - 例えば、「Next.js uses React」ではなく「React is used by Next.js」。「was」や「by」といった言葉を使用している場合、受動態を使用している可能性があります。
- 「簡単」「迅速」「シンプル」「単に」などの単語は避けましょう。これらは主観的であり、ユーザーを落胆させる可能性があります。
- 否定的な言葉「しない」「できない」「しません」などを避けましょう。これらは読者を落胆させる可能性があります。
  - 例えば、「ページ間のリンクを作成するには `Link` コンポーネントを使用する」とすべきであり、「`<a>` タグを使用してページ間のリンクを作成しないでください」としてはいけません。
- 二人称（あなた/あなたの）で書きます。これはより親しみやすく、関わりやすいです。
- ジェンダー中立な言葉を使用します。オーディエンスを指すときは「開発者」「ユーザー」「読者」を使用します。
- コード例を追加する場合は、適切にフォーマットされ、動作することを確認してください。

これらのガイドラインは網羅的ではありませんが、出発点として役立つはずです。技術文書作成についてさらに詳しく知りたい場合は、[Google のテクニカルライティングコース](https://developers.google.com/tech-writing/overview)をチェックしてください。
