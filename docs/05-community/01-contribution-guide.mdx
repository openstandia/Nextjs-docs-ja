Next.jsドキュメント貢献ガイドへようこそ！ここに来ていただきありがとうございます。

このページでは、Next.jsのドキュメントを編集するための手順を提供します。私たちの目標は、コミュニティの誰もが貢献し、ドキュメントを改善することに自信を持てるようにすることです。

## なぜ貢献するのか？

オープンソースの作業は終わりがなく、ドキュメンテーションも同様です。ドキュメントへの貢献は、初心者がオープンソースに関与するための良い方法であり、経験豊富な開発者がより複雑なトピックを明確にし、コミュニティと知識を共有するための方法です。

Next.jsのドキュメントに貢献することで、すべての開発者にとってより堅牢な学習リソースを構築するのを手助けしてくれます。誤字脱字を見つけたり、分かりにくいセクションがあったり、特定のトピックが欠けていることを認識したりした場合、あなたの貢献は歓迎され、感謝されます。

## どのように貢献するか

ドキュメントのコンテンツは[Next.jsリポジトリ](https://github.com/vercel/next.js/tree/canary/docs)で見つけることができます。貢献するには、ファイルをGitHub上で直接編集するか、リポジトリをクローンしてローカルでファイルを編集できます。

### GitHubのワークフロー

GitHubを初めて使用する場合は、 [GitHub Open Source Guide](https://opensource.guide/how-to-contribute/#opening-a-pull-request)を読んで、リポジトリをフォークする方法、ブランチを作成する方法、プルリクエストを送信する方法を学ぶことをお勧めします。

> **Good to know**: 基本的なドキュメントコードはNext.jsの公開リポジトリに同期される非公開のコードベースに存在します。これは、ローカルでドキュメントをプレビューできないことを意味します。ただし、プルリクエストがマージされた後に[nextjs.org](https://nextjs.org/docs)で変更を確認できます。

### MDXの書き方

ドキュメントはJSX構文をサポートするマークダウン形式の[MDX](https://mdxjs.com/)で書かれています。これにより、ドキュメントにReactコンポーネントを埋め込むことができます。マークダウン構文の概要については、[GitHub Markdownガイド](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)をご覧ください。

### VSCode

#### ローカルでの変更プレビュー

VSCodeにはビルトインのマークダウンプレビュー機能があります。これを使用して、ローカルでの編集を確認できます。MDXファイルに対してプレビューを有効にするには、ユーザー設定に設定オプションを追加する必要があります。

コマンドパレットを開きます（Macでは`⌘ + ⇧ + P`、Windowsでは`Ctrl + Shift + P`）; `Preferences: Open User Settings (JSON)`を検索します;

次に、`settings.json`ファイルに次の行を追加します:

```json filename="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

次にもう一度コマンドパレットを開き、`Markdown: Preview File`または`Markdown: Open Preview to the Side`を検索します。これにより、フォーマットされた変更を確認できるプレビューウィンドウが開きます。

#### 拡張機能

VSCodeユーザーには、次の拡張機能も推奨します:

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx): MDX用のインテリセンスと構文ハイライト
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode): ファイル保存時にMDXファイルを整形

### レビュープロセス

貢献を送信すると、Next.jsまたはDeveloper Experienceチームが変更をレビューし、フィードバックを提供し、準備が整ったらプルリクエストをマージします。

プルリクエストのコメントで質問がある場合や、さらに支援が必要な場合はお知らせください。Next.jsドキュメントに貢献し、コミュニティの一部であることに感謝します。

> **ヒント：** プレーン before submitting your PR;

## ファイル構造

ドキュメントは**ファイルシステムルーティング**を使用しています。[`/docs`](https://github.com/vercel/next.js/tree/canary/docs)内の各フォルダとファイルはルートセグメントを表しています。これらのセグメントは、URLパス、ナビゲーション、およびパンくずリストを生成するために使用されます。

ファイル構造はサイト上のナビゲーションに反映され、デフォルトではナビゲーションアイテムはアルファベット順にソートされます。ただし、フォルダまたはファイル名に二桁の数字（`00-`）をプレフィックスとして付けることでアイテムの順序を変更できます。

たとえば、[functions API Reference](/docs/app/api-reference/functions)では、開発者が特定の関数を見つけやすくするために、ページがアルファベット順にソートされています:

```txt
03-functions
├── cookies.mdx
├── draft-mode.mdx
├── fetch.mdx
└── ...
```

しかし、[ルーティングセクション](/docs/app/building-your-application/routing)では、ファイルが二桁の番号でプレフィックスされ、開発者がこれらの概念を学ぶべき順序でソートされています:

```txt
02-routing
├── 01-defining-routes.mdx
├── 02-pages-and-layouts.mdx
├── 03-linking-and-navigating.mdx
└── ...
```

ページを迅速に見つけるには、VSCodeで`⌘ + P`（Mac）または`Ctrl + P`（Windows）を使用して検索バーを開くことができます。次に、探しているページのスラグを入力します。例：`defining-routes`

> **なぜマニフェストを使用しないのか？**
> 
> ドキュメントのナビゲーションを生成するためのもう1つの一般的な方法であるマニフェストファイルの使用を検討しましたが、マニフェストはファイルとの同期がすぐに途切れると判断しました。ファイルシステムルーティングは、ドキュメントの構造について考えさせ、Next.jsにより自然に感じられます。

## メタデータ

各ページには、3つのダッシュで区切られたメタデータブロックがあります。

### 必須フィールド

次のフィールドは**必須**です：

| フィールド      | 説明                                                                |
| ------------- | -------------------------------------------------------------------- |
| `title`       | ページの`<h1>`タイトルで、SEOおよびOGイメージで使用します                                         |
| `description` | ページの説明で、SEOのために`<meta name="description">`タグで使用されます; |

```yaml filename="required-fields.mdx"
---
title: ページタイトル
description: ページ説明
---
```

ページタイトルは2-3語（例：画像の最適化）、説明は1-2文（例：Next.jsでの画像の最適化方法を学ぶ）に制限することをお勧めします。

### オプションフィールド

次のフィールドは**オプション**です：

| フィールド      | 説明                                                                                                                                    |
| ----------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | ナビゲーションのページタイトルを上書きします。ページのタイトルが長すぎる場合に便利です。提供されていない場合は`title`フィールドが使用されます。                         |
| `source`    | 共有ページにコンテンツを引き込みます。詳しくは[共有ページ](#shared-pages)をご覧ください。                                                  |
| `related`   | ドキュメントの下部に関連ページのリストを表示します。これらは自動的にカードに変換されます。詳しくは[関連リンク](#related-links)をご覧ください。                      |

```yaml filename="optional-fields.mdx"
---
nav_title: ナビゲーションアイテムタイトル
source: app/building-your-application/optimizing/images
related:
  description: 画像コンポーネントのAPIリファレンスを参照してください。
  links:
    - app/api-reference/components/image
---
```

## `App`と`Pages`のドキュメント

**App Router**と**Pages Router**の機能のほとんどは完全に異なるため、それぞれのドキュメントは別々のセクション（`02-app`および`03-pages`）に保管されています。ただし、両方に共通する機能もいくつかあります。

### 共有ページ

コンテンツを重複させず、一貫性を保つために、`source`フィールドを使用して、一方のページのコンテンツを他方に引き込みます。たとえば、`<Link>`コンポーネントは**App**と**Pages**でほとんど同じように動作します。コンテンツを重複させる代わりに、`app/.../link.mdx`から`pages/.../link.mdx`にコンテンツを引き込むことができます：

```mdx filename="app/.../link.mdx"
---
title: <Link>
description: <Link>コンポーネントのAPIリファレンス。
---

このAPIリファレンスは、Linkコンポーネントで使用できるプロップと構成オプションの使用方法を理解する手助けをします。
```

```mdx filename="pages/.../link.mdx"
---
title: <Link>
description: <Link>コンポーネントのAPIリファレンス。
source: app/api-reference/components/link
---

{/* このページは編集しないでください。 */}
{/* このページのコンテンツは上記のソースから引き込まれています。 */}
```

これにより、両方のセクションでコンテンツを一箇所で編集および反映できます。

### 共有コンテンツ

共有ページでは、**App Router**または**Pages Router**特有のコンテンツがあることがあります。たとえば、`<Link>`コンポーネントには**Pages**でのみ利用可能な`shallow`プロップがありますが、**App**では利用できません。

コンテンツが正しいルーターにのみ表示されるようにするために、コンテンツブロックを`<AppOnly>`または`<PagesOnly>`コンポーネントでラップすることができます：

```mdx filename="app/.../link.mdx"
このコンテンツはAppとPagesの間で共有されています。

<PagesOnly>

このコンテンツはPagesドキュメントにのみ表示されます。

</PagesOnly>

このコンテンツはAppとPagesの間で共有されています。
```

これらのコンポーネントは、例やコードブロックに使用される可能性が高いです。

## コードブロック

コードブロックには、コピー＆ペースト可能な最小の作業例を含める必要があります。つまり、コードは追加の設定なしで実行できる必要があります。

たとえば、`<Link>`コンポーネントの使用方法を示す場合、`import`文と`<Link>`コンポーネント自体が含まれている必要があります。

```tsx filename="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

例をコミットする前に必ずローカルで実行してください。これにより、コードが最新で機能していることを確認できます。

### 言語とファイル名

コードブロックには言語と`filename`を含むヘッダーを付ける必要があります。特別なターミナルアイコンを表示するために`filename`プロップを追加し、ユーザーがどこでコマンドを入力するかを示します。例：

````mdx filename="code-example.mdx"
```bash filename="Terminal"
npx create-next-app
```
````

ドキュメント内のほとんどの例は`tsx`と`jsx`で書かれており、一部は`bash`で書かれています。ただし、サポートされている言語を自由に使用できます。こちらが[完全な一覧](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)です。

JavaScriptコードブロックを書く場合、以下の言語と拡張子の組み合わせを使用します。

|                                | 言語       | 拡張子   |
| ------------------------------ | -------- | ------- |
| JSXコードを含むJavaScriptファイル   | ```jsx  | .js     |
| JSXを含まないJavaScriptファイル     | ```js   | .js     |
| JSXコードを含むTypeScriptファイル | ```tsx  | .tsx    |
| JSXを含まないTypeScriptファイル   | ```ts   | .ts     |

### TSとJSのスイッチャー

TypeScriptとJavaScriptを切り替える言語スイッチャーを追加します。コードブロックはまずTypeScriptで書き、JavaScriptバージョンも用意してユーザーに対応します。

現在、TSとJSの例は一つの後の他に書かれており、`switcher`プロップでリンクされています：

````mdx filename="code-example.mdx"
```tsx filename="app/page.tsx" switcher

```

```jsx filename="app/page.js" switcher

```
````

> **Good to know**: 将来的には、TypeScriptスニペットを自動的にJavaScriptにコンパイルする予定です。それまでの間は、 [transform.tools](https://transform.tools/typescript-to-javascript)を使用できます。

### 行のハイライト

コード行をハイライトすることができます。特定のコード部分に注意を引くために役立ちます。`highlight`プロップに番号を渡すことで行をハイライトできます。

**単一行：** `highlight={1}`

```tsx filename="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**複数行：** `highlight={1,3}`

```tsx filename="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**範囲行：** `highlight={1-5}`

```tsx filename="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## 注意

重要ですが重要ではない情報には注意を使用します。注意は、ユーザーが主なコンテンツから気をそらさないように情報を追加する良い方法です。

```mdx filename="notes.mdx"
> **Good to know**: これは単一行の注意です。

> **Good to know**:
>
> - マルチラインの注意にはこのフォーマットも使用します。
> - 知っておくべきアイテムが複数ある場合があります。
```

**出力：**

> **Good to know**: これは単一行の注意です。

> **Good to know**:
>
> - マルチラインの注意にはこのフォーマットも使用します。
> - 知っておくべきアイテムが複数ある場合があります。

## 関連リンク

関連リンクは、ユーザーの学習過程をガイドするために、論理的な次のステップへのリンクを追加します。

- リンクはページのメインコンテンツの下にあるカードに表示されます
- 子ページがあるページに対しては、リンクが自動的に生成されます。たとえば、[Optimizing](/docs/app/building-your-application/optimizing)セクションには、すべての子ページへのリンクがあります;

ページのメタデータ内の`related`フィールドを使用して関連リンクを作成します。

```yaml filename="example.mdx"
---
related:
  description: 最初のアプリケーションを迅速に開始する方法を学ぶ。
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### 入れ子になったフィールド

| フィールド        | 必須か？   | 説明                                                                                                                                                 |
| ------------- | ------ | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | 任意    | カードリストのタイトル。デフォルトは**次のステップ**。                                                                                                           |
| `description` | 任意    | カードリストの説明。                                                                                                                                      |
| `links`       | 必須    | 他のドキュメントページへのリンクのリスト。各リスト項目は相対URLパス（先頭のスラッシュ無し）である必要があります。例：`app/api-reference/file-conventions/page` |

## スタイルガイド

このセクションは、技術文書を書く初心者のためのガイドラインを含んでいます。

### ページテンプレート

ページには厳しいテンプレートはありませんが、ドキュメント全体で繰り返されるページセクションがあります：

- **概要：** ページの最初のパラグラフは、ユーザーにその機能が何であり、何に使われるかを伝えるべきです。最小限の作業例またはそのAPIリファレンスを後に続けます。
- **規約：** 機能に規約がある場合、それはここで説明されるべきです。
- **例：** 異なるユースケースで機能をどのように使用できるかを示します。
- **APIテーブル：** APIページには、ページの最初にセクションリンクジャンプの概要テーブルがあるべきです（できるだけ）。
- **次のステップ（関連リンク）：** ユーザーの学習過程をガイドするために関連ページへのリンクを追加します。

必要に応じて、これらのセクションを追加してください。

### ページタイプ

ドキュメントページは、概念と参照の2つのカテゴリに分かれています：

- **概念**ページは、概念や機能を説明するために使用されます。通常は長くて、参照ページよりも多くの情報を含んでいます。Next.jsのドキュメントでは、概念ページは**アプリケーションの構成**セクションで見つかります。
- **参照**ページは、特定のAPIを説明するために使用されます。通常は短く、より焦点が絞られています。Next.jsのドキュメントでは、参照ページは**APIリファレンス**セクションで見つかります。

> **Good to know**: 貢献しているページに応じて、異なる声とスタイルに従う必要があるかもしれません。例えば、概念ページはより指示的で、ユーザーに「あなた」という言葉を使っており、参照ページはより技術的で、「作成、更新、受け入れ」などの命令的な言葉を使い、「あなた」という言葉を省略する傾向があります。

### 声

ドキュメント全体で一貫したスタイルと声を維持するためのガイドラインをいくつか示します：

- 明確で簡潔な文章を書きましょう。脱線を避けましょう。
  - コンマを多く使用している場合は、文を複数の文に分けるかリストを使用することを検討してください。
  - 複雑な言葉をシンプルな言葉に置き換えましょう。例えば、「利用する」ではなく「使用する」。
- _this_という単語を慎重に使用してください。それは曖昧で誤解を招く可能性があります。文の主語が不明確な場合は、繰り返しを恐れずに書いてください。
  - 例えば、「Next.js uses React」ではなく「Next.jsはReactを使用します」。
- 受動態ではなく能動態を使用しましょう。能動態の文は読みやすいです。
  - 例えば、「Next.js uses React」ではなく「ReactはNext.jsによって使用されます」。_was_や_by_といった言葉を使用している場合は、受動態を使用しているかもしれません。
- _easy_、_quick_、_simple_、_just_などの言葉を使用しないでください。これは主観的であり、ユーザーにとっては不快感を与える可能性があります。
- _don't_、_can't_、_won't_などの否定的な言葉を避けてください。これは読者にとって不快に感じることがあります。
  - 例えば、「`Link`コンポーネントを使用してページ間のリンクを作成できます」ではなく「`<a>`タグを使用してページ間のリンクを作成しないでください」。
- 二人称（あなた/あなたの）で書いてください。これはより個人的で魅力的です。
- 性別に中立な言葉を使いましょう。読者を指すときは「開発者」、「ユーザー」、または「読者」を使用します。
- コード例を追加する場合は、適切にフォーマットされ、機能していることを確認してください。

これらのガイドラインは包括的ではありませんが、始めるのに役立つはずです。技術文書作成についてさらに詳しく知りたい場合は、[Google Technical Writing Course](https://developers.google.com/tech-writing/overview)をチェックしてください。
