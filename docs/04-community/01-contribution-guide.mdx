Next.js ドキュメントへの貢献ガイドへようこそ！あなたがここに来てくれてとても嬉しいです。

このページでは、Next.js ドキュメントを編集する方法を説明します。私たちの目標は、コミュニティのすべての人がドキュメントの改善と貢献に参加できるようになることです。

## なぜ貢献するのか？ {#why-contribute}

オープンソースの開発は終わることがありません。そしてドキュメントも同じです。ドキュメントへの貢献は、初心者がオープンソースに参加する良い方法であり、経験豊富な開発者がコミュニティと知識を共有しつつ、より複雑なトピックを明確にするための手段となります。

Next.js ドキュメントに貢献することで、すべての開発者のためにより堅牢な学習リソースを構築する手助けをしています。タイプミスを見つけたり、混乱するセクションがあったり、特定のトピックが欠けていることに気づいたりした場合でも、あなたの貢献は歓迎され、感謝されます。

## どのように貢献するか {#how-to-contribute}

ドキュメントのコンテンツは [Next.js リポジトリ](https://github.com/vercel/next.js/tree/canary/docs)にあります。貢献するには、ファイルを GitHub 上で直接編集するか、リポジトリをクローンしてローカルで編集します。

### GitHub ワークフロー {#github-workflow}

GitHub に不慣れな場合は、リポジトリのフォーク、ブランチの作成、プルリクエストの提出方法について学ぶために、[GitHub オープンソースガイド](https://opensource.guide/how-to-contribute/#opening-a-pull-request)を読むことをお勧めします。

> **Good to know**: 基になるドキュメントのコードは、Next.js の公開リポジトリに同期されるプライベートなコードベースで管理されています。ローカルでドキュメントをプレビューすることはできませんが、プルリクエストをマージした後に [nextjs.org](https://nextjs.org/docs) で変更を確認できます。

### MDX の記述 {#writing-mdx}

ドキュメントは、JSX 構文をサポートするマークダウン形式である [MDX](https://mdxjs.com/)で記述されています。これにより、ドキュメントに React コンポーネントを埋め込むことができます。マークダウン構文の概要については、[GitHub マークダウンガイド](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)を参照してください。

### VSCode {#vscode}

#### ローカルでの変更のプレビュー {#previewing-changes-locally}

VSCode には、ローカルで編集を確認するための組み込みのマークダウンプレビューワーがあります。MDX ファイル用のプレビューワーを有効にするには、ユーザー設定に設定オプションを追加する必要があります。

コマンドパレットを開き（Mac では`⌘ + ⇧ + P`、Windows では `Ctrl + Shift + P`）、`Preferences: Open User Settings (JSON)`を検索します。

次に、`settings.json`ファイルに次の行を追加します:

```json title="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

次に再度コマンドパレットを開き、`Markdown: Preview File`または`Markdown: Open Preview to the Side`を検索します。これにより、フォーマットされた変更を確認できるプレビューウィンドウが開かれます。

#### 拡張機能 {#extensions}

VSCode ユーザーには、次の拡張機能をお勧めします:

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx): MDX のインテリセンスと構文ハイライト。
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode): 保存時に MDX ファイルをフォーマットします。

### レビュー プロセス {#review-process}

提出した貢献は、Next.js または Developer Experience チームがレビューし、変更点をフィードバックし、準備が整ったらプルリクエストをマージします。

プルリクエストのコメントで質問がある場合や追加の支援が必要な場合はお知らせください。Next.js ドキュメントへの貢献とコミュニティの一員になってくれてありがとうございます！

> **Tip:** プルリクエストを提出する前に `pnpm prettier-fix` を実行して Prettier を実行してください。

## ファイル構造 {#file-structure}

ドキュメントは、**ファイルシステムルーティング**を使用しています。各フォルダと[`/docs`](https://github.com/vercel/next.js/tree/canary/docs) 内のファイルは、route segment を表します。これらのセグメントは、URL パス、ナビゲーション、およびパンくずリストの生成に使用されます。

ファイル構造は、サイトで表示されるナビゲーションを反映しており、デフォルトではナビゲーションアイテムはアルファベット順に並べられています。ただし、アイテムの順序を、フォルダまたはファイル名に二桁の数字（`00-`）を付け加えることで変更できます。

たとえば、[functions API Reference](/docs/app/api-reference/functions)では、開発者が特定の関数を簡単に見つけられるように、ページがアルファベット順に並べられています:

```txt
04-functions
├── after.mdx
├── cacheLife.mdx
├── cacheTag.mdx
└── ...
```

しかし、[routing セクション](/docs/app/building-your-application/routing)では、ファイルが二桁の数字でプレフィックスされており、開発者がこれらの概念を学ぶべき順序に並べられています:

```txt
01-routing
├── 01-defining-routes.mdx
├── 02-pages.mdx
├── 03-layouts-and-templates.mdx
└── ...
```

ページを迅速に見つけるには、VSCode で `⌘ + P`（Mac）または `Ctrl + P`（Windows）を使用し、検索バーを開きます。その後、探しているページのスラッグを入力してください。例： `defining-routes`

> **なぜマニフェストを使用しないのか？**
>
> ドキュメント ナビゲーションを生成するもう 1 つの一般的な方法であるマニフェスト ファイルを使用することを検討しましたが、マニフェストはファイルとすぐに同期が取れなくなることに気付きました。ファイルシステムルーティングでは、ドキュメントの構造を考慮する必要があり、Next.js にはよりネイティブに感じられます。

## メタデータ {#metadata}

各ページには、ファイルの先頭に三つのダッシュで区切られたメタデータブロックがあります。

### 必須フィールド {#required-fields}

次のフィールドが**必須**です:

| Field         | Description                                                           |
| ------------- | --------------------------------------------------------------------- |
| `title`       | ページの`<h1>`タイトルで、SEO と OG 画像に使用されます。              |
| `description` | `<meta name="description">` タグで SEO に使用されるページの説明です。 |

```yaml title="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

ページのタイトルは 2 ～ 3 語（例：イメージ最適化）、説明は 1 ～ 2 文（例：Next.js での画像の最適化方法を学ぶ）に抑えるのが良い習慣です。

### オプションフィールド {#optional-fields}

次のフィールドは**オプション**です:

| Field       | Description                                                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | ナビゲーションでページのタイトルを上書きします。ページのタイトルが長すぎて適さない場合に使用します。指定されていない場合、`title`フィールドが使用されます。 |
| `source`    | コンテンツを共有ページに引き込みます。[共有ページ](#shared-pages)を参照してください。                                                                       |
| `related`   | ドキュメントの下部にある関連ページのリストです。これらは自動的にカードに変換されます。[関連リンク](#related-links)を参照してください。                      |
| `version`   | 開発の段階を示します。例： `experimental`, `legacy`, `unstable`, `RC`                                                                                       |

```yaml title="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
version: experimental
---
```

## `App` と `Pages` ドキュメント {#app-and-pages-docs}

**App Router** と **Pages Router** におけるほとんどの機能は完全に異なるため、それぞれのドキュメントは別々のセクション（`02-app` および `03-pages`）に分かれています。ただし、両者で共有されている機能もいくつかあります。

### 共有ページ {#shared-pages}

コンテンツの重複と同期が取れなくなるリスクを避けるために、`source`フィールドを使用して、1つのページから別のページにコンテンツを引き込みます。たとえば、 `<Link>` コンポーネントは **App** と **Pages** の両方でほぼ同じ動作をします。コンテンツを重複させる代わりに、`app/.../link.mdx` から `pages/.../link.mdx` にコンテンツを引き込むことができます：

```mdx title="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx title="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* このページを編集しないでください。 */}
{/* このページのコンテンツは上記のソースから引き込まれています。 */}
```

これにより、コンテンツを一か所で編集し、両方のセクションに反映させることができます。

### 共有コンテンツ {#shared-content}

共有ページでは、**App Router** または **Pages Router** に特有のコンテンツが含まれることがあります。たとえば、 `<Link>` コンポーネントには **App** ではなく **Pages** でのみ使用可能な `shallow` prop があります。

コンテンツを正しい router のみに表示するために、 `<AppOnly>` または `<PagesOnly>` コンポーネントでコンテンツブロックをラップすることができます:

```mdx title="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

このコンテンツは Pages ドキュメントでのみ表示されます。

</PagesOnly>

This content is shared between App and Pages.
```

これらのコンポーネントは主に例やコードブロックでよく使用されます。

## コードブロック {#code-blocks}

コードブロックには、コピー＆ペーストできる最小限の動作例を含めるべきです。つまり、追加の設定なしで実行できるようにコードを作成する必要があります。

たとえば、 `<Link>` コンポーネントの使用方法を示す場合、`import` 文と`<Link>` コンポーネント自体を含める必要があります。

```tsx title="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

コミットする前に、必ずローカルで例を実行してください。これにより、コードが最新かつ動作していることを確認できます。

### 言語とファイル名 {#language-and-filename}

コードブロックには、言語と `filename` を含むヘッダーが必要です。`filename` prop を追加して、コマンドを入力する場所をユーザーに示す特別なターミナルアイコンを表示します。例：

````mdx title="code-example.mdx"
```bash title="Terminal"
npx create-next-app
```
````

ドキュメント内のほとんどの例は `tsx` と `jsx` で書かれており、一部は `bash` です。ただし、サポートされている任意の言語を使用できます。詳細は[こちらのリスト](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)をご覧ください。

JavaScript コードブロックを記述する際には、次の言語と拡張子の組み合わせを使用します。

|                                      | Language | Extension |
| ------------------------------------ | -------- | --------- |
| JSX コードを含む JavaScript ファイル | ```jsx   | .js       |
| JSX を含まない JavaScript ファイル   | ```js    | .js       |
| JSX を含む TypeScript ファイル       | ```tsx   | .tsx      |
| JSX を含まない TypeScript ファイル   | ```ts    | .ts       |

### TS と JS のスイッチャー {#ts-and-js-switcher}

TypeScript と JavaScript を切り替えるための言語スイッチャーを追加します。コードブロックは最初に TypeScript が記述され、JavaScript バージョンが用意されている必要があります。

現在、TS と JS の例を連続して記述し、それらを `switcher` prop でリンクしています:

````mdx title="code-example.mdx"
<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/page.tsx" switcher

```

</TabItem>
<TabItem value="jsx" label="JavaScript">

```jsx title="app/page.js" switcher

```

</TabItem>
</Tabs>
````

> **Good to know**: 将来、TypeScript のスニペットを自動的に JavaScript にコンパイルする予定です。現在は [transform.tools](https://transform.tools/typescript-to-javascript)を利用できます。

### ラインハイライト {#line-highlighting}

コード行をハイライトできます。これは、コードの特定の部分に注意を引きたい場合に便利です。`highlight` prop に番号を渡すことで行をハイライトできます。

**単一行:** `highlight={1}`

```tsx title="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**複数行:** `highlight={1,3}`

```tsx title="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**行の範囲:** `highlight={1-5}`

```tsx title="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## 関連リンク {#related-links}

関連リンクは、論理的な次のステップへのリンクを追加することで、ユーザーの学習の旅を案内します。

- リンクはページのメインコンテンツの下にカード形式で表示されます。
- 子ページがあるページは、自動的にリンクが生成されます。たとえば、[Optimizing](/docs/app/building-your-application/optimizing)セクションには、そのすべての子ページへのリンクがあります。

ページのメタデータの `related` フィールドを使用して関連リンクを作成します。

```yaml title="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### ネストされたフィールド {#nested-fields}

| Field         | Required? | Description                                                                                                                                                      |
| ------------- | --------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | Optional  | カードリストのタイトルです。デフォルトでは **Next Steps** に設定されています。                                                                                   |
| `description` | Optional  | カードリストの説明です。                                                                                                                                         |
| `links`       | Required  | 他のドキュメントページへのリンクのリストです。各リストアイテムは相対 URL パス（先行スラッシュなし）であるべきです 例： `app/api-reference/file-conventions/page` |

## スタイルガイド {#style-guide}

このセクションは、技術文書の初心者向けに文書を執筆するためのガイドラインを含んでいます。

### ページのテンプレート {#page-templates}

ページには厳密なテンプレートはありませんが、ドキュメント全体で繰り返し見られるページ セクションには次があります:

- **概要:** ページの最初の段落では、ユーザーに機能の内容と使用目的を説明する必要があります。次に、最小限の動作例または API リファレンスが続きます。
- **規約:** 機能に規約がある場合は、ここで説明してください。
- **例**: この機能がさまざまなユースケースでどのように使用できるかを示します。
- **API テーブル**: API ページには、可能であればページの概要テーブルとセクションごとのリンクを追加します。
- **次のステップ（関連リンク）**: 新しい情報の学習の旅を案内するために、関連するページへのリンクを追加します。

必要に応じてこれらのセクションを自由に追加してください。

### ページの種類 {#page-types}

ドキュメントページは、概念ページとリファレンスページの 2 つのカテゴリーに分かれています。

- **概念ページ** は、概念や機能を説明するために使用されます。通常、リファレンスページよりも長く、情報が多く含まれています。Next.js ドキュメントでは、概念ページは**アプリケーションの構築**セクションにあります。
- **リファレンスページ**は、特定の API を説明するために使用されます。通常、短く、より集中的です。Next.js ドキュメントでは、リファレンスページは**API リファレンス**セクションで見つけることができます。

> **Good to know**: あなたが貢献しているページによっては、異なる声やスタイルに従う必要があるかもしれません。たとえば、概念ページはより instructional であり、ユーザーに対して「_you_」という言葉を使用します。リファレンスページはより技術的であり、省略形の言葉を使う傾向があります。

### 声のトーン {#voice}

ドキュメント全体で一貫したスタイルと声のトーンを維持するためのガイドラインは以下の通りです:

- 明確で簡潔な文を書きます。脱線を避ける。
  - コロンをたくさん使っていると感じたら、文章を複数の文に分けるか、リストを使うことを検討します。
  - 複雑な単語をより簡単な単語に置き換えましょう。たとえば、_use_ の代わりに _utilize_ を使用します。
- *this*という言葉を慎重に使ってください。それは曖昧で、混乱を招く可能性があります。不明確な場合は、文の主題を繰り返すことを恐れないでください。
  - たとえば、_Next.js uses React_ の代わりに _Next.js uses this_ とします。
- 受動態の代わりに能動態を使用してください。能動態の方が読みやすい文です。
  - たとえば、_Next.js uses React_ の代わりに _React is used by Next.js_ とします。_was_ や _by_ という単語を使用している場合、受動態を使用している可能性があります。
- _easy_, _quick_, _simple_, _just_ などの単語を避けてください。これらは主観的であり、ユーザーにとっては不安を感じさせる可能性があります。
- _don't_, _can't_, _won't_ などの否定的な言葉を避けてください。これも読者にとっては不安を感じさせる可能性があります。
  - たとえば、_「You can use the `Link` component to create links between pages」_ の代わりに _「Don't use the `<a>` tag to create links between pages」_ という表現を使用します。
- 二人称（you/your）で書きます。これにより、よりパーソナルで魅力的なものになります。
- 性別に依存しない言語を使用します。聴衆を指すときには _developers_, _users_, _readers_ を使用します。
- コード例を追加する場合、それらが適切にフォーマットされ、機能しているか確認してください。

これらのガイドラインは包括的ではありませんが、始めるための手助けとなるはずです。技術執筆についてさらに詳しく知りたい場合は、 [Google Technical Writing Course](https://developers.google.com/tech-writing/overview)をご覧ください。
