Next.jsドキュメント寄稿ガイドへようこそ！あなたがここにいてくれることを嬉しく思います。

このページでは、Next.jsドキュメントを編集する方法を説明します。私たちの目標は、コミュニティの誰もが寄稿し、ドキュメントを改善する力を持てるようにすることです。

## なぜ寄稿するのか？ {#why-contribute}

オープンソース作業には終わりがありませんし、ドキュメンテーションにも終わりはありません。ドキュメントへの寄稿は、初心者がオープンソースに参加する良い方法であり、経験豊富な開発者が自分の知識をコミュニティと共有しながら、より複雑なトピックを明確にする方法でもあります。

Next.jsのドキュメントに寄稿することで、すべての開発者にとってより堅牢な学習資源を構築する手助けをしています。タイプミスや混乱を招くセクションを見つけた場合や、特定のトピックが欠けていることに気づいた場合、あなたの寄稿は歓迎され、感謝されます。

## どのように寄稿するのか {#how-to-contribute}

ドキュメントの内容は [Next.jsリポジトリ](https://github.com/vercel/next.js/tree/canary/docs) にあります。寄稿するには、GitHub上でファイルを直接編集するか、リポジトリをクローンしてローカルでファイルを編集できます。

### GitHubワークフロー {#github-workflow}

GitHubに不慣れな場合、リポジトリをフォークし、ブランチを作成し、プルリクエストを提出する方法を学ぶために[GitHubオープンソースガイド](https://opensource.guide/how-to-contribute/#opening-a-pull-request)を読むことをお勧めします。

> **Good to know**: 基礎となるドキュメントのコードは、Next.jsの公開リポジトリと同期されている非公開のコードベースにあります。つまり、ドキュメントをローカルでプレビューすることはできません。しかし、プルリクエストをマージした後、[nextjs.org](https://nextjs.org/docs) で変更を確認できます。

### MDXを書く {#writing-mdx}

ドキュメントは、JSX構文をサポートするマークダウン形式である[MDX](https://mdxjs.com/)で書かれています。これにより、ドキュメントにReactコンポーネントを埋め込むことができます。マークダウン構文のクイック概要については[GitHub Markdownガイド](https://docs.github.com/en/get-started/writing-on-github/getting-started-with-writing-and-formatting-on-github/basic-writing-and-formatting-syntax)を参照してください。

### VSCode {#vscode}

#### ローカルでの変更プレビュー {#previewing-changes-locally}

VSCodeには、ローカルで編集を確認するための組み込みのマークダウンプレビュー機能があります。MDXファイル用のプレビュー機能を有効にするには、ユーザー設定に設定オプションを追加する必要があります。

コマンドパレットを開き（Macでは `⌘ + ⇧ + P`、Windowsでは `Ctrl + Shift + P`）、`Preferences: Open User Settings (JSON)` を検索します。

次に、`settings.json`ファイルに次の行を追加します：

```json title="settings.json"
{
  "files.associations": {
    "*.mdx": "markdown"
  }
}
```

次に、コマンドパレットを再び開き、`Markdown: Preview File` または `Markdown: Open Preview to the Side` を検索します。これにより、フォーマットされた変更を確認できるプレビューウィンドウが開きます。

#### 拡張機能 {#extensions}

VSCodeユーザーには次の拡張機能をお勧めします：

- [MDX](https://marketplace.visualstudio.com/items?itemName=unifiedjs.vscode-mdx): MDXのIntellisenseと構文ハイライト
- [Prettier](https://marketplace.visualstudio.com/items?itemName=esbenp.prettier-vscode): 保存時にMDXファイルをフォーマット

### レビュープロセス {#review-process}

一度寄稿を送信すると、Next.jsまたはDeveloper Experienceチームが変更をレビューし、フィードバックを提供し、準備が整ったらプルリクエストをマージします。

プルリクエストのコメントで質問がある場合や、さらなるサポートが必要な場合はお知らせください。Next.jsドキュメントに寄稿し、コミュニティの一員であることに感謝します！

> **Tip:** `pnpm prettier-fix`を実行して、プルリクエストを送信する前にPrettierを実行してください。

## ファイル構造 {#file-structure}

ドキュメントは**ファイルシステムルーティング**を使用しています。`/docs` 内の各フォルダーとファイルがroute segumentを表しています。これらのセグメントは、URLパスの生成、ナビゲーション、およびパンくずリストに使用されます。

ファイル構造は、サイト上で見ることができるナビゲーションを反映しており、デフォルトでは、ナビゲーション項目はアルファベット順に並べ替えられます。ただし、フォルダーやファイル名に2桁の数字 (`00-`) を前もって付けることで、項目の順序を変更できます。

たとえば、[関数APIリファレンス](/docs/app/api-reference/functions) では、開発者が特定の関数を見つけやすくなるため、ページがアルファベット順に並べ替えられています：

```txt
04-functions
├── after.mdx
├── cacheLife.mdx
├── cacheTag.mdx
└── ...
```

しかし、[ルーティングセクション](/docs/app/building-your-application/routing) では、ファイルに2桁の数字が付いており、これらの概念を学ぶべき順序で並べ替えられています：

```txt
01-routing
├── 01-defining-routes.mdx
├── 02-pages.mdx
├── 03-layouts-and-templates.mdx
└── ...
```

ページをすばやく見つけるには、`⌘ + P` (Mac) または `Ctrl + P` (Windows) を使用してVSCodeの検索バーを開きます。次に、探しているページのスラッグを入力します。例：`defining-routes`

> **なぜマニフェストを使わないのか？**
>
> ドキュメントのナビゲーションを生成するための他の一般的な方法として、マニフェストファイルの使用を検討しましたが、マニフェストはファイルとすぐに同期が取れなくなることがわかりました。ファイルシステムルーティングは、ドキュメントの構造について考えることを強制し、Next.jsに対してよりネイティブに感じられます。

## メタデータ {#metadata}

各ページには、三つのダッシュで区切られたメタデータブロックがファイルの先頭にあります。

### 必須フィールド {#required-fields}

次のフィールドは**必須**です：

| Field         | Description                                                              |
| ------------- | ------------------------------------------------------------------------ |
| `title`       | ページの`<h1>`タイトル。SEOとOG画像に使用されます。                      |
| `description` | ページの説明。SEOのために`<meta name="description">`タグに使用されます。 |

```yaml title="required-fields.mdx"
---
title: Page Title
description: Page Description
---
```

ページタイトルを2〜3語（例：画像の最適化）に制限し、説明を1〜2文（例：Next.jsで画像を最適化する方法を学べます）に制限することが良い習慣です。

### オプションのフィールド {#optional-fields}

次のフィールドは**任意**です：

| Field       | Description                                                                                                                                                 |
| ----------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `nav_title` | ナビゲーションでページのタイトルを上書きします。これは、ページのタイトルが長すぎて収まらない場合に便利です。未提供の場合、`title`フィールドが使用されます。 |
| `source`    | コンテンツを共有ページにプルします。[共有ページ](#shared-pages)を参照してください。                                                                         |
| `related`   | ドキュメントの末尾に関連ページのリストを表示します。これらは自動的にカードになります。[関連リンク](#related-links)を参照してください。                      |
| `version`   | 開発のステージ。例：`experimental`、`legacy`、`unstable`、`RC`                                                                                              |

```yaml title="optional-fields.mdx"
---
nav_title: Nav Item Title
source: app/building-your-application/optimizing/images
related:
  description: See the image component API reference.
  links:
    - app/api-reference/components/image
version: 'experimental'
---
```

## `App`および`Pages`ドキュメント {#app-and-pages-docs}

ほとんどの**App Router**と**Pages Router**の機能は完全に異なるため、それぞれのドキュメントは別々のセクション（`02-app`および`03-pages`）に保持されています。ただし、これらの間で共有されているいくつかの機能があります。

### 共有ページ {#shared-pages}

コンテンツの重複を避け、内容が非同期になるリスクを抑えるために、`source`フィールドを使用して、あるページから別のページにコンテンツをプルします。たとえば、`<Link>`コンポーネントは、**App**と**Pages**で*ほとんど*同じ動作をします。コンテンツを重複させるのではなく、`app/.../link.mdx`から`pages/.../link.mdx`にコンテンツをプルすることができます：

```mdx title="app/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
---

This API reference will help you understand how to use the props
and configuration options available for the Link Component.
```

```mdx title="pages/.../link.mdx"
---
title: <Link>
description: API reference for the <Link> component.
source: app/api-reference/components/link
---

{/* このページを編集しないでください。 */}
{/* このページの内容は上記のソースから取得されます。 */}
```

したがって、一カ所のコンテンツを編集すれば、それが両方のセクションに反映されます。

### 共有コンテンツ {#shared-content}

共有ページでは、場合によっては**App Router**または**Pages Router**専用のコンテンツが含まれることがあります。たとえば、`<Link>`コンポーネントには**App**ではなく**Pages**でのみ利用可能な`shallow` propがあります。

正しいルーターにのみコンテンツが表示されるようにするために、`<AppOnly>`または`<PagesOnly>`コンポーネントでコンテンツブロックをラップすることができます：

```mdx title="app/.../link.mdx"
This content is shared between App and Pages.

<PagesOnly>

This content will only be shown on the Pages docs.

</PagesOnly>

This content is shared between App and Pages.
```

これらのコンポーネントは、例やコードブロックに使用することが多いでしょう。

## コードブロック {#code-blocks}

コードブロックには、コピーして貼り付けることができる最低限の動作例を含めるべきです。つまり、コードは追加の設定なしで動作する必要があります。

たとえば、`<Link>`コンポーネントの使用方法を示している場合、`import`文と`<Link>`コンポーネント自体を含めるべきです。

```tsx title="app/page.tsx"
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

例をコミットする前に、必ずローカルで実行してください。これにより、コードが最新で動作していることが確認できます。

### 言語とファイル名 {#language-and-filename}

コードブロックには、言語と`filename`を含むヘッダーが必要です。`filename`プロップを追加して、ユーザーにコマンドを入力する位置を示す特別なターミナルアイコンをレンダリングします。例：

````mdx title="code-example.mdx"
```bash title="Terminal"
npx create-next-app
```
````

ドキュメントのほとんどの例は`tsx`と`jsx`で書かれていますが、一部は`bash`で書かれています。ただし、サポートされている任意の言語を使用できます。こちらが[完全なリスト](https://github.com/shikijs/shiki/blob/main/docs/languages.md#all-languages)です。

JavaScriptのコードブロックを書くとき、次の言語と拡張子の組み合わせを使用します。

|                                     | Language | Extension |
| ----------------------------------- | -------- | --------- |
| JavaScriptファイルでJSXコードを含む | ```jsx   | .js       |
| JSXを含まないJavaScriptファイル     | ```js    | .js       |
| JSXを含むTypeScriptファイル         | ```tsx   | .tsx      |
| JSXを含まないTypeScriptファイル     | ```ts    | .ts       |

### TSとJSのスイッチャー {#ts-and-js-switcher}

TypeScriptとJavaScriptを切り替えるための言語スイッチャーを追加します。コードブロックは最初にTypeScriptで書かれ、JavaScriptバージョンも用意してユーザーに対応します。

現在、TSとJSの例を一つずつ連続して書き、`switcher`プロップでリンクします：

````mdx title="code-example.mdx"
<Tabs>
<TabItem value="tsx" label="TypeScript">

```tsx title="app/page.tsx" switcher

```

</TabItem>
<TabItem value="jsx" label="JavaScript">

```jsx title="app/page.js" switcher

```

</TabItem>
</Tabs>
````

> **Good to know**: 将来的に、TypeScriptスニペットをJavaScriptに自動変換する計画があります。当面の間、[transform.tools](https://transform.tools/typescript-to-javascript)を使用できます。

### 行のハイライト {#line-highlighting}

コードの行はハイライトできます。これにより、コードの特定の部分に注意を引くことができます。`highlight`プロップに数値を渡すことで行をハイライトできます。

**単一行:** `highlight={1}`

```tsx title="app/page.tsx" {1}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**複数行:** `highlight={1,3}`

```tsx title="app/page.tsx" highlight={1,3}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

**行の範囲:** `highlight={1-5}`

```tsx title="app/page.tsx" highlight={1-5}
import Link from 'next/link'

export default function Page() {
  return <Link href="/about">About</Link>
}
```

## 関連リンク {#related-links}

関連リンクはユーザーの学習の旅を誘導し、次のステップに論理的に進むリンクを追加します。

- リンクはページのメインコンテンツの下のカードに表示されます。
- リンクは、子ページを持っているページに対して自動的に生成されます。たとえば、[最適化](/docs/app/building-your-application/optimizing)セクションは、すべての子ページへのリンクを持っています。

関連リンクをページのメタデータの`related`フィールドを使用して作成します。

```yaml title="example.mdx"
---
related:
  description: Learn how to quickly get started with your first application.
  links:
    - app/building-your-application/routing/defining-routes
    - app/building-your-application/data-fetching
    - app/api-reference/file-conventions/page
---
```

### ネストされたフィールド {#nested-fields}

| フィールド    | 必須? | 説明                                                                                                                                                             |
| ------------- | ----- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `title`       | 任意  | カードリストのタイトル。デフォルトは **次のステップ**。                                                                                                          |
| `description` | 任意  | カードリストの説明。                                                                                                                                             |
| `links`       | 必須  | 他のドキュメントページへのリンクのリスト。各リストアイテムは相対URLパス（前にスラッシュなし）である必要があります。例：`app/api-reference/file-conventions/page` |

## スタイルガイド {#style-guide}

このセクションは、技術文書の執筆が初めての方向けに、ドキュメントの作成に関するガイドラインを含んでいます。

### ページテンプレート {#page-templates}

ページに厳格なテンプレートはありませんが、ドキュメント全体で繰り返されるページセクションがあります：

- **概要:** ページの最初の段落は、その機能が何で、何に使用されるかをユーザーに伝える必要があります。続いて、最低限の動作例またはAPIリファレンスが紹介されます。
- **規約:** 機能に規約がある場合、ここで説明されるべきです。
- **例:** 異なる使用例で機能をどのように使用できるかを示します。
- **APIテーブル:** APIページには、可能な限りジャンプ先セクションリンクを含む概要テーブルが最初に表示されるべきです。
- **次のステップ (関連リンク):** ユーザーの学習の旅を導くために関連ページへのリンクを追加します。

これらのセクションを必要に応じて追加してください。

### ページタイプ {#page-types}

ドキュメントページは、概念的および参照に分かれています。

- **概念的**ページは、概念または機能を説明するために使用されます。通常、これらは参照ページよりも長く、より多くの情報を含みます。Next.jsドキュメントでは、概念的ページは **アプリケーションの構築** セクションにあります。
- **参照**ページは、特定のAPIを説明するために使用されます。通常、これらは短くフォーカスされています。Next.jsドキュメントでは、参照ページは **APIリファレンス** セクションにあります。

> **Good to know**: 寄稿するページに応じて、異なる口調とスタイルに従う必要があるかもしれません。たとえば、概念的なページはより指導的で、ユーザーを指すために*あなた*という言葉を使用します。参照ページはより技術的で、_作成、更新、受け入れ_ などの命令形の言葉を使用し、_あなた_ という言葉を省く傾向があります。

### 文体 {#voice}

ドキュメント全体で一貫したスタイルと口調を維持するためのガイドラインは次のとおりです：

- 明確で簡潔な文章を書くこと。余談は避ける。
  - カンマを多用していることに気付いた場合、文を複数の文に分割するか、リストを使用することを検討してください。
  - 複雑な言葉を簡単な言葉に置き換えてください。たとえば、*利用する*ではなく*使う*。
- *これ*という単語に注意してください。不明確で混乱を招く可能性があります。文の主語が不明確な場合、繰り返すことを恐れないでください。
  - たとえば、*Next.js uses React*の代わりに*NS.js uses this*。
- 受動態ではなく能動態を使用してください。能動的な文は読みやすいです。
  - たとえば、*Next.js uses React*の代わりに*React is used by Next.js*。*was*や*by*のような言葉を使用している場合、それは受動態かもしれません。
- _簡単_、_素早い_、_単純_、*ただ*などの言葉を避けてください。これは主観的であり、ユーザーのやる気を削ぐ可能性があります。
- _しないでください_、_できません_、*しません*などの否定的な言葉を避けてください。これは読者のやる気を削ぐ可能性があります。
  - たとえば、*`Link`コンポーネントを使用してページ間のリンクを作成できます*の代わりに、_`<a>`タグを使ってページ間のリンクを作成しないでください_。
- 二人称（あなた/あなたの）で書きます。これは、より個人的で魅力的です。
- 性別中立の言葉を使用してください。_開発者_、_ユーザー_、または*読者*を使用して、対象者を指します。
- コード例を追加する場合は、適切にフォーマットされており、かつ動作していることを確認してください。

これらのガイドラインは包括的ではありませんが、始めるのに役立ちます。技術文書についてより深く学びたい場合は、[Google Technical Writing Course](https://developers.google.com/tech-writing/overview)をチェックしてください。
